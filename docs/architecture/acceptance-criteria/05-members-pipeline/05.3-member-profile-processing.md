# 5.3 Member Profile Processing

> Part of [Component 5 — Members Pipeline Projects](../05-MEMBERS-PIPELINE.md) — Project A: `member-profile-pipeline`

## Overview

FS2 streaming pipeline that fetches member profiles from Congress.gov, detects changes, archives history, upserts to AlloyDB, and emits `member.updated` events with chamber-aware logic.

## Classes

### `MemberProfileProcessor[F[_]]`

```
class MemberProfileProcessor[F[_]: Async](
  apiClient: MembersApiClient[F],
  memberRepo: MemberRepository[F],
  termRepo: MemberTermRepository[F],
  partyHistoryRepo: MemberPartyHistoryRepository[F],
  historyArchiver: MemberHistoryArchiver[F],
  changeDetector: ChangeDetector[MemberDO],
  eventPublisher: IngestionEventPublisher[F],
  config: MemberProfileConfig,
  logger: PipelineLogger[F]
) {
  def streamAll(correlationId: UUID): Stream[F, ProcessingResult]
  def processMember(listItem: MemberListItemDTO, correlationId: UUID): F[ProcessingResult]
}
```

### `MemberProfileConfig`

```
final case class MemberProfileConfig(
  congress: Int,
  parallelism: Int = 4,
  pageDelay: FiniteDuration = 500.millis
)
```

**`streamAll` behavior:**

1. Fetch all members for `config.congress` via `apiClient.fetchAll`
2. For each member: `processMember` via `parEvalMap(config.parallelism)`
3. Collect `ProcessingResult` per member (Success or Failure with error)

**`processMember` behavior:**

1. Fetch detail via `apiClient.fetchDetail(listItem.url)`
2. Convert to DO via `MemberDetailDTO.toDO` → `MemberConversionResult` (member + terms + party history)
3. Check if member exists in DB via `memberRepo.findByMemberId`
4. **If exists**: detect change via `changeDetector`
   - If `Unchanged`: skip, return `ProcessingResult.Skipped`
   - If `Updated`: proceed to step 5
5. **If new (not in DB)**: proceed directly to step 5 (archive will no-op)
6. **Archive + Upsert (single transaction):** Steps 6–9 execute in a **single Doobie transaction**. If any step fails, the entire transaction rolls back.
   - Archive current state via `historyArchiver.archiveMember(memberId)` (no-op for new members)
7. Upsert member via `memberRepo.upsert`
8. Replace terms via `termRepo.replaceAll`
9. Append new party history via `partyHistoryRepo.appendNew`
10. Determine chamber from the member's most recent term (`MemberTermDO` with highest `congress` value). Use `chamber` field from that term.
11. **Emit event (outside transaction)** based on chamber:
    - **House member**: emit `MemberUpdatedEvent(memberId)` immediately
    - **Senate member**: check `memberRepo.existsWithLisMapping(memberId)`
      - If true: emit `MemberUpdatedEvent(memberId)`
      - If false: log that senate member is incomplete (no LIS mapping yet), skip event
12. Return `ProcessingResult.Success`

**Placeholder handling:**

Placeholder members (created by bills pipeline) have null `updateDate`. The change detector treats null → non-null as `Updated`, so the normal flow handles them: archive placeholder state → upsert full data → emit event (if eligible).

### Acceptance Criteria

| Criterion | Test |
|-----------|------|
| Streams all members for configured congress | Mock API returns 5 members → 5 `ProcessingResult` |
| Fetches detail for each member | Mock API: list → detail calls match 1:1 |
| Converts `MemberDetailDTO.toDO` correctly | Valid DTO → `MemberConversionResult` with member + terms + party history |
| Skips unchanged members | DB has member with same updateDate → `Skipped` |
| Archives before upsert on changed members | Changed member → `archiveMember` called before `upsert` |
| Does not archive new members | New member (not in DB) → `archiveMember` no-op, upsert proceeds |
| Upserts member profile | Changed member → `upsert` called with new DO |
| Replaces terms | Changed member → `replaceAll` called with new terms |
| Appends new party history | Changed member → `appendNew` called |
| House member emits `member.updated` immediately | House member upserted → event emitted |
| Senate member with LIS mapping emits event | Senate member + mapping exists → event emitted |
| Senate member without LIS mapping skips event | Senate member + no mapping → no event, log message |
| Placeholder member gets filled in | Placeholder in DB + real data from API → archive + upsert + event |
| `toDO` failure produces `ProcessingResult.Failure` | Invalid DTO → failure result with error detail |
| API failure produces `ProcessingResult.Failure` | Mock API throws → failure result |
| Parallelism respected | `config.parallelism = 2` → at most 2 concurrent `processMember` calls |
| Correlation ID flows through all operations | All log messages and results carry the correlation ID |
