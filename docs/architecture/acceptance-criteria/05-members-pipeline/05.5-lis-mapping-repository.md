# 5.5 LIS Mapping Repository

> Part of [Component 5 — Members Pipeline Projects](../05-MEMBERS-PIPELINE.md) — Project B: `lis-mapping-refresher`

## Overview

Doobie repository for the `lis_member_mapping` table. Supports upsert with `lastVerified` timestamp tracking and detection of newly created mappings.

## Classes

### `LisMappingRepository[F[_]]`

```
class LisMappingRepository[F[_]: Sync](
  transactor: Transactor[F]
) {
  def upsert(mapping: LisMemberMappingDO): F[UpsertResult]
  def upsertBatch(mappings: List[LisMemberMappingDO]): F[List[UpsertResult]]
  def findByLisId(lisMemberId: String): F[Option[LisMemberMappingDO]]
  def findByMemberId(memberId: String): F[Option[LisMemberMappingDO]]
}
```

### `UpsertResult`

```
sealed trait UpsertResult
object UpsertResult {
  case object Inserted extends UpsertResult
  case object Updated extends UpsertResult
}
```

**`upsert` behavior:**

1. `INSERT ... ON CONFLICT (lis_member_id) DO UPDATE SET member_id = ?, last_verified = ? RETURNING xmax`
2. PostgreSQL's `xmax` system column distinguishes insert from update: `xmax = 0` means the row was newly inserted; `xmax > 0` means it was updated in-place
3. Return `Inserted` if `xmax = 0`, `Updated` if `xmax > 0`
4. The `Inserted` vs `Updated` distinction is critical — `Inserted` means a senator just gained a mapping and may need a `member.updated` event

**`upsertBatch` behavior:**

1. Process mappings in a single transaction
2. Return one `UpsertResult` per mapping, preserving order

### Error Types

```
final case class LisMappingUpsertFailed(
  lisMemberId: String,
  detail: String,
  cause: Option[Throwable] = None
) extends Exception(
  s"Failed to upsert LIS mapping for $lisMemberId: $detail",
  cause.orNull
)
```

### Acceptance Criteria

| Criterion | Test |
|-----------|------|
| Upsert inserts new mapping | Insert → `findByLisId` returns mapping |
| Upsert returns `Inserted` for new mapping | New lisMemberId → `Inserted` |
| Upsert updates existing mapping | Insert, upsert with new `lastVerified` → updated timestamp |
| Upsert returns `Updated` for existing mapping | Existing lisMemberId → `Updated` |
| `findByLisId` returns None for unknown LIS ID | Unknown ID → None |
| `findByMemberId` returns mapping by bioguideId | Insert mapping → `findByMemberId` returns it |
| Batch upsert processes all mappings | 10 mappings → 10 `UpsertResult` |
| Batch upsert is transactional | Simulate failure mid-batch → no partial writes |
| Batch preserves order | 3 mappings → results in same order |
| `lisMemberId` is primary key | Two upserts with same LIS ID → one row |
| Doobie `Read`/`Write` round-trip for `LisMemberMappingDO` | Insert + read → identical fields |
