# 7.3 Amendment Processing Pipeline

> Part of [Component 7 — Amendments Pipeline](../07-AMENDMENTS-PIPELINE.md)

## Overview

The core pipeline logic for `amendments-pipeline`. Streams amendments from `AmendmentsApiClient`, detects changes via `ChangeDetector`, creates placeholder members/bills, and upserts to AlloyDB. No events are emitted — bill re-analysis is triggered by new bill text, not by amendment recording.

## Classes

### `AmendmentProcessor[F[_]]`

```
class AmendmentProcessor[F[_]: Async](
  apiClient: AmendmentsApiClient[F],
  amendmentRepository: AmendmentRepository[F],
  placeholderCreator: PlaceholderCreator[F],
  memberRepository: EntityRepository[F, MemberDO],
  billRepository: EntityRepository[F, BillDO],
  config: AmendmentsConfig,
  logger: PipelineLogger[F]
)
```

#### `streamAll(correlationId: UUID): Stream[F, ProcessingResult]`

Primary pipeline entry point.

1. Compute lookback: `fromDateTime = Instant.now.minus(config.lookbackDays, ChronoUnit.DAYS)`
2. `apiClient.fetchAll(FetchParams(fromDateTime = Some(fromDateTime)))` → `Stream[F, AmendmentListItemDTO]`
3. For each amendment via `parEvalMap(config.parallelism)`:
   - Construct `amendmentId`: `{congress}-{amendmentType}-{number}` from list item fields
   - Fetch stored: `amendmentRepository.findByAmendmentId(amendmentId)` → `Option[AmendmentDO]`
   - Detect change: `ChangeDetector.detect(incoming, stored, incomingUpdateDate, storedUpdateDate)` → `ChangeReport`
   - `New` or `Updated` → `processAmendment(listItem, changeReport, correlationId)`
   - `Unchanged` → log at debug, return `ProcessingResult.Skipped(amendmentId, "unchanged")`

> **`ChangeDetector` pre-filter:** The list endpoint provides `updateDate` for each amendment. `ChangeDetector` uses the fast `updateDate` comparison to skip unchanged amendments without a detail API call. Only amendments with a newer `updateDate` proceed to `fetchDetail`. This follows the same pattern as bills (Component 4 §4.3).

#### `processAmendment(listItem: AmendmentListItemDTO, changeReport: ChangeReport, correlationId: UUID): F[ProcessingResult]`

Per-amendment processing for new or updated amendments.

1. **Fetch detail:** `apiClient.fetchDetail(listItem.url)` → `AmendmentDetailDTO`
2. **Convert:** `amendmentDetail.toDO` → `Either[String, AmendmentDO]`
   - `Left(reason)` → log warning, return `ProcessingResult.Failed(amendmentId, reason)`
   - `Right(amendmentDO)` → continue
3. **Placeholder sponsor:** If `amendmentDO.sponsorBioguideId` is `Some(memberId)`:
   - `placeholderCreator.ensureExists[MemberDO](memberId, memberRepository)`
   - Creates placeholder if member not in DB; no-op if exists
4. **Placeholder bill:** If `amendmentDO.billId` is `Some(billId)`:
   - `placeholderCreator.ensureExists[BillDO](billId, billRepository)`
   - Creates placeholder if bill not in DB; no-op if exists
5. **Upsert:** `amendmentRepository.upsert(amendmentDO)`
   - No history archival (per index file design decision)
   - No child table writes (single DO)
6. **Log:** `"Amendment {amendmentId} upserted ({N} field changes)"` at info level
7. Return `ProcessingResult.Succeeded(amendmentId)`

> **Transaction boundary:** The upsert is a single SQL statement — no multi-table transaction needed (unlike bills which wrap archive + upsert + replace-all in a single transaction).

#### `summarize(results: List[ProcessingResult]): PipelineRunSummary`

Aggregates individual results into a summary:

```
PipelineRunSummary(
  totalProcessed = results.size,
  succeeded = results.count(_.isSucceeded),
  skipped = results.count(_.isSkipped),
  failed = results.count(_.isFailed),
  eventsEmitted = 0,   // amendments pipeline emits no events
  errors = results.collect { case f: Failed => f.reason }
)
```

### Key Design Decisions

- **Detail fetch for every changed amendment** — the list endpoint lacks sponsor and amended bill data. Every new or updated amendment gets a detail API call.
- **Single amendment failure does not halt the stream** — errors are captured per-amendment as `ProcessingResult.Failed`. The stream continues.
- **Placeholders before upsert** — sponsor and bill placeholder creation happens before amendment upsert so FK constraints are satisfied.
- **Change detection per BEHAVIORAL_SPECS §1** — `updateDate` comparison only. The full field-by-field diff from `ChangeDetector` is logged for observability but does not gate persistence. Any date change triggers a full upsert.
- **No events emitted** — this pipeline persists amendment data only. Bill re-analysis is triggered by new bill text (via `bill-text-availability-checker`, Component 4 Project B), not by amendment recording. See the index file for the full rationale.
- **No history archival** — amendments rarely change substantively. If needed in the future, add `AmendmentHistoryArchiver` following the same pattern as bills/votes/members.

## Acceptance Criteria

| Criterion | Test |
|-----------|------|
| New amendment (not in DB) is fetched, converted, and upserted | Mock API returns amendment → repository empty → amendment upserted |
| Updated amendment (`updateDate` newer) is re-fetched and upserted | Stored amendment with older `updateDate` → API returns newer → upserted |
| Unchanged amendment (same `updateDate`) is skipped | Same `updateDate` → `ProcessingResult.Skipped` |
| Data regression (API `updateDate` older than stored) is skipped with warning | Older API date → `Unchanged`, warning logged |
| Amendment detail is fetched for new/updated amendments | New amendment detected → `fetchDetail` called once |
| Amendment detail is NOT fetched for unchanged amendments | Unchanged amendment → `fetchDetail` not called |
| DTO-to-DO conversion failure produces `ProcessingResult.Failed` | `toDO` returns `Left` → `Failed` result, amendment not persisted |
| Sponsor placeholder created for unknown member | Amendment with sponsor `S001217` not in members → placeholder inserted |
| Sponsor placeholder not created when `sponsorBioguideId` is `None` | Amendment without sponsor → `ensureExists` not called for members |
| Bill placeholder created for unknown amended bill | Amendment referencing bill `117-hr-3684` not in bills → placeholder inserted |
| Bill placeholder not created when `billId` is `None` | Standalone amendment → `ensureExists` not called for bills |
| Placeholder not created if entity already exists | Sponsor already in members → `ensureExists` is no-op |
| No events emitted by this pipeline | Verify no `IngestionEventPublisher` dependency or calls |
| Pipeline uses `parEvalMap(config.parallelism)` | Config parallelism = 4 → verify parallel execution |
| Single amendment failure does not halt stream | First amendment fails, second succeeds → 1 Failed, 1 Succeeded |
| `PipelineRunSummary` accurately counts results | 3 succeeded, 2 skipped, 1 failed → summary matches |
| `eventsEmitted` is always 0 | Verify summary `eventsEmitted = 0` |
| All processing uses correlation ID from run context | Verify `correlationId` propagated to logger context |
| Lookback window computed from config | `lookbackDays = 7` → `fromDateTime` is 7 days ago |
