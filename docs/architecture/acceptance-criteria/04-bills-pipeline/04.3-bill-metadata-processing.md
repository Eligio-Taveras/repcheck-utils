# 4.3 Bill Metadata Processing

> Part of [Component 4 — Bills Pipeline Projects](../04-BILLS-PIPELINE.md) — Project A: `bill-metadata-pipeline`

## Overview

The core pipeline logic for `bill-metadata-pipeline`. Streams bills from `BillsApiClient`, detects changes via `ChangeDetector`, archives old data to history tables, upserts new data, and creates placeholder members for unknown sponsors.

## Classes

### `BillMetadataProcessor[F[_]]`

```
class BillMetadataProcessor[F[_]: Async](
  apiClient: BillsApiClient[F],
  billRepository: BillRepository[F],
  cosponsorRepository: BillCosponsorRepository[F],
  subjectRepository: BillSubjectRepository[F],
  historyArchiver: BillHistoryArchiver[F],
  placeholderCreator: PlaceholderCreator[F],
  memberRepository: EntityRepository[F, MemberDO],
  config: BillMetadataConfig,
  logger: PipelineLogger[F]
)
```

#### `streamAll(correlationId: UUID): Stream[F, ProcessingResult]`

Primary pipeline entry point.

1. Compute lookback: `fromDateTime = Instant.now.minus(config.lookbackDays, ChronoUnit.DAYS)`
2. `apiClient.fetchAll(FetchParams(fromDateTime = Some(fromDateTime)))` → `Stream[F, BillListItemDTO]`
3. For each bill via `parEvalMap(config.parallelism)`:
   - Fetch stored: `billRepository.findByBillId(bill.billId)` → `Option[BillDO]`
   - Detect change: `ChangeDetector.detect(incoming, stored, incomingUpdateDate, storedUpdateDate)` → `ChangeReport`
   - `New` or `Updated` → `processBill(listItem, changeReport, correlationId)`
   - `Unchanged` → log at debug, return `ProcessingResult.Skipped(billId, "unchanged")`

#### `processBill(listItem: BillListItemDTO, changeReport: ChangeReport, correlationId: UUID): F[ProcessingResult]`

Per-bill processing for new or updated bills.

1. **Fetch detail:** `apiClient.fetchDetail(listItem.url)` → `BillDetailDTO`
2. **Convert:** `billDetail.toDO` → `Either[String, BillConversionResult]`
   - `Left(reason)` → log warning, return `ProcessingResult.Failed(billId, reason)`
   - `Right(BillConversionResult(billDO, cosponsors, subjects))` → continue
3. **Placeholder sponsors:** For `billDO.sponsorBioguideId` (if `Some`) and each cosponsor's `memberId`:
   - `placeholderCreator.ensureExists[MemberDO](memberId, memberRepository)`
   - Creates placeholder if member not in DB; no-op if exists
4. **Archive + Upsert (single transaction):** Steps 4–7 execute in a **single Doobie transaction** (`transact`). If any step fails, the entire transaction rolls back — no partial writes.
   - If `changeReport` is `Updated` (not `New`): `historyArchiver.archiveBill(billId)` → archives current bill + cosponsors + subjects
5. `billRepository.upsert(billDO)`
6. `cosponsorRepository.replaceAll(billId, cosponsors)`
7. `subjectRepository.replaceAll(billId, subjects)`
8. **Log:** `"Bill {billId} upserted ({N} field changes)"` at info level
9. Return `ProcessingResult.Succeeded(billId)`

> **Transaction boundary:** Archive, upsert, and replace-all operations are wrapped in a single transaction to prevent partial writes (e.g., bill upserted but cosponsors lost). Event publishing (in pipelines that emit events) happens **outside** the transaction — after a successful commit. If event publishing fails, it is retried via `RetryWrapper`. If retries are exhausted, the failure is logged at error level and triggers an alert for operator intervention. The DB state is correct; only the downstream notification is missing.

#### `summarize(results: List[ProcessingResult]): PipelineRunSummary`

Aggregates individual results into a summary:

```
PipelineRunSummary(
  totalProcessed = results.size,
  succeeded = results.count(_.isSucceeded),
  skipped = results.count(_.isSkipped),
  failed = results.count(_.isFailed),
  eventsEmitted = 0,   // metadata pipeline emits no events
  errors = results.collect { case f: Failed => f.reason }
)
```

### Key Design Decisions

- **Detail fetch for every changed bill** — the list endpoint lacks sponsor/cosponsor/subject data. Every new or updated bill gets a detail API call.
- **Single bill failure does not halt the stream** — errors are captured per-bill as `ProcessingResult.Failed`. The stream continues.
- **Archive before upsert** — history is captured before data is overwritten, so the archive reflects the prior state.
- **Placeholders before upsert** — sponsor placeholder creation happens before bill upsert so FK constraints are satisfied.
- **Change detection per BEHAVIORAL_SPECS §1** — `updateDate` comparison only. The full field-by-field diff from `ChangeDetector` is logged for observability but does not gate persistence. Any date change triggers a full upsert.
- **No events emitted** — this pipeline persists metadata only. Event emission is the responsibility of `bill-text-availability-checker` (Project B).

### Acceptance Criteria

| Criterion | Test |
|-----------|------|
| New bill (not in DB) is fetched, converted, and upserted | Mock API returns bill → repository empty → bill upserted |
| Updated bill (`updateDate` newer) is re-fetched and upserted | Stored bill with older `updateDate` → API returns newer → upserted |
| Unchanged bill (same `updateDate`) is skipped | Same `updateDate` → `ProcessingResult.Skipped` |
| Data regression (API `updateDate` older than stored) is skipped with warning | Older API date → `Unchanged`, warning logged |
| Bill detail is fetched for new/updated bills | New bill detected → `fetchDetail` called once |
| Bill detail is NOT fetched for unchanged bills | Unchanged bill → `fetchDetail` not called |
| DTO-to-DO conversion failure produces `ProcessingResult.Failed` | `toDO` returns `Left` → `Failed` result, bill not persisted |
| Sponsor placeholder created for unknown member | Bill with sponsor `B000944` not in members → placeholder inserted |
| Cosponsor placeholders created for unknown members | 3 cosponsors, 1 unknown → 1 placeholder created |
| Placeholder not created if member already exists | Sponsor already in members → `ensureExists` is no-op |
| History archived before upsert for updated bills | Updated bill → `archiveBill` called before `upsert` |
| History NOT archived for new bills | New bill → `archiveBill` not called |
| Cosponsors and subjects replaced alongside bill | Bill with 3 cosponsors, 5 subjects → all persisted |
| Pipeline uses `parEvalMap(config.parallelism)` | Config parallelism = 4 → verify parallel execution |
| Single bill failure does not halt stream | First bill fails, second succeeds → 1 Failed, 1 Succeeded |
| `PipelineRunSummary` accurately counts results | 3 succeeded, 2 skipped, 1 failed → summary matches |
| `eventsEmitted` is always 0 for metadata pipeline | Verify summary `eventsEmitted = 0` |
| All processing uses correlation ID from run context | Verify `correlationId` propagated to logger context |
| Lookback window computed from config | `lookbackDays = 7` → `fromDateTime` is 7 days ago |
