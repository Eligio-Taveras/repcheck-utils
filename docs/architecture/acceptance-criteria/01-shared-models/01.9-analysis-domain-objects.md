# 1.9 Analysis & Decomposition Domain Objects

> Part of [Component 1 — repcheck-shared-models](../01-SHARED-MODELS.md)

Domain objects for bill text decomposition results and LLM analysis results — the storage layer for concept groups, multi-pass bill analysis, per-concept findings, and amendment findings. These DOs map to tables in the database schema (`001-initial-schema.sql` + `002-schema-expansion.sql` + `003-text-sections-concept-groups.sql` + `005-analysis-schema-alignment.sql`).

> **Cross-component note:** `BillFindingDO` (specifically stance findings with `findingTypeId` for `stance_detection`) feeds the stance materializer (§11.9) via DB status tracking (`stance_materialization_status`), not via direct event consumption. When the analysis pipeline completes, it updates `stance_materialization_status.has_analysis = true`. A scheduled scanner job detects readiness and launches stance materialization, which reads findings from `bill_findings` to populate `member_bill_stance_topics` with per-topic stances, reasoning, and finding FK provenance. `AmendmentFindingDO` follows the same pattern for amendment votes.

### Two Table Layers

Decomposition and analysis produce different kinds of artifacts stored in separate table layers:

| Layer | Tied To | Tables | Purpose |
|-------|---------|--------|---------|
| **Bill text** | Text version | `bill_text_sections`, `bill_concept_groups`, `bill_concept_group_sections` | Parsed sections + clustered concept groups from decomposition. Immutable once produced. Reusable across re-analyses. |
| **Analysis** | Analysis run | `bill_analyses`, `bill_concept_summaries`, `bill_analysis_topics`, `bill_findings`, `bill_fiscal_estimates`, `amendment_findings` | LLM analysis results per pass. Each analysis run produces its own rows. Multiple rows per table with individual embeddings. |

> **Bill text DOs** for sections are defined in §1.4 (`BillTextSectionDO`). The concept group and junction DOs below bridge both layers.

### Field Notes

> **`BillConceptGroupDO` is a decomposition artifact, not an analysis artifact.** It is tied to a `versionId` (text version), not an `analysisId`. The `simplifiedText` is the Haiku-produced coherent summary of the grouped sections. The `embedding` is the 1536-dim vector of the simplified text for semantic search. Multiple analysis runs can reference the same concept groups. For short bills that skip decomposition, there is one concept group covering the entire bill.

> **`BillConceptGroupSectionDO` is the junction table** linking which `bill_text_sections` belong to which concept group. A section belongs to exactly one group.

> **`BillAnalysisDO` tracks the analysis run, not the findings.** The `summary` and `topics` fields are the bill-wide Pass 1 overall summary and topic tags — retained for backward compatibility and event payloads (`analysis.completed` events carry `topics[]`). The detailed per-concept summaries and topic scores are in `BillConceptSummaryDO` and `BillAnalysisTopicDO`. `highestModelUsed` stores the most capable model used across all passes. Per-pass models are tracked in `pass1Model`, `pass2Model`, `pass3Model`. The `status` field tracks the lifecycle: `"in_progress"` → `"completed"` or `"failed"`. The routing score columns (`highProfileScore`, `mediaCoverageLevel`, etc.) are denormalized from `Pass1RoutingScores` and `Pass2RoutingScores` for auditability — Pass 2 columns are NULL when only Pass 1 ran. `passesExecuted` is an integer array (e.g., `[1, 2]`) indicating which passes completed.

> **`BillConceptSummaryDO` is a pure analysis artifact.** It references a concept group via `conceptGroupId` (FK → `bill_concept_groups`) and carries the Pass 1 analysis results for that group: `summary`, `readingLevel`, `keyPoints`, `topics`. The decomposition data (group title, simplified text, section references) lives in `BillConceptGroupDO` — not duplicated here. For short bills that skip decomposition, `conceptGroupId` points to the single concept group covering the entire bill.

> **`BillFindingDO` is the universal finding row.** All finding types (pork, impact, stance, topic extraction, etc.) use the same table. The `findingTypeId` discriminates. The `conceptGroupId` is `None` for bill-wide findings and set for per-concept findings. The `severity`, `confidence`, `affectedSection`, and `affectedGroup` columns are nullable because not all finding types use them:
>
> | Finding Type | Uses `severity` | Uses `confidence` | Uses `affectedSection` | Uses `affectedGroup` |
> |-------------|----------------|-------------------|----------------------|---------------------|
> | `pork` | Yes (ImpactSeverity) | No | Yes | No |
> | `rider` | Yes | No | Yes | No |
> | `impact_analysis` | Yes | No | No | Yes |
> | `stance_detection` | No | Yes | No | No |
> | `topic_extraction` | No | Yes | No | No |
> | `bill_summary` | No | No | No | No |
> | `fiscal_estimate` | No | Yes | No | No |
> | `lobbying` | Yes | No | Yes | No |
> | `constitutional` | Yes | No | Yes | No |

> **`BillFiscalEstimateDO` is separate from findings** because fiscal estimates have a distinct structure (`estimatedCost`, `timeframe`, `assumptions`) that doesn't fit the findings pattern. There is at most one fiscal estimate per analysis run.

> **`BillAnalysisTopicDO` normalizes topic confidence scores.** The `topics TEXT[]` on `bill_analyses` carries flat topic strings (for event payloads). `BillAnalysisTopicDO` carries topic + confidence pairs from `TopicClassificationOutput`, plus an optional `conceptGroupId` to distinguish per-concept topics from bill-wide topics.

### Mapping: LLM Output Schemas → DOs

How the §1.6 output types map to these DOs when persisted by Component 10:

| Output Schema (§1.6) | Persisted As | DO |
|----------------------|-------------|-----|
| `BillSummaryOutput` (overall) | `bill_analyses.summary` + `bill_analyses.reading_level` + `bill_analyses.key_points` | `BillAnalysisDO` |
| `BillSummaryOutput` (per-concept) | `bill_concept_summaries.summary` + `.reading_level` + `.key_points` (linked to `bill_concept_groups` via `concept_group_id`) | `BillConceptSummaryDO` → `BillConceptGroupDO` |
| `TopicClassificationOutput` (overall) | `bill_analyses.topics` (flat) + `bill_analysis_topics` (with confidence) | `BillAnalysisDO` + `BillAnalysisTopicDO` |
| `TopicClassificationOutput` (per-concept) | `bill_analysis_topics` with `concept_group_id` set | `BillAnalysisTopicDO` |
| `PorkDetectionOutput` | One `bill_findings` row per `PorkFinding` | `BillFindingDO` (findingTypeId = `"pork"`) |
| `ImpactAnalysisOutput` | One `bill_findings` row per `ImpactItem` | `BillFindingDO` (findingTypeId = `"impact_analysis"`) |
| `StanceClassificationOutput` | One `bill_findings` row per `TopicStance` | `BillFindingDO` (findingTypeId = `"stance_detection"`) |
| `FiscalEstimateOutput` | One `bill_fiscal_estimates` row | `BillFiscalEstimateDO` |
| `AlignmentScoreOutput` | Score tables (§1.5) — not analysis tables | N/A (see §1.5) |

### Acceptance Criteria — 1.9 Analysis & Decomposition Domain Objects

**Functional — Decomposition DOs:**
1. `BillConceptGroupDO` fields match all columns in `bill_concept_groups` (column names → camelCase).
2. `BillConceptGroupDO` has Doobie auto-derived `Read` and `Write` instances.
3. `BillConceptGroupDO.embedding` is `Option[Array[Float]]` to represent pgvector.
4. `BillConceptGroupDO.versionId` is UUID FK to `bill_text_versions`.
5. Multiple `BillConceptGroupDO` records per version, uniquely constrained on `(versionId, groupId)`.
6. `BillConceptGroupSectionDO` fields match `bill_concept_group_sections`: conceptGroupId (UUID), sectionId (UUID).
7. `BillConceptGroupSectionDO` has Doobie auto-derived `Read` and `Write` instances.

**Functional — Analysis DOs:**
8. Every analysis DO case class has fields matching every column in its corresponding database table (column names converted to camelCase).
9. Every analysis DO has Doobie auto-derived `Read` and `Write` instances (no manual `toPojo` methods).
10. `BillAnalysisDO.embedding` is `Option[Array[Float]]` to represent pgvector.
11. `BillConceptSummaryDO.embedding` is `Option[Array[Float]]`.
12. `BillConceptSummaryDO.conceptGroupId` is `Option[UUID]` — FK to `bill_concept_groups`. `None` only for legacy data or short bills without decomposition.
13. `BillConceptSummaryDO.topics` and `keyPoints` are `List[String]` mapping to `TEXT[]`.
14. `BillFindingDO.embedding` is `Option[Array[Float]]`.
15. `AmendmentFindingDO.embedding` is `Option[Array[Float]]`.
16. `BillAnalysisDO.topics` is `List[String]` mapping to PostgreSQL `TEXT[]`.
17. `BillFiscalEstimateDO.assumptions` is `List[String]` mapping to `TEXT[]`.
17a. `BillAnalysisDO.status` is `String` with valid values "in_progress", "completed", "failed".
17b. `BillAnalysisDO.versionId` is `UUID` FK to `bill_text_versions`.
17c. `BillAnalysisDO.passesExecuted` is `List[Int]` mapping to PostgreSQL `INT[]`.
17d. `BillAnalysisDO.highestModelUsed` is `Option[String]` — populated on completion.
17e. `BillAnalysisDO` routing score columns (`highProfileScore`, `mediaCoverageLevel`, `appropriationsEstimate`, `stanceConfidence`, `routingReasoning`) are all `Option` — populated after Pass 1.
17f. `BillAnalysisDO` Pass 2 routing columns (`overallConfidence`, `crossConceptContradictionScore`, `expectedVoteContention`, `contradictionDetails`, `routingReasoningPass2`) are all `Option` — NULL when Pass 2 did not run.
17g. `BillAnalysisDO.failureReason` is `Option[String]` — populated only when `status = "failed"`.
17h. `BillAnalysisDO.createdAt` is `Instant`, `completedAt` is `Option[Instant]`.
17i. `BillFindingDO.passNumber` is `Int` — indicates which analysis pass (1, 2, or 3) produced this finding.
17j. `BillAnalysisTopicDO.passNumber` is `Int`.
17k. `BillConceptSummaryDO.passNumber` is `Int`.
17l. `BillFiscalEstimateDO.passNumber` is `Int`.
17m. `BillFiscalEstimateDO.conceptGroupId` is `Option[UUID]` FK to `bill_concept_groups`.
18. `BillFindingDO.conceptGroupId` is `Option[String]` — `None` for bill-wide findings, `Some(groupId)` for per-concept findings.
19. `BillFindingDO.severity` is `Option[String]` — only set for finding types that use it (pork, rider, impact, lobbying, constitutional).
20. `BillFindingDO.confidence` is `Option[Float]` — only set for finding types that use it (stance, topic, fiscal).
21. `BillAnalysisTopicDO.conceptGroupId` is `Option[String]` — `None` for bill-wide topics, `Some(groupId)` for per-concept topics.
22. `FindingTypeDO` fields match the `finding_types` lookup table: findingTypeId (Int PK, auto-generated SERIAL), code (String — snake_case lookup key, e.g., "pork_barrel", "stance", "civil_liberties", UNIQUE), name (String), description (Option[String]). The `code` field is unique and used by Component 10 for finding type resolution (never hardcode numeric IDs).

**Error Handling:**
23. All DOs with Float fields validate range 0.0-1.0 at the database level (CHECK constraints) or document that validation is caller-enforced.

**Test Requirements:**
24. Round-trip Doobie `Read`/`Write` test for `BillConceptGroupDO`.
25. Round-trip Doobie `Read`/`Write` test for `BillConceptGroupSectionDO`.
26. `BillConceptGroupDO` with `embedding = None` (not yet vectorized) round-trips correctly.
27. Multiple concept groups per version with distinct `groupId` values round-trip correctly.
28. Round-trip Doobie `Read`/`Write` test for every analysis DO.
29. `BillConceptSummaryDO` with `conceptGroupId = Some(uuid)` round-trips correctly.
30. `BillConceptSummaryDO` with `conceptGroupId = None` (legacy/short bills) round-trips correctly.
31. `BillFindingDO` with all nullable fields as `None` round-trips correctly.
32. `BillFindingDO` with all nullable fields populated round-trips correctly.
33. `BillConceptSummaryDO` with empty `List` fields round-trips as empty `TEXT[]`.
34. `BillAnalysisTopicDO` with `conceptGroupId = None` (bill-wide) and `Some("group-1")` (per-concept) both round-trip.
35. `BillFiscalEstimateDO` with multiple assumptions round-trips the `TEXT[]` correctly.
36. Circe codec round-trip test for each decomposition and analysis DO (for Pub/Sub serialization).
