# 1.4 Bill Text Versioning

> Part of [Component 1 — repcheck-shared-models](../01-SHARED-MODELS.md)

> **Design rationale:** Congress.gov publishes a new text version at each legislative stage (Introduced → Reported → Engrossed → Enrolled). The diff between consecutive versions reveals what changed — and when combined with committee membership data (Section 1.3), enables naive attribution of changes to committee members. The current `bills` table stores only one text version (single `text_url`, `text_content`, etc. columns). A separate `bill_text_versions` table stores all versions, enabling full lifecycle tracking and diffing.
>
> **Sync coupling:** Text version checks run as part of the same bill metadata sync cycle. When `updateDateIncludingText` changes on a bill, the bills-pipeline fetches the new text version and stores it in `bill_text_versions`. The `bills` table retains a reference to the latest version for quick access.

### Text Version Lifecycle

| Version Code | Version Type | Legislative Stage |
|-------------|-------------|-------------------|
| IH | Introduced in House | Bill filed in House |
| IS | Introduced in Senate | Bill filed in Senate |
| RH | Reported in House | Reported out of House committee |
| RS | Reported in Senate | Reported out of Senate committee |
| RFS | Referred to Senate | House bill referred to Senate |
| RFH | Referred to House | Senate bill referred to House |
| EH | Engrossed in House | Passed by House |
| ES | Engrossed in Senate | Passed by Senate |
| ENR | Enrolled Bill | Final version sent to President |
| CPH | Committee Print (House) | Committee working document |
| CPS | Committee Print (Senate) | Committee working document |

### New DOs

| Class | Table | Key Columns |
|-------|-------|-------------|
| `BillTextVersionDO` | `bill_text_versions` | versionId (UUID PK), billId (FK → bills), versionCode (`TextVersionCode` enum), versionType (full name, e.g., "Reported in House"), versionDate, formatType (`FormatType` enum), url, content (Option — full text if fetched), embedding (Option — vector, computed after content fetched), fetchedAt (Option — when content was downloaded), createdAt |
| `BillTextSectionDO` | `bill_text_sections` | sectionId (UUID PK), versionId (FK → bill_text_versions), billId (FK → bills), sectionIndex (Int — ordinal position), sectionIdentifier (Option[String] — e.g., "Sec. 101"), heading (Option[String]), content (String), embedding (Option[Array[Float]] — 1536-dim for semantic search), createdAt |

> **`BillTextSectionDO` stores individual sections parsed from bill XML.** When the decomposition pipeline parses a bill's XML structure, each structural section becomes a row. Sections are immutable once parsed from a text version. The `embedding` column enables per-section semantic search — e.g., "find all sections across all bills that discuss broadband funding." Sections are uniquely constrained on `(versionId, sectionIndex)`.

> **Section-to-concept-group linkage** is handled by the `bill_concept_group_sections` junction table (see §1.9). The concept groups themselves are stored in `bill_concept_groups` (§1.9), tied to the text version rather than an analysis run.

### Changes to `BillDO`

The existing text fields on `BillDO` (`textUrl`, `textFormat`, `textVersionType`, `textDate`, `textContent`, `textEmbedding`) remain for **latest version quick access**. A new field is added:
- `latestTextVersionId: Option[UUID]` — FK to `bill_text_versions.version_id`, pointing to the most recent version

When a new text version is stored in `bill_text_versions`, the `bills` row is updated to point to it and copy the latest fields for backward compatibility.

### New LLM Output Schema

| Class | Fields | Purpose |
|-------|--------|---------|
| `BillTextDiffOutput` | previousVersionCode, currentVersionCode, billId, sections: List[SectionChange], summary: String, significanceScore: Double (0.0-1.0) | Structured diff between two consecutive text versions |
| `SectionChange` | sectionId: String, changeType: ChangeType, previousText: Option[String], currentText: Option[String], description: String | Individual section-level change |

### New Enums

| Enum | Values |
|------|--------|
| `TextVersionCode` | IH, IS, RH, RS, RFS, RFH, EH, ES, ENR, CPH, CPS (extensible — new codes can be added) |
| `ChangeType` | Added, Removed, Modified, Renumbered |

### Event Flow

When a new text version is detected:
1. Bills-pipeline stores it in `bill_text_versions`
2. Bills-pipeline updates `bills.latest_text_version_id` and inline text fields
3. Bills-pipeline emits `bill.text.available` event with `billId`, `versionCode`, and `previousVersionCode`
4. Bill-analysis-pipeline receives event, fetches both versions, runs LLM diff
5. LLM diff output stored as a `BillTextDiffOutput` in `bill_findings` with `finding_type = "TextVersionDiff"`
6. If committee referral data exists, the diff is linked to committee membership for attribution

### Acceptance Criteria — 1.4 Bill Text Versioning

**Functional:**
1. `BillTextVersionDO` fields match all columns in the `bill_text_versions` table.
2. `BillTextVersionDO` has Doobie auto-derived `Read` and `Write` instances.
3. `BillTextSectionDO` fields match all columns in the `bill_text_sections` table.
4. `BillTextSectionDO` has Doobie auto-derived `Read` and `Write` instances.
5. `BillTextSectionDO.embedding` is `Option[Array[Float]]` to represent pgvector.
6. `BillTextSectionDO.sectionIdentifier` and `heading` are `Option[String]` — untitled sections have `None`.
7. Multiple `BillTextSectionDO` records per version, uniquely constrained on `(versionId, sectionIndex)`.
8. `TextVersionCode` enum covers all 11 known version codes with `fromString` parser.
9. `TextVersionCode.fromString` accepts both the code ("RH") and the full name ("Reported in House") and returns the same value.
10. `ChangeType` enum has Circe codecs (lowercase string) and `fromString` parser.
11. `BillTextDiffOutput` has Circe semi-auto derived `Decoder` and `Encoder`.
12. `BillDO.latestTextVersionId` is `Option[UUID]` — `None` for bills with no text yet.
13. The existing `BillDO` text fields (`textUrl`, `textFormat`, etc.) remain populated from the latest version for backward compatibility.
14. Multiple `BillTextVersionDO` records can exist per bill, uniquely constrained on `(billId, versionCode, versionDate)`.

**Error Handling:**
15. `TextVersionCode.fromString` returns `Left` with unrecognized code and list of valid codes.
16. `BillTextDiffOutput` decoder returns clear errors when LLM returns malformed diff JSON.

**Test Requirements:**
17. Round-trip Doobie `Read`/`Write` test for `BillTextVersionDO`.
18. Round-trip Doobie `Read`/`Write` test for `BillTextSectionDO`.
19. `BillTextSectionDO` with `embedding = None` (not yet vectorized) round-trips correctly.
20. `BillTextSectionDO` with `sectionIdentifier = None` and `heading = None` round-trips correctly.
21. Multiple sections per version with sequential `sectionIndex` values round-trip correctly.
22. `TextVersionCode` exhaustive parsing test: every code and full name parses correctly.
23. `TextVersionCode` unknown value test: verify descriptive `Left`.
24. `BillTextDiffOutput` round-trip codec test with populated sections.
25. `BillTextDiffOutput` decode-from-LLM test: sample JSON matching expected LLM output.
26. Test that `BillDO` with `latestTextVersionId = None` represents a bill with no text yet.
27. Test that `BillDO` with `latestTextVersionId = Some(uuid)` and populated text fields is consistent.
28. Circe codec round-trip test for `BillTextSectionDO` (for Pub/Sub serialization).
