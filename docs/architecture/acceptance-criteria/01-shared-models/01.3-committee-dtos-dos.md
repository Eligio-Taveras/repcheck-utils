# 1.3 Committee DTOs & DOs

> Part of [Component 1 — repcheck-shared-models](../01-SHARED-MODELS.md)

> **Data source rationale:** The Congress.gov API provides committee metadata and bill-to-committee referrals, but does **not** expose committee membership. Committee membership comes from official chamber XML feeds:
> - **Senate:** `senate.gov/legislative/LIS_MEMBER/cvc_member_data.xml` — per-senator records with `bioguideId`, `lis_member_id`, and nested committee assignments (code, name, position).
> - **House:** `clerk.house.gov/xml/lists/memberdata.xml` — per-member records with committee/subcommittee assignments and rankings.
> - **Congress.gov:** `/bill/{congress}/{billType}/{billNumber}/committees` — which committees a bill was referred to (referral link, not membership).
>
> **Sync cadence:** Committee membership must be refreshed on the same cycle as bill metadata ingestion. The orchestrator runs both in the same workflow to ensure consistent attribution data. When a bill's text version changes (e.g., Introduced → Reported), the committee membership at the time of the change must already be current.

### Committee DOs

| Class | Table | Key Columns |
|-------|-------|-------------|
| `CommitteeDO` | `committees` | committeeCode (PK, e.g., "SSFI00", "hspw00"), name, chamber, committeeType (Standing, Special, Select, Joint, Subcommittee), parentCommitteeCode (nullable FK for subcommittees), isCurrent, updateDate, createdAt, updatedAt |
| `CommitteeMemberDO` | `committee_members` | committeeCode (FK), memberId (FK → members), position (Chairman, Ranking Member, Vice Chairman, Member), side (Majority, Minority), rank (Int, seniority on committee), beginDate, endDate (nullable — null means current), congress, createdAt, updatedAt |
| `BillCommitteeReferralDO` | `bill_committee_referrals` | billId (FK → bills), committeeCode (FK → committees), referralDate, reportDate (nullable — null means still in committee), activity (e.g., "Referred to", "Reported by", "Discharged from"), createdAt |

> **Temporal membership for attribution:** `CommitteeMemberDO` tracks `beginDate` and `endDate` per congress. When attributing bill changes to committee members, the query joins on the time window between `referralDate` and `reportDate`:
> ```sql
> SELECT cm.member_id, cm.position
> FROM committee_members cm
> JOIN bill_committee_referrals bcr ON cm.committee_code = bcr.committee_code
> WHERE bcr.bill_id = ?
>   AND cm.begin_date <= bcr.report_date
>   AND (cm.end_date IS NULL OR cm.end_date >= bcr.referral_date)
> ```
> This returns everyone who served on the committee during the window the bill was under review — not just current membership.

### Conversion Methods

| Source | Produces | Notes |
|--------|----------|-------|
| `SenatorCommitteeDataXmlDTO.toMemberCommittees` | `List[CommitteeMemberDO]` | One DO per committee assignment. May be empty if senator has no committee assignments. |
| `SenatorCommitteeDataXmlDTO.toLisMemberMapping` | `Option[LisMemberMappingDO]` | Produces a mapping if both `lisMemberId` and `bioguideId` are present. This is a **supplementary** source for LIS-to-bioguide mappings — the primary source is `senator-lookup.xml` (§1.1 `SenatorLookupXmlDTO`), which covers all senators including those with no committee assignments. |
| `HouseMemberDataXmlDTO.toMemberCommittees` | `List[CommitteeMemberDO]` | One DO per committee assignment. |
| `CommitteeListItemDTO.toDO` | `Either[String, CommitteeDO]` | Committee master data. |
| `BillCommitteeReferralDTO.toDO` | `Either[String, BillCommitteeReferralDO]` | Links bill to committee with activity and dates. |

### Acceptance Criteria — 1.3 Committee DTOs & DOs

**Functional:**
1. `SenatorCommitteeDataXmlDTO` parses all fields from the `cvc_member_data.xml` `<senator>` element, including nested `<committees>`.
2. `HouseMemberDataXmlDTO` parses all fields from the `memberdata.xml` `<member>` element, including nested committee assignments.
3. `CommitteeDO` fields match all columns in the `committees` table. `parentCommitteeCode` is `Option[String]` for subcommittees.
4. `CommitteeMemberDO` tracks temporal membership with `beginDate` (required) and `endDate` (`Option` — `None` means currently serving).
5. `CommitteeMemberDO` has a `congress` field to scope membership to a specific Congress session.
6. `BillCommitteeReferralDO` tracks `referralDate` (required) and `reportDate` (`Option` — `None` means bill is still in committee).
7. `SenatorCommitteeDataXmlDTO.toMemberCommittees` produces one `CommitteeMemberDO` per committee assignment for that senator.
8. `SenatorCommitteeDataXmlDTO.toLisMemberMapping` produces an `Option[LisMemberMappingDO]` when both `lisMemberId` and `bioguideId` are present. This is a **supplementary** LIS-to-bioguide mapping source. The primary source is `senator-lookup.xml` (§1.1 `SenatorLookupXmlDTO`) which covers all senators — including those who sit on no committees and may not appear in the committee XML at all.
9. All committee DOs have Doobie auto-derived `Read` and `Write` instances.
10. `CommitteeListItemDTO.toDO` maps Congress.gov `systemCode` to `committeeCode` directly (they use the same coding system).
11. `BillCommitteeReferralDTO.toDO` extracts `referralDate` from the earliest "Referred to" activity and `reportDate` from the earliest "Reported by" activity (if present).

**Error Handling:**
12. XML DTO parsers return `Either[String, DTO]` with descriptive messages when required XML elements are missing.
13. `toDO` conversions return `Left` with specific field and reason when validation fails.

**Test Requirements:**
14. Unit test per Senate XML DTO with sample XML fragment from `cvc_member_data.xml` — verify all fields parsed including nested committees.
15. Unit test per House XML DTO with sample XML fragment from `memberdata.xml` — verify all fields parsed.
16. Unit test for `SenatorCommitteeDataXmlDTO.toMemberCommittees` — verify correct number of `CommitteeMemberDO` records produced.
17. Unit test for `SenatorCommitteeDataXmlDTO.toLisMemberMapping` — verify `Some(LisMemberMappingDO)` produced when both `lisMemberId` and `bioguideId` are present.
18. Unit test for `SenatorCommitteeDataXmlDTO.toLisMemberMapping` — verify `None` when `lisMemberId` is missing from the XML element.
19. Unit test for temporal membership query pattern — two members with overlapping date ranges, verify correct members returned for a given referral window.
20. Unit test for `BillCommitteeReferralDTO.toDO` — verify `referralDate` and `reportDate` extraction from activities list.
21. Round-trip Doobie `Read`/`Write` test for `CommitteeDO`, `CommitteeMemberDO`, and `BillCommitteeReferralDO`.
