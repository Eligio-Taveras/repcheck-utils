### Committee Attribution Weights

`CommitteePosition` carries a `weight: Double` property used by the scoring engine when attributing bill changes to committee members. These weights multiply with the committee vote weight (0.4 from BEHAVIORAL_SPECS) to compute alignment impact.

| Position | Weight | Rationale |
|----------|--------|-----------|
| Chairman | 1.0 | Drives agenda, controls markup process, maximum responsibility for reported bill |
| Ranking Member | 0.7 | Leads opposition/minority strategy, shapes negotiations |
| Vice Chairman | 0.6 | Deputy leadership role |
| Member | 0.4 | Participated in committee process |

> **Example:** A chairman's committee-stage bill changes carry `1.0 × 0.4 = 0.40` toward their alignment score. A regular member carries `0.4 × 0.4 = 0.16`. This reflects that the chairman has substantially more control over what the committee produces.

### Acceptance Criteria — 1.8 Shared Serializers & Constants

**Functional:**
1. `ZonedDateTime` encoder/decoder round-trips in ISO-8601 format, matching existing `Serializers` behavior.
2. `LocalDate` encoder/decoder round-trips in ISO-8601 date format (`yyyy-MM-dd`).
3. `UUID` encoder/decoder round-trips standard UUID string format.
4. Every enum has: Circe encoder (to canonical string), Circe decoder (case-insensitive + alias-aware), `fromString` returning `Either`.
5. Every enum's `fromString` accepts **all known source representations** from every data source and maps them to the canonical enum value. Returns `Left` with the unrecognized value and the list of valid values + aliases when no match is found.

> **Multi-source alias mapping:** Each enum value may appear differently across Congress.gov JSON, Senate XML, House XML, and senate.gov website data. The `fromString` parser must handle all known representations:
>
> | Enum | Source Aliases → Canonical Value |
> |------|--------------------------------|
> | `Party` | "Democratic", "Democrat", "D" → `Democrat`; "Republican", "R" → `Republican`; "Independent", "I", "ID" → `Independent` |
> | `Chamber` | "House", "House of Representatives" → `House`; "Senate" → `Senate` |
> | `VoteCast` | "Yea", "Aye", "Yes" → `Yea`; "Nay", "No" → `Nay`; "Present" → `Present`; "Not Voting", "NotVoting" → `NotVoting`; "Absent" → `Absent` |
> | `CommitteePosition` | "Chairman", "Chair", "Chairwoman", "Chairperson" → `Chairman`; "Ranking", "Ranking Member" → `RankingMember`; "Vice Chairman", "Vice Chair", "Vice Chairwoman" → `ViceChairman`; "Member" → `Member` |
> | `CommitteeSide` | "majority", "Majority", "M" → `Majority`; "minority", "Minority" → `Minority` |
> | `TextVersionCode` | "RH", "Reported in House" → `RH`; "IH", "Introduced in House" → `IH`; etc. (code + full name for all values) |
>
> This alias table must be validated against live data from each source and updated if new representations are discovered. Each alias has a unit test.
6. `BillType` enum includes all 13 values from existing `BillTypes` (renamed for consistency).
7. `VoteType` enum matches the 8 vote type classifications from BEHAVIORAL_SPECS.md Section 2. `VoteType.fromQuestion` correctly classifies known question patterns (e.g., "On Passage" → `Passage`, "On Cloture" → `Cloture`) and returns `Other` for unrecognized patterns.
8. `FindingType` enum matches the `finding_types` database seed values plus `TextVersionDiff`, `Impact`, and `FiscalEstimate` (11 total).
9. `TextVersionCode` enum covers all 11 known version codes. `fromString` accepts both code ("RH") and full name ("Reported in House").
10. `CommitteePosition`, `CommitteeSide`, and `CommitteeType` enums each have Circe codecs and `fromString`.
11. `CommitteePosition` has a `weight: Double` property: Chairman = 1.0, RankingMember = 0.7, ViceChairman = 0.6, Member = 0.4.
12. `CongressGovApiPaths` path templates produce correct URLs when interpolated with congress number, bill type, etc.
12. `ChamberDataPaths` contains correct URLs for Senate and House XML feeds.

**Test Requirements:**
13. Round-trip codec test for `ZonedDateTime` with timezone offsets, UTC, and edge cases (migrated from existing `SerializersSpec`).
14. Round-trip codec test for `LocalDate` with various valid date strings.
15. Round-trip codec test for `UUID`.
16. Exhaustive enum parsing tests: every value of every enum parses correctly from its canonical string representation. This includes all new enums: `TextVersionCode`, `ChangeType`, `CommitteePosition`, `CommitteeSide`, `CommitteeType`.
17. Unknown enum value tests: verify descriptive `Left` for each enum, including the list of valid values and known aliases.
18. Case-insensitivity tests: "HOUSE", "house", "House" all parse to `Chamber.House`.
19. **Multi-source alias tests per enum** — every alias listed in the alias mapping table has its own test case:
    - `Party`: "Democratic" → Democrat, "D" → Democrat, "Republican" → Republican, "R" → Republican, "Independent" → Independent, "I" → Independent, "ID" → Independent
    - `Chamber`: "House of Representatives" → House, "House" → House, "Senate" → Senate
    - `VoteCast`: "Yea" → Yea, "Aye" → Yea, "Yes" → Yea, "Nay" → Nay, "No" → Nay, "Not Voting" → NotVoting, "Absent" → Absent
    - `CommitteePosition`: "Chairman" → Chairman, "Chair" → Chairman, "Chairwoman" → Chairman, "Chairperson" → Chairman, "Ranking" → RankingMember, "Ranking Member" → RankingMember, "Vice Chairman" → ViceChairman, "Vice Chair" → ViceChairman
    - `CommitteeSide`: "majority" → Majority, "M" → Majority, "minority" → Minority
    - `TextVersionCode`: both code and full name for all 11 values
20. `CommitteePosition.weight` test: verify Chairman = 1.0, RankingMember = 0.7, ViceChairman = 0.6, Member = 0.4.
