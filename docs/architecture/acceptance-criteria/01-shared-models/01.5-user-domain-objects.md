> **Schema note:** The `score_history` table in `001-initial-schema.sql` does not currently have a `reasoning` column. Migration `002-schema-expansion.sql` adds `reasoning TEXT` to `score_history` to store the LLM's explanation of the alignment score. Migration `008-score-reasoning.sql` adds `reasoning TEXT` and `reasoning_embedding vector(1536)` to `scores`, and `reasoning_embedding vector(1536)` to `score_history`. Both tables store reasoning: `scores.reasoning` is the current explanation (overwritten on each re-score), and `score_history.reasoning` preserves the explanation at each point in time for trend display. The `reasoning_embedding` columns enable semantic search (e.g., "find legislators where the explanation mentions healthcare") — on `scores` for current-state search, on `score_history` for temporal search across scoring runs. Embeddings are generated by DJL/ONNX (`SemanticEmbeddingService`) after the LLM explanation step in the scoring orchestrator (§11.7).

> **Schema note — amendment_id on member_bill_stances:** The `member_bill_stances` table in `001-initial-schema.sql` does not have an `amendment_id` column. Migration `006-amendment-vote-linkage.sql` adds `amendment_id TEXT REFERENCES amendments(amendment_id)` (nullable). Set when the vote is on an amendment (`vote_type = 'Amendment'`). Used by §11.9 (stance materializer) to look up amendment-specific stance findings instead of parent bill stance.

> **Schema note — new scoring infrastructure tables (migration 009):** Migration `009-scoring-architecture.sql` adds 5 tables: `user_legislator_pairings`, `member_bill_stance_topics`, `user_bill_alignments`, `user_amendment_alignments`, `stance_materialization_status`. It also adds `last_stance_change_at TIMESTAMPTZ` to the `users` table (used by the skip-unchanged optimization in §3.6 of BEHAVIORAL_SPECS.md), `status TEXT` and `non_overlapping_topics TEXT[]` to `scores`, and `status TEXT` to `score_history`.

> **Field naming convention for score tables:** `aggregateScore` (on `ScoreDO`, `ScoreHistoryDO`) is the all-time weighted alignment score across all congresses. `overallScore` (on `ScoreCongressDO`, `ScoreHistoryCongressDO`) is the alignment score for a single congress. The naming distinguishes scope: "aggregate" = cross-congress, "overall" = within one congress. Both are `Float` in range 0.0-1.0.

> **`ScoreDO` vs `ScoreHistoryDO` field differences:** `ScoreDO` represents the *current* score (mutable, overwritten on each re-score) and uses `lastUpdated` + `llmModel` + `reasoning` + `reasoningEmbedding` + counts. `ScoreHistoryDO` is the *append-only audit trail* and uses `computedAt` + `triggerEvent` + `reasoning` + `reasoningEmbedding`. Both carry `reasoning` and `reasoningEmbedding` — the current score stores them so the API can serve the explanation and support semantic search without joining to history, and history preserves them for trend visualization and temporal semantic search. `ScoreDO` doesn't track `triggerEvent` — that's history-only context for understanding *why* a re-score happened. In the batch model, `triggerEvent` is `"scheduled"` or `"ad-hoc"` rather than event type names.

> **Score status field:** `status` is always one of `"scored"` (alignment computed with at least one shared topic) or `"no_overlap"` (zero shared topics between user and legislator). When `status = "no_overlap"`, the `aggregateScore` is meaningless (stored as 0.0) and the frontend should display a message like "This member has not voted on any of your topics" and prompt the user to answer questions on topics the member has voted on. `nonOverlappingTopics` lists the user's topics that the legislator has no votes on — the frontend uses this to suggest which additional survey topics would create overlap. `nonOverlappingTopics` is populated even when `status = "scored"` (partial overlap is common) so the frontend can always show "Your rep hasn't voted on: X, Y".

> **`MemberBillStanceTopicDO` — pre-materialized per-topic stances:** This child table of `member_bill_stances` stores per-topic stance information with LLM-generated reasoning and embeddings. Each row links a member's vote on a bill to a specific topic, with: `stanceDirection` (computed from vote position + analysis finding), `reasoning` (LLM explanation of why the vote indicates this stance), `reasoningEmbedding` (vector(1536) for semantic search), and `findingId` (FK to `bill_findings` as provenance/proof). This is populated by the stance materializer (§11.9) and consumed by the legislator profile builder (§11.2) and user-bill alignment (§11.10).

> **`UserBillAlignmentDO` / `UserAmendmentAlignmentDO` — pre-computed alignments:** These tables store per-bill, per-topic alignment between a user's stance and the bill's/amendment's stance findings. Pre-computed on a schedule by §11.10 to avoid redundant work during scoring. Each row includes LLM-generated reasoning and embedding for semantic search, plus a `findingId` FK for provenance.

### Question Bank DOs (all new — migration 004)

These DOs support the pre-tagged question bank for determining user political stances. Multiple-choice answers yield deterministic stance extraction (no LLM). Custom fill-in answers require LLM interpretation via the `preference-interpretation` profile (Component 9 §9.2).

| Class | Table | Key Columns |
|-------|-------|-------------|
| `QaQuestionDO` | `qa_questions` | questionId (String PK, e.g. "q-healthcare-medicare-expansion"), questionText, category, displayOrder, allowCustom, active, createdAt |
| `QaQuestionTopicDO` | `qa_question_topics` | questionId, topic (composite PK), agreeStance ("Progressive" or "Conservative"), weight (Float, 0.0-1.0 exclusive-inclusive) |
| `QaAnswerOptionDO` | `qa_answer_options` | questionId, optionValue (composite PK, e.g. "strongly_agree"), displayText, stanceMultiplier (Float -1.0 to 1.0), importanceSignal (Int 1-10), displayOrder |
| `QaUserResponseDO` | `qa_user_responses` | responseId (UUID), userId (UUID), questionId, selectedOption (Option[String]), customText (Option[String]), respondedAt |

> **Stance extraction logic:** For a multiple-choice response, the user's stance on each topic is computed as: `agreeStance` direction × `stanceMultiplier` × `weight`. For example, if `agreeStance = "Progressive"`, `stanceMultiplier = 0.6` (agree), and `weight = 1.0`, the user is moderately Progressive on that topic. A negative multiplier (disagree/strongly disagree) reverses the direction. This computation lives in Component 11 (scoring engine), not in the DOs.

> **Answer type constraint:** Exactly one of `selectedOption` or `customText` must be non-NULL per response. This is enforced by a CHECK constraint in migration 004 (`chk_answer_type`). The DO should model both as `Option[String]`.

> **Question lifecycle:** Questions can be soft-deleted via `active = false`. The `QaQuestionDO.active` field controls whether the question appears in the onboarding flow. Existing responses to deactivated questions are preserved.

### Acceptance Criteria — 1.5 User Domain Objects

**Functional:**
1. `UserDO` fields match all columns in the `users` table. `userId` is `java.util.UUID`.
2. `UserPreferenceDO` fields match all columns in the `user_preferences` table. `importance` is `Int` constrained 1-10.
3. `MemberBillStanceDO` fields match all columns in the `member_bill_stances` table. `topics` is `List[String]` (maps to PostgreSQL `TEXT[]`). `amendmentId` is `Option[String]`.
4. All user, scoring infrastructure, and score DOs have Doobie auto-derived `Read` and `Write` instances.
5. `ScoreDO` fields match all columns in the `scores` table: userId, memberId (composite PK), aggregateScore, status, lastUpdated, llmModel, totalBills, totalVotes, nonOverlappingTopics, reasoning, reasoningEmbedding.
6. `ScoreCongressDO` fields match all columns in `score_congress`: userId, memberId, congress (composite PK), overallScore, billsConsidered, votesAnalyzed.
7. `ScoreHistoryDO` includes `reasoning: Option[String]` (column added in migration 002) and `reasoningEmbedding: Option[Array[Float]]` (column added in migration 008) for storing the LLM-generated explanation and its embedding. Also includes `status: String` ("scored" or "no_overlap").
8. `ScoreHistoryCongressDO` fields match all columns in `score_history_congress`: scoreId, congress (composite PK), overallScore, billsConsidered, votesAnalyzed.
9. `ScoreHistoryHighlightDO` fields map directly to `AlignmentScoreOutput.highlights` entries (billId, topic, stance, vote, alignment).
10. All score Float fields are constrained to 0.0-1.0 range.

**Scoring Infrastructure DOs:**
10a. `UserLegislatorPairingDO` fields match all columns in `user_legislator_pairings`: userId (UUID), memberId, state, district (Option[Int]), chamber, pairedAt, validatedAt.
10b. `MemberBillStanceTopicDO` fields match all columns in `member_bill_stance_topics`: id (UUID PK), memberId, billId, voteId, topic, stanceDirection, reasoning (Option[String]), reasoningEmbedding (Option[Array[Float]]), findingId (Option[UUID]), confidence (Option[Double]), conceptSummary (Option[String]), createdAt.
10c. `UserBillAlignmentDO` fields match all columns in `user_bill_alignments`: userId (UUID), billId, topic, userStanceScore, billStanceDirection, alignmentScore, reasoning (Option[String]), reasoningEmbedding (Option[Array[Float]]), findingId (Option[UUID]), computedAt.
10d. `UserAmendmentAlignmentDO` fields match all columns in `user_amendment_alignments`: userId (UUID), amendmentId, billId (Option[String]), topic, userStanceScore, amendmentStanceDirection, alignmentScore, reasoning (Option[String]), reasoningEmbedding (Option[Array[Float]]), findingId (Option[UUID]), computedAt.
10e. `StanceMaterializationStatusDO` fields match all columns in `stance_materialization_status`: billId (String PK), hasVotes, hasAnalysis, allPassesCompleted, votesUpdatedAt (Option[Instant]), analysisCompletedAt (Option[Instant]), stancesMaterializedAt (Option[Instant]), lastScoringRunAt (Option[Instant]).
10f. `MemberBillStanceTopicDO.stanceDirection` is `String` with valid values "Progressive", "Conservative", "Neutral".
10g. `MemberBillStanceTopicDO.findingId` is `Option[UUID]` FK to `bill_findings` — provides provenance for the stance determination.
10h. `UserBillAlignmentDO.alignmentScore` and `UserAmendmentAlignmentDO.alignmentScore` are `Double` in range 0.0-1.0.
10i. All embedding fields (`reasoningEmbedding`) are `Option[Array[Float]]` representing vector(1536).

**Question Bank DOs:**
11. `QaQuestionDO` fields match all columns in `qa_questions`. `questionId` is `String` (natural key, not UUID). `allowCustom` and `active` are `Boolean`.
12. `QaQuestionTopicDO` fields match all columns in `qa_question_topics`. Composite PK `(questionId, topic)`. `agreeStance` is `String` (values: "Progressive", "Conservative"). `weight` is `Float`.
13. `QaAnswerOptionDO` fields match all columns in `qa_answer_options`. Composite PK `(questionId, optionValue)`. `stanceMultiplier` is `Float` in range -1.0 to 1.0. `importanceSignal` is `Int` in range 1-10.
14. `QaUserResponseDO` fields match all columns in `qa_user_responses`. `responseId` is `UUID`. Both `selectedOption` and `customText` are `Option[String]`.
15. All Q&A DOs have Doobie auto-derived `Read` and `Write` instances.
