# 1.2 Legislative DOs

> Part of [Component 1 — repcheck-shared-models](../01-SHARED-MODELS.md)

### Classes to Migrate and Expand

| Existing Class | Migration Notes |
|---------------|-----------------|
| `LegislativeBillDO` | Rename to `BillDO`. Expand fields to match full `bills` table schema (see below). |
| `BillTextLinksDO` | Migrate as-is for text version storage. |
| `LegislativeBillDTO.toDO` | Update to produce expanded `BillDO`. |

**`BillDO` expanded fields** (all columns from `bills` table):
- billId (PK), congress, billType, number, title, originChamber, originChamberCode
- introducedDate, policyArea, latestActionDate, latestActionText
- constitutionalAuthorityText, sponsorBioguideId
- textUrl, textFormat, textVersionType, textDate, textContent, textEmbedding (vector)
- summaryText, summaryActionDesc, summaryActionDate
- updateDate, updateDateIncludingText, legislationUrl, apiUrl
- createdAt, updatedAt

> **Bill text lifecycle:** Bill metadata arrives from Congress.gov before text is available. `textUrl`, `textFormat`, `textVersionType`, `textDate`, `textContent`, and `textEmbedding` are all `Option` fields — `None` on initial ingestion. Text versions are **immutable once published**; when a bill advances through legislative stages (committee, floor passage, enrollment), a new text version is published rather than modifying the existing one. The `updateDateIncludingText` field changes whenever a new text version appears, distinguishing text updates from metadata-only updates. The bills-pipeline (Component 4) uses `updateDateIncludingText` to detect bills that need text fetching:
> - **No text yet:** `textUrl` is `None` → bill needs text check on every sync run
> - **Has text, not final:** `textUrl` is set, `textVersionType` is not `"ENR"` → re-check if `updateDateIncludingText` changed (new version may exist)
> - **Has enrolled text:** `textVersionType` is `"ENR"` → text is final, no further checks needed
>
> **`textVersionType` stores the version code** (e.g., `"IH"`, `"ENR"`), not the full name (e.g., "Introduced in House", "Enrolled Bill"). This matches the `TextVersionCode` enum values from §1.4 and allows direct comparison with `BillTextVersionDO.versionCode`.
>
> When new text is detected, a `bill.text.available` event is emitted, triggering re-analysis by the bill-analysis-pipeline.

### Conversion Methods (`toDO`)

Each detail-level DTO provides a conversion method returning `Either[String, ...]`:

| DTO | Produces | Notes |
|-----|----------|-------|
| `BillListItemDTO.toDO` | `Either[String, BillDO]` | Partial BillDO (list-level fields only, detail fields null/None) |
| `BillDetailDTO.toDO` | `Either[String, BillConversionResult]` | `BillConversionResult` contains: `BillDO` + `List[BillCosponsorDO]` + `List[BillSubjectDO]` |
| `MemberDetailDTO.toDO` | `Either[String, MemberConversionResult]` | Contains: `MemberDO` + `List[MemberTermDO]` + `List[MemberPartyHistoryDO]` |
| `VoteMembersDTO.toDO` | `Either[String, VoteConversionResult]` | Contains: `VoteDO` + `List[VotePositionDO]`. Works for both House (from JSON) and Senate (from XML after LIS resolution). |
| `SenateVoteXmlDTO.toDO` | `Either[String, VoteMembersDTO]` | Converts Senate XML structure to unified `VoteMembersDTO`. Requires `Map[String, String]` (lisMemberId → bioguideId) for member resolution. Returns `Left` if any member ID cannot be resolved. |
| `AmendmentDetailDTO.toDO` | `Either[String, AmendmentDO]` | Single DO |

### Acceptance Criteria — 1.2 Legislative DOs

**Functional:**
1. Every DO case class has fields matching every column in its corresponding database table (column names converted to camelCase).
2. Every DO has Doobie auto-derived `Read` and `Write` instances (no manual `toPojo` methods).
3. `BillDO` contains all 27 non-audit columns from the `bills` table. `textEmbedding` is `Option[Array[Float]]` to represent pgvector.
4. `toDO` conversions validate required fields and return `Left` with a descriptive message when a required field is missing or malformed.
5. `BillDetailDTO.toDO` produces a `BillConversionResult` containing one `BillDO`, zero or more `BillCosponsorDO`, and zero or more `BillSubjectDO`.
6. `MemberDetailDTO.toDO` produces a `MemberConversionResult` containing one `MemberDO`, zero or more `MemberTermDO`, and zero or more `MemberPartyHistoryDO`.
7. `VoteMembersDTO.toDO` produces a `VoteConversionResult` containing one `VoteDO` and one `VotePositionDO` per member result. This works identically for House and Senate sources — the chamber distinction is in the `VoteDO.chamber` field.
8. `SenateVoteXmlDTO.toDO` accepts a LIS-to-bioguide mapping and produces a unified `VoteMembersDTO`. If any `lisMemberId` has no mapping, it returns `Left` listing the unresolved IDs.
9. `VoteDO.voteId` is constructed as `{congress}-{chamber}-{rollNumber}` matching the natural key in the schema.
10. `BillDO.billId` is constructed as `{congress}-{billType}-{number}` matching the natural key in the schema.
11. `AmendmentDO.amendmentId` is constructed as `{congress}-{amendmentType}-{number}` matching the natural key in the schema.
12. `LisMemberMappingDO` has fields matching the `lis_member_mapping` table: lisMemberId (String), memberId (String, bioguideId), lastVerified (Instant).
13. `MemberHistoryDO` mirrors `MemberDO` fields plus `historyId` (UUID) and `archivedAt` (Instant). Used by the archive-before-overwrite pattern in Component 5.
14. `MemberTermHistoryDO` mirrors `MemberTermDO` fields plus `historyId` (UUID, FK → `member_history.history_id`). Shares `historyId` with the corresponding `MemberHistoryDO` row to link a member snapshot with its term snapshot.

**Error Handling:**
11. `toDO` never throws. All failures return `Left[String]` with the specific field that failed and why.
12. Each conversion failure type has a unique exception class (e.g., `BillConversionFailed`, `MemberConversionFailed`, `VoteConversionFailed`, `AmendmentConversionFailed`).

**Test Requirements:**
13. Unit test per `toDO` method with fully populated DTO — verify all DO fields are correctly mapped.
14. Unit test per `toDO` method with each required field missing — verify `Left` with descriptive message.
15. Unit test per `toDO` method verifying natural key construction (billId, voteId, amendmentId).
16. Test that `BillDetailDTO.toDO` fans out sponsors into `BillCosponsorDO` list correctly, including empty sponsor list.
17. Test that `MemberDetailDTO.toDO` fans out terms and partyHistory correctly.
18. Test that `VoteMembersDTO.toDO` produces one `VotePositionDO` per member result entry.
19. Test that `SenateVoteXmlDTO.toDO` with a complete LIS mapping produces a valid `VoteMembersDTO` with `chamber = Senate`.
20. Test that `SenateVoteXmlDTO.toDO` with a missing LIS mapping returns `Left` listing the unresolved `lisMemberId` values.
21. Test that `LisMemberMappingDO` round-trips through Doobie `Read`/`Write`.
