# 9.2 User Scoring Profiles

> Part of [Component 9 — Prompt Engine Users](../09-PROMPT-ENGINE-USERS.md)

## Overview

Defines the scoring profiles and the user-specific block catalog. Each profile is a `PromptProfile` (Component 1 §1.7) — a named chain of `StageConfig` entries that specify which blocks to assemble and in what order.

Unlike Component 8 (which has decomposition + 3 analysis passes), Component 9 has **3 scoring profiles** at different cost/detail tiers, plus an optional **preference interpretation** utility profile.

## Scoring Profiles

### `full-alignment` — Comprehensive Scoring

The primary scoring profile. Produces a complete `AlignmentScoreOutput` (§1.6) with per-topic scores, highlights, and reasoning.

```yaml
name: "full-alignment"
chain:
  - stage: "system"
    blocks: ["base-scoring-analyst"]
    weight: 1.0
  - stage: "persona"
    blocks: ["plain-language-explainer"]
    weight: 0.9
  - stage: "lens"
    blocks: ["topic-alignment-lens", "voting-consistency-lens"]
    weight: 0.8
  - stage: "context"
    blocks: ["user-preferences", "voting-record", "bill-analyses", "legislator-profile"]
    weight: 1.0
  - stage: "guardrails"
    blocks: ["fairness-constraint", "no-party-bias"]
    weight: 1.0
  - stage: "output"
    blocks: ["full-alignment-schema"]
    weight: 1.0
```

**Used by**: Component 11 as the default scoring profile for all scoring runs.

**Expected model tier**: Sonnet — needs nuanced reasoning about alignment between user values and legislator voting patterns.

### `topic-breakdown` — Lightweight Topic Scoring

A lighter profile that produces per-topic scores without detailed reasoning or highlights. Faster and cheaper for batch re-scoring.

```yaml
name: "topic-breakdown"
chain:
  - stage: "system"
    blocks: ["base-scoring-analyst"]
    weight: 1.0
  - stage: "lens"
    blocks: ["topic-alignment-lens"]
    weight: 0.8
  - stage: "context"
    blocks: ["user-preferences", "voting-record", "bill-analyses"]
    weight: 1.0
  - stage: "guardrails"
    blocks: ["fairness-constraint"]
    weight: 1.0
  - stage: "output"
    blocks: ["topic-breakdown-schema"]
    weight: 1.0
```

**Used by**: Scoring Pipeline (§11.7) for batch re-scoring when many users need updated scores. Selected when score delta is moderate.

**Expected model tier**: Haiku — classification-level reasoning is sufficient for topic scores without explanations.

**Differences from `full-alignment`**:
- No Persona stage (no natural-language explanation needed)
- Only one lens (no voting-consistency cross-check)
- No `legislator-profile` context block (just topic scores from votes)
- Output schema requests topic scores only, no highlights or reasoning

### `quick-score` — Rapid Aggregate Score

The lightest profile. Produces a single aggregate alignment score with a brief one-sentence explanation. For immediate feedback when a user updates their profile.

```yaml
name: "quick-score"
chain:
  - stage: "system"
    blocks: ["base-scoring-analyst"]
    weight: 1.0
  - stage: "context"
    blocks: ["user-preferences", "voting-record"]
    weight: 1.0
  - stage: "guardrails"
    blocks: ["fairness-constraint"]
    weight: 1.0
  - stage: "output"
    blocks: ["quick-score-schema"]
    weight: 1.0
```

**Used by**: Scoring Pipeline (§11.7) when the score delta from the previous run is below the configured threshold — provides rapid, low-cost feedback for small changes.

**Expected model tier**: Haiku — simple aggregation is sufficient.

**Differences from `full-alignment`**:
- No Persona, no Lens stages
- Only `user-preferences` and `voting-record` context (no bill analyses, no legislator profile)
- Output schema requests aggregate score + one-sentence reasoning only

### `preference-interpretation` — Utility Profile

Interprets a user's Q&A responses to produce structured topic preferences. This is not a scoring profile — it converts raw user answers into the `UserPreferenceDO` structure (topic, stance, importance).

```yaml
name: "preference-interpretation"
chain:
  - stage: "system"
    blocks: ["preference-interpreter"]
    weight: 1.0
  - stage: "context"
    blocks: ["user-qa-responses"]
    weight: 1.0
  - stage: "guardrails"
    blocks: ["no-party-bias"]
    weight: 1.0
  - stage: "output"
    blocks: ["preference-output-schema"]
    weight: 1.0
```

**Used by**: Component 11 (or a future user profile pipeline) when a user completes the onboarding questionnaire.

**Expected model tier**: Haiku — straightforward extraction from Q&A responses.

## Block Catalog

### System Blocks

| Block Name | Purpose |
|-----------|---------|
| `base-scoring-analyst` | Core system role: "You are an alignment analyst. You evaluate how well a legislator's voting record matches a citizen's political preferences. You are strictly nonpartisan." |
| `preference-interpreter` | System role for Q&A interpretation: "You extract structured political preferences from a user's answers to policy questions." |

### Persona Blocks

| Block Name | Purpose |
|-----------|---------|
| `plain-language-explainer` | "Explain alignment scores in everyday language. Use specific bill examples. Avoid political jargon. A voter with no policy expertise should understand your explanation." |
| `data-driven-analyst` | "Present alignment analysis with supporting data. Reference specific vote counts, bill numbers, and topic match percentages. Suitable for politically engaged users." |

### Lens Blocks

| Block Name | Purpose |
|-----------|---------|
| `topic-alignment-lens` | Score alignment per topic: for each topic the user cares about, evaluate how the legislator's votes on bills in that topic area align with the user's stated stance. Weight by user's importance rating. |
| `voting-consistency-lens` | Evaluate consistency of the legislator's voting pattern over time within each topic. Flag flip-flops or evolving positions. This lens is additive — it supplements topic alignment with a consistency dimension. |
| `bipartisan-lens` | Evaluate the legislator's tendency toward bipartisan legislation. Relevant when the user values pragmatism or cross-party cooperation (indicated by "bipartisan" stance on topics). |

### Context Blocks

| Block Name | Placeholder | Content Description |
|-----------|-------------|---------------------|
| `user-preferences` | `{{user_preferences}}` | User's political preferences: topics, stances, and importance weights from `user_preferences` table |
| `voting-record` | `{{voting_record}}` | Legislator's voting record: bill-by-bill votes with topics and analysis summaries, from `member_bill_stances` joined with `bill_analyses` |
| `bill-analyses` | `{{bill_analyses}}` | Detailed bill analysis summaries for bills the legislator voted on, from `bill_analyses` + `bill_concept_summaries`. Only included in `full-alignment` profile. |
| `legislator-profile` | `{{legislator_profile}}` | Legislator's profile: name, party, state, district, committee memberships. Provides context but must not influence scoring (per guardrails). |
| `user-qa-responses` | `{{user_qa_responses}}` | Raw Q&A responses from user onboarding. Only used by `preference-interpretation` profile. |

### Guardrail Blocks

| Block Name | Purpose |
|-----------|---------|
| `fairness-constraint` | "Score based on voting record alignment with stated user preferences ONLY. Do not penalize or reward legislators for party membership, seniority, media coverage, or personal characteristics." |
| `no-party-bias` | "Do not assume that a party label implies policy positions. Evaluate each vote on its own merits relative to the user's stated preferences. Two legislators in the same party can have very different alignment scores." |

### Output Blocks

| Block Name | Schema Referenced | Fields Requested |
|-----------|------------------|-----------------|
| `full-alignment-schema` | `AlignmentScoreOutput` (§1.6) | topicScores (with explanation per topic), overallScore, highlights (notable bill alignments), reasoning (paragraph explaining the score) |
| `topic-breakdown-schema` | Subset of `AlignmentScoreOutput` | topicScores (score only, no explanation), overallScore |
| `quick-score-schema` | Subset of `AlignmentScoreOutput` | overallScore, reasoning (one sentence) |
| `preference-output-schema` | List of `{topic, stance, importance}` objects | Structured extraction from Q&A responses |

## Key Design Decisions

- **Three scoring tiers, not one** — mirrors Component 8's tiered analysis approach. Full scoring is expensive (Sonnet, detailed context). For batch re-scoring of hundreds of user-legislator pairs after a vote event, `topic-breakdown` with Haiku is sufficient and 10x cheaper.
- **Preference interpretation is a profile, not a separate engine** — converting Q&A responses to structured preferences uses the same prompt assembly mechanism. No reason to build a separate system.
- **Legislator profile is context, not a scoring input** — the `legislator-profile` block provides context for the LLM to reference ("Senator X voted...") but the guardrails explicitly prohibit scoring based on party, seniority, or personal characteristics.
- **No bill text in scoring prompts** — the scoring engine never sends raw bill text to the LLM. It sends bill *analysis summaries* (from Component 10's output). This keeps scoring prompts within context window limits even for users whose representatives voted on hundreds of bills.

## Acceptance Criteria

| Criterion | Test |
|-----------|------|
| `full-alignment` profile chains: system → persona → lens(2) → context(4) → guardrails(2) → output | Load profile → verify 6 stages, correct block counts per stage |
| `topic-breakdown` profile chains: system → lens(1) → context(3) → guardrails(1) → output | Load profile → verify 5 stages, no persona stage |
| `quick-score` profile chains: system → context(2) → guardrails(1) → output | Load profile → verify 4 stages, no persona or lens |
| `preference-interpretation` profile chains: system → context(1) → guardrails(1) → output | Load profile → verify 4 stages, uses `preference-interpreter` system block |
| All profiles reference only blocks that exist in the block catalog | Cross-reference each profile's block names against the catalog → no orphan references |
| All context blocks declare their placeholder tokens | Each context block's content contains the expected `{{placeholder}}` token |
| `full-alignment` profile includes `legislator-profile` context block | Profile chain inspection |
| `topic-breakdown` profile does NOT include `legislator-profile` context block | Profile chain inspection |
| `quick-score` profile does NOT include `bill-analyses` or `legislator-profile` context blocks | Profile chain inspection |
| All profiles include at least one guardrail block | Every profile has guardrails stage with ≥1 block |
| Output schema block names match the expected output schema reference | Each output block's content references the correct `AlignmentScoreOutput` fields |
| `UserScoringProfiles.allProfiles` returns all 4 profile names | `allProfiles` → `["full-alignment", "topic-breakdown", "quick-score", "preference-interpretation"]` |
| `UserScoringProfiles.scoringProfiles` returns only the 3 scoring profiles (not preference-interpretation) | `scoringProfiles` → `["full-alignment", "topic-breakdown", "quick-score"]` |
| Profile weights are all in 0.0-1.0 range | Every `StageConfig.weight` in every profile ∈ [0.0, 1.0] |
