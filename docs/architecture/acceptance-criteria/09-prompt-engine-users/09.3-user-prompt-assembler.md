# 9.3 User Prompt Assembler

> Part of [Component 9 — Prompt Engine Users](../09-PROMPT-ENGINE-USERS.md)

## Overview

User-specific implementation of `ChainAssembler` (Component 1 §1.7) that handles context injection — serializing structured user/legislator data into prompt-friendly text and injecting it into Context stage blocks at runtime. This is the main entry point consumed by the scoring-pipeline (Component 11).

The assembler handles richly structured context: a user has multiple topic preferences with different importance weights, a legislator has voted on dozens to hundreds of bills across multiple congresses, and each bill may have analysis summaries with per-concept breakdowns.

## Classes

### `UserPromptAssembler[F[_]]` — Trait + Impl

```
trait UserPromptAssembler[F[_]] {
  def assembleForScoring(
    profileName: String,
    context: UserScoringContext,
    version: Option[String] = None     // None → use config default
  ): F[String]

  def assembleForPreferenceInterpretation(
    context: PreferenceInterpretationContext,
    version: Option[String] = None
  ): F[String]
}
```

Both methods follow the same flow:
1. Resolve version: `version.getOrElse(config.defaultVersion)`
2. Load profile via `ProfileLoader.loadProfile(profileName, v)`
3. Collect all block names from the profile's chain
4. Load all blocks via `BlockLoader.loadBlocksByNames(names, v)`
5. Serialize structured context into prompt text via `UserContextInjector`
6. Delegate to `ChainAssembler.assemble(profile, blocks, contextMap)` (Component 1 §1.7)
7. Return the assembled prompt string

**`assembleForScoring`** is the primary method used by Component 11. It takes `UserScoringContext` containing all structured context — the assembler serializes and injects only those values referenced by the profile's context blocks.

**`assembleForPreferenceInterpretation`** is used when interpreting user Q&A responses. It takes `PreferenceInterpretationContext` which carries the raw Q&A answers.

## Structured Context Types

### `UserScoringContext`

Runtime context for scoring prompts. Contains everything the LLM needs to compute alignment between a user's preferences and a legislator's voting record.

```
final case class UserScoringContext(
  userId: String,
  memberId: String,
  preferences: List[UserPreference],
  votingRecord: List[VotingSummary],
  billAnalyses: List[BillAnalysisSummary],
  legislatorProfile: Option[LegislatorProfile]
)
```

### `UserPreference`

A single topic preference from the user's profile. Maps to `UserPreferenceDO` (Component 1 §1.5).

```
final case class UserPreference(
  topic: String,                            // e.g., "healthcare", "defense"
  stance: String,                           // "support", "oppose", "neutral"
  importance: Int                           // 1-10 weight
)
```

### `VotingSummary`

A summary of one legislator vote on a bill, enriched with analysis topics. Maps to `MemberBillStanceDO` (Component 1 §1.5).

```
final case class VotingSummary(
  billId: String,
  billTitle: String,
  congress: Int,
  voteDate: String,                         // ISO-8601 date string
  position: String,                         // "Yea", "Nay", "Present", "Not Voting"
  voteType: String,                         // "Passage", "Cloture", "Amendment", etc.
  topics: List[String],                     // from bill_analyses.topics
  analysisSummary: Option[String]           // Pass 1 overall summary from bill_analyses, if available
)
```

> **`analysisSummary`** is the bill's overall Pass 1 summary from `BillAnalysisDO.summary`. It gives the LLM context about what the bill does without requiring the full bill text. For bills not yet analyzed, this is `None` and the LLM scores based on topics and vote position alone.

### `BillAnalysisSummary`

A deeper analysis summary for bills where the legislator voted. Only included in the `full-alignment` profile — not in `topic-breakdown` or `quick-score`. Derived from the analysis layer tables.

```
final case class BillAnalysisSummary(
  billId: String,
  billTitle: String,
  overallSummary: String,                   // from BillAnalysisDO.summary
  topics: List[String],                     // from BillAnalysisDO.topics
  keyPoints: List[String],                  // from BillAnalysisDO.keyPoints
  stances: List[TopicStanceSummary],        // from bill_findings where findingTypeId = 'stance_detection'
  porkFindings: List[String],               // summary text from bill_findings where findingTypeId = 'pork'
  impactSummary: Option[String]             // summary from bill_findings where findingTypeId = 'impact_analysis'
)
```

> **This is a pre-assembled summary, not raw analysis data.** Component 11 fetches analysis results from AlloyDB and constructs `BillAnalysisSummary` objects before passing them to the prompt engine. The prompt engine never touches the database.

### `TopicStanceSummary`

Summary of a bill's stance on a topic, from Pass 2 stance detection findings.

```
final case class TopicStanceSummary(
  topic: String,
  stance: String,                           // "Conservative", "Progressive", "Bipartisan", "Neutral"
  confidence: Double
)
```

### `LegislatorProfile`

Basic legislator profile for context. Only included in `full-alignment` — the LLM uses it for reference (e.g., naming the legislator in explanations) but guardrails prohibit scoring based on party or personal characteristics.

```
final case class LegislatorProfile(
  memberId: String,
  name: String,                             // display name
  party: String,
  state: String,
  district: Option[Int],
  chamber: String                           // "House" or "Senate"
)
```

### `PreferenceInterpretationContext`

Runtime context for interpreting user Q&A responses.

```
final case class PreferenceInterpretationContext(
  userId: String,
  qaResponses: List[QaResponse]
)

final case class QaResponse(
  questionId: String,
  questionText: String,
  answer: String
)
```

## Context Serialization

### `UserContextInjector` — Object

Serializes structured context types into prompt-friendly text and injects them into Context stage blocks.

```
object UserContextInjector {
  def serialize(context: UserScoringContext): Map[String, String]
  def serialize(context: PreferenceInterpretationContext): Map[String, String]

  def inject(
    blocks: Map[String, InstructionBlock],
    contextMap: Map[String, String]
  ): Either[String, Map[String, InstructionBlock]]
}
```

**`serialize(UserScoringContext)`** produces:

| Placeholder | Serialization |
|-------------|--------------|
| `{{user_preferences}}` | Each `UserPreference` formatted as a row: topic, stance, importance (sorted by importance descending). Presented as a structured list so the LLM sees the user's priorities clearly. |
| `{{voting_record}}` | Each `VotingSummary` formatted as a chronological entry: bill ID, title, vote date, position, topics, and analysis summary (if available). Grouped by congress (most recent first). |
| `{{bill_analyses}}` | Each `BillAnalysisSummary` formatted as a titled section: bill title, overall summary, key points, stance findings, pork findings, impact summary. Only present in `full-alignment` profile. |
| `{{legislator_profile}}` | `LegislatorProfile` formatted as a brief header: name, party, state/district, chamber. Only present in `full-alignment` profile. |

**`serialize(PreferenceInterpretationContext)`** produces:

| Placeholder | Serialization |
|-------------|--------------|
| `{{user_qa_responses}}` | Each `QaResponse` formatted as: question number, question text, user's answer. Numbered list. |

**Empty lists** serialize to explicit "None" indicators:
- Empty preferences: `"No preferences have been set by this user."`
- Empty voting record: `"No voting record available for this legislator."`
- Empty bill analyses: `"No bill analyses available."`

**`inject`** behavior: identical to Component 8 §8.3.

### Serialization Format

The serializer produces structured, readable text — not JSON. Example for preferences:

```
## User Preferences (sorted by importance)

1. Healthcare — Support (Importance: 9/10)
2. Environment — Support (Importance: 8/10)
3. Defense — Oppose (Importance: 7/10)
4. Immigration — Neutral (Importance: 5/10)
```

Example for voting record:

```
## Voting Record — 118th Congress (most recent)

1. HR-1234 "American Healthcare Act" — Yea (Passage, 2024-03-15)
   Topics: healthcare, insurance, federal-spending
   Summary: Expands Medicare coverage to include dental and vision...

2. S-567 "Climate Investment Act" — Nay (Passage, 2024-02-28)
   Topics: environment, energy, tax-credits
   Summary: Establishes $500B in clean energy tax credits...

---

## Voting Record — 117th Congress

3. HR-890 "Infrastructure Investment Act" — Yea (Passage, 2023-11-05)
   Topics: infrastructure, transportation, broadband
   Summary: Authorizes $1.2T in infrastructure spending...
```

### `UserContextFormatter` — Trait

```
trait UserContextFormatter {
  def formatPreferences(preferences: List[UserPreference]): String
  def formatVotingRecord(votes: List[VotingSummary]): String
  def formatBillAnalyses(analyses: List[BillAnalysisSummary]): String
  def formatLegislatorProfile(profile: LegislatorProfile): String
  def formatQaResponses(responses: List[QaResponse]): String
}
```

Default implementation uses markdown-style formatting. Swappable for testing or if a different format yields better LLM results.

## Key Design Decisions

- **Structured context, not strings** — `UserScoringContext` carries typed lists of preferences, votes, and analyses. This allows the assembler to format them optimally for the LLM.
- **`UserContextFormatter` is swappable** — same pattern as Component 8 §8.3. Different LLMs may respond better to different formatting.
- **Voting record grouped by congress** — most recent congress first, so the LLM weights recent votes more naturally. The `full-alignment` profile's `voting-consistency-lens` block may explicitly instruct the LLM on how to weight across congresses.
- **Bill analyses only in `full-alignment`** — the `topic-breakdown` and `quick-score` profiles don't include detailed analysis summaries. The voting record's `analysisSummary` field (one-liner from Pass 1) is sufficient for lightweight scoring.
- **Preferences sorted by importance** — highest importance first, so the LLM sees the user's top priorities at the start of the context.
- **No raw bill text** — scoring prompts contain analysis summaries, not bill text. This keeps prompts within context window limits.
- **No LLM calls** — the assembler produces a string. It never calls the LLM.
- **No data fetching** — the assembler receives pre-fetched context. Component 11 fetches the data and passes it in.
- **Version override per request** — same as Component 8, the scoring pipeline can test a new prompt version for a specific run.

```
final case class ContextInjectionFailed(
```

## Acceptance Criteria

| Criterion | Test |
|-----------|------|
| `assembleForScoring` loads profile and blocks, injects context, returns assembled prompt | Mock loader → provide `UserScoringContext` with 5 preferences and 10 votes → verify assembled string contains all preference topics and vote entries |
| `assembleForScoring` uses config default version when `version = None` | No version override → `ProfileLoader` called with `config.defaultVersion` |
| `assembleForScoring` uses override version when provided | `version = Some("2.0.0")` → `ProfileLoader` called with `"2.0.0"` |
| `assembleForPreferenceInterpretation` works with `preference-interpretation` profile | Provide `PreferenceInterpretationContext` with 5 Q&A responses → prompt contains all questions and answers |
| Preferences serialized sorted by importance (descending) | Preferences with importance [3, 9, 5] → output order: 9, 5, 3 |
| Preferences include topic, stance, and importance | Each preference entry → formatted with all 3 fields |
| Voting record grouped by congress (most recent first) | Votes from congresses 117 and 118 → 118th congress section appears first |
| Voting record includes bill title, position, topics, and summary | Each vote entry → all fields present in serialized output |
| Voting record with `analysisSummary = None` serializes without summary line | Vote without analysis → still serialized with bill ID, position, topics |
| Bill analyses serialized with overall summary, key points, and findings | 3 `BillAnalysisSummary` items → 3 titled sections with all sub-fields |
| Bill analyses not present in `topic-breakdown` profile | Assemble `topic-breakdown` → no `{{bill_analyses}}` content in output |
| Bill analyses not present in `quick-score` profile | Assemble `quick-score` → no `{{bill_analyses}}` content in output |
| Legislator profile serialized as brief header | `LegislatorProfile` → output contains name, party, state, chamber |
| Legislator profile not present in `topic-breakdown` or `quick-score` profiles | Assemble → no `{{legislator_profile}}` content |
| Empty preferences list produces explicit "None" text | `preferences = List()` → "No preferences have been set by this user." |
| Empty voting record produces explicit "None" text | `votingRecord = List()` → "No voting record available for this legislator." |
| Empty bill analyses list produces "None" text | `billAnalyses = List()` → "No bill analyses available." |
| Q&A responses serialized as numbered list with question and answer | 5 `QaResponse` items → 5 numbered entries |
| `full-alignment` prompt contains preferences, voting record, analyses, and legislator profile | Assemble `full-alignment` with all context populated → all 4 context sections present |
| `quick-score` prompt contains only preferences and voting record | Assemble `quick-score` → only 2 context sections present |
| Assembled prompt stages appear in correct order | Any profile → System before Persona before Lens before Context before Guardrails before Output |
| Weight translation applied to block content | Block with `weight: 1.0` → content wrapped with "You MUST" emphasis |
| Weight translation for low-weight blocks | Block with `weight: 0.3` → content framed as advisory |
| `UserContextFormatter` is swappable | Provide custom formatter → serialized output uses custom format |
| Profile load failure propagates as `ProfileLoadFailed` | Mock loader throws → `ProfileLoadFailed` raised |
| Block load failure propagates as `BlockLoadFailed` | Mock loader throws → `BlockLoadFailed` raised |
| Context injection failure propagates as `ContextInjectionFailed` | Unresolvable placeholder → `ContextInjectionFailed` with placeholder name |
| User with single preference works | `preferences = List(onePreference)` → serialized as single entry |
| Legislator with 100+ votes serializes all of them | 100 votes → all 100 appear in serialized output |
| `UserContextFormatter` handles all 5 formatting methods | Custom formatter → all context types use custom format |
| `TopicStanceSummary` appears in bill analysis serialization | `BillAnalysisSummary` with stances → stance entries appear in output |
| Pork findings appear in bill analysis serialization | `BillAnalysisSummary` with pork findings → pork summaries appear |
