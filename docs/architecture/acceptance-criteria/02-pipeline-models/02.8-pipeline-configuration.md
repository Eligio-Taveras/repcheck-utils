# 2.8 Pipeline Configuration

> **Design rationale:** All pipeline configuration uses PureConfig auto-derivation. Vote type weights, committee attribution weights, and analysis pass routing are configurable per environment. From BEHAVIORAL_SPECS.md §2 (vote weights) and §3 (Batch Scoring Architecture).

### Types

| Class | Fields | Notes |
|-------|--------|-------|
| `VoteWeights` | passage: Double (1.0), conferenceReport: Double (1.0), cloture: Double (0.8), vetoOverride: Double (0.9), amendment: Double (0.5), committee: Double (0.4), motionToRecommit: Double (0.6), other: Double (0.5) | `forVoteType(vt: VoteType): Double` lookup method. |
| `CommitteeAttributionWeights` | chairman: Double (1.0), rankingMember: Double (0.7), viceChairman: Double (0.6), member: Double (0.4) | Mirrors `CommitteePosition.weight` from shared-models but configurable per environment. |
| `AnalysisPassConfig` | pass1Model: String ("claude-3-haiku"), pass2Model: String ("claude-3-sonnet"), pass3Model: String ("claude-3-opus"), pass1Enabled: Boolean (true), pass2Enabled: Boolean (true), pass3Enabled: Boolean (true) | Which LLM model runs each analysis pass. |
| `PipelineConfig[T]` | parallelism: Int (4), batchSize: Int (100), pageSize: Int (250), retry: RetryConfig, appConfig: T | Generic pipeline configuration root. `T` is a pipeline-specific app config type (e.g., `BillsPipelineAppConfig`, `ScoringAppConfig`). Requires `ConfigReader[T]` in implicit scope for PureConfig auto-derivation. `VoteWeights`, `CommitteeAttributionWeights`, and `AnalysisPassConfig` are NOT direct fields — they are building-block types that consuming pipelines compose into their own `T`. Each pipeline defines its own app config case class containing whichever building blocks it needs. |

### Acceptance Criteria — 2.8 Pipeline Configuration

**Functional:**
1. `VoteWeights` has PureConfig auto-derivation and loads from HOCON with sensible defaults.
2. `VoteWeights.forVoteType(vt: VoteType): Double` returns the correct weight for all 8 vote types. Unknown types default to `other` (0.5).
3. `CommitteeAttributionWeights` has PureConfig auto-derivation and defaults match `CommitteePosition.weight` from shared-models.
4. `AnalysisPassConfig` has PureConfig auto-derivation. All passes enabled by default.
5. `PipelineConfig[T]` is generic over a pipeline-specific app config type `T`. Common fields are `parallelism`, `batchSize`, `pageSize`, and `retry`. Pipeline-specific configuration is provided via `appConfig: T`.
6. `PipelineConfig[T]` has PureConfig auto-derivation. Derivation requires `ConfigReader[T]` in implicit scope for the pipeline-specific type.
7. All weight values are `Double` in 0.0-1.0 range.
8. Each consuming pipeline defines its own app config case class (e.g., `BillsPipelineAppConfig` containing `VoteWeights`; `ScoringAppConfig` containing `AnalysisPassConfig` + `CommitteeAttributionWeights`). These are not part of `PipelineConfig` itself.

**Test Requirements:**
9. Test `VoteWeights.forVoteType` for all 8 vote types — verify correct default weight for each.
10. Test `VoteWeights` loaded from HOCON with overridden values — verify overrides take effect.
11. Test `PipelineConfig[T]` loaded from a minimal HOCON string with a simple test app config — verify defaults applied for missing common fields.
12. Test `PipelineConfig[T]` loaded from a full HOCON string with a test app config — verify all common fields and `appConfig` populated.
13. Test `CommitteeAttributionWeights` defaults match `CommitteePosition.weight` values from shared-models.
14. Test that `PipelineConfig[T]` fails to derive when `ConfigReader[T]` is not in scope (compile-time check).
