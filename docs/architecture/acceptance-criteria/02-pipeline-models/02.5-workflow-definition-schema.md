# 2.5 Workflow Definition Schema

> **Design rationale:** Workflow definitions are stored in GCS as JSON files. They describe the complete Cloud Run Job specification for each step ‚Äî container settings, resource limits, execution settings, networking, volumes, identity, health checks, and metadata. The Launcher reads these at runtime and passes the full configuration to the Cloud Run Jobs API. This gives full control over launch parameters without redeploying infrastructure.

### Top-Level Structure

| Class | Fields | Notes |
|-------|--------|-------|
| `WorkflowDefinition` | workflowId: String, name: String, steps: List[WorkflowStepDefinition], schedule: Option[String] (cron expression) | The full workflow DAG template stored in GCS |

### Step Definition

| Class | Fields | Notes |
|-------|--------|-------|
| `WorkflowStepDefinition` | stepName: String (unique within workflow ‚Äî used as PK in `workflow_run_steps` and as reference in `dependencies`), triggeredBy: List[String] (event types that trigger this step), dependencies: List[String] (stepName values that must complete first), container: ContainerConfig, resources: ResourceConfig, execution: ExecutionConfig, networking: NetworkConfig, volumes: VolumeConfig, identity: IdentityConfig, healthChecks: HealthCheckConfig, metadata: MetadataConfig | Complete specification for launching one Cloud Run Job. `stepName` serves as both the human-readable name and the unique identifier ‚Äî no separate `stepId`. |

### Container Configuration

| Class | Fields | Notes |
|-------|--------|-------|
| `ContainerConfig` | image: String (container image URI), command: Option[List[String]] (override entrypoint), args: List[String] (supports macros like `{{run_id}}`), env: Map[String, String] (plain env vars), secretMounts: Map[String, String] (env var name ‚Üí Secret Manager resource path) | Secrets are mounted by Cloud Run natively ‚Äî the Launcher never sees secret values, only reference paths |

### Resource Configuration

| Class | Fields | Notes |
|-------|--------|-------|
| `ResourceConfig` | cpu: String (default "1" ‚Äî valid: "1", "2", "4", "6", "8"), memory: String (default "512Mi" ‚Äî up to "32Gi"), gpu: Option[String] (GPU type if needed) | Per-step resource allocation |

### Execution Configuration

| Class | Fields | Notes |
|-------|--------|-------|
| `ExecutionConfig` | timeout: String (duration, default "10m", max "168h"), maxRetries: Int (0-10, default 3), taskCount: Int (1-10000, default 1), parallelism: Int (default 1) | Cloud Run Job execution parameters |

### Network Configuration

| Class | Fields | Notes |
|-------|--------|-------|
| `NetworkConfig` | vpcConnector: Option[String] (VPC connector name for AlloyDB private IP), vpcEgress: Option[String] ("all-traffic" or "private-ranges-only"), cloudSqlConnections: List[String] (Cloud SQL instance names ‚Äî dev environment) | Networking for Cloud Run Job |

### Volume Configuration

| Class | Fields | Notes |
|-------|--------|-------|
| `VolumeConfig` | gcsVolumes: Map[String, String] (mount path ‚Üí GCS bucket), inMemoryVolumes: Map[String, String] (mount path ‚Üí size limit) | Storage mounts for the container |

### Identity Configuration

| Class | Fields | Notes |
|-------|--------|-------|
| `IdentityConfig` | serviceAccount: String | Which SA to run as: launcher_sa, ingestion_sa, analysis_sa, scoring_sa |

### Health Check Configuration

| Class | Fields | Notes |
|-------|--------|-------|
| `ProbeConfig` | initialDelaySeconds: Int (default 0, max 240), timeoutSeconds: Int (default 1, max 3600), periodSeconds: Int (default 10, max 240), failureThreshold: Int (default 3) | Shared config for startup and liveness probes |
| `HealthCheckConfig` | startupProbe: Option[ProbeConfig], livenessProbe: Option[ProbeConfig] | Container health checks |

### Metadata Configuration

| Class | Fields | Notes |
|-------|--------|-------|
| `MetadataConfig` | labels: Map[String, String] (cost tracking, environment tagging), annotations: Map[String, String] (additional metadata) | Cloud Run Job labels and annotations |

### Macro Resolution

The Launcher replaces macro tokens in `args` and `env` values before launching. Supported macros:

| Macro | Source | Example |
|-------|--------|---------|
| `{{run_id}}` | Workflow run ID | `"a1b2c3d4-..."` |
| `{{date}}` | Current date (ISO-8601) | `"2026-03-30"` |
| `{{timestamp}}` | Current timestamp (ISO-8601) | `"2026-03-30T12:00:00Z"` |
| `{{message.*}}` | Field extracted from triggering Pub/Sub message payload | `{{message.billId}}` ‚Üí `"hr1319-117"` |

> **Note:** The full set of macros will be expanded as implementation proceeds. The macro resolution framework is extensible ‚Äî new macros can be added without changing the workflow schema.

### Workflow Versioning

Workflow definitions are versioned in GCS using semver in filenames (e.g., `workflows/daily-ingestion-v1.2.0.json`). When a workflow run starts, the Launcher records the GCS path in `workflow_runs.workflow_definition_path`. This pins the run to the exact definition version used ‚Äî even if the workflow definition is updated in GCS while the run is in progress, the run continues using the version it started with. The Launcher reads the definition once at run start and does not re-read it.

### Acceptance Criteria ‚Äî 2.5 Workflow Definition Schema

**Functional:**
1. `WorkflowDefinition` has Circe `Encoder` and `Decoder` for reading/writing JSON files in GCS.
2. All nested config types (`ContainerConfig`, `ResourceConfig`, etc.) have Circe codecs.
3. `WorkflowDefinition` validates at construction: no circular dependencies in the step graph.
4. `WorkflowDefinition` validates at construction: all `stepName` values referenced in `dependencies` exist in the `steps` list.
5. `ContainerConfig.secretMounts` stores only Secret Manager resource paths ‚Äî never secret values.
6. `ResourceConfig.cpu` is validated against known values ("1", "2", "4", "6", "8").
7. `ResourceConfig.memory` is validated against format (number + unit: Mi, Gi).
8. `ExecutionConfig.maxRetries` is validated in range 0-10.
9. `ExecutionConfig.taskCount` is validated in range 1-10000.
10. `ProbeConfig` field ranges are validated (initialDelaySeconds 0-240, timeoutSeconds 1-3600, etc.).
11. All config types have sensible defaults ‚Äî a step with only `image` specified should be launchable with defaults for everything else.
12. `WorkflowStepDefinition.triggeredBy` lists the event types that cause this step to run.

**Error Handling:**
13. `WorkflowDefinition` construction returns `Either[String, WorkflowDefinition]` ‚Äî `Left` if circular dependencies, missing step references, or invalid config values.
14. Loading a malformed workflow JSON from GCS returns a descriptive error including the GCS path and the parsing failure.

**Test Requirements:**
15. Test `WorkflowDefinition` with valid DAG ‚Äî verify construction succeeds.
16. Test `WorkflowDefinition` with circular dependency (A ‚Üí B ‚Üí A) ‚Äî verify `Left` with descriptive error.
17. Test `WorkflowDefinition` with dependency referencing non-existent `stepName` ‚Äî verify `Left`.
18. Test `ResourceConfig` with invalid CPU value ‚Äî verify `Left`.
19. Test `ResourceConfig` with invalid memory format ‚Äî verify `Left`.
20. Test all config types with defaults only ‚Äî verify valid construction.
21. Circe round-trip test for `WorkflowDefinition` with all fields populated.
22. Circe round-trip test for `WorkflowDefinition` with minimal fields (defaults applied).
