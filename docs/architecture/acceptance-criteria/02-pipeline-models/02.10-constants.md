# 2.10 Constants

### Acceptance Criteria — 2.10 Constants

**Functional:**
1. `Tables` object contains a `val` for every table in the database schema including new tables: `workflow_runs`, `workflow_run_steps`, `bill_text_versions`, `bill_text_sections`, `bill_concept_groups`, `bill_concept_group_sections`, `bill_concept_summaries`, `bill_analysis_topics`, `bill_fiscal_estimates`, `committees`, `committee_members`, `bill_committee_referrals`, `member_history`, `member_term_history`, `qa_questions`, `qa_question_topics`, `qa_answer_options`, `qa_user_responses`, `user_topic_priorities`, `user_legislator_pairings`, `member_bill_stance_topics`, `user_bill_alignments`, `user_amendment_alignments`, `stance_materialization_status`.
2. Each constant's string value exactly matches the table name in the SQL schema (lowercase, underscored).
3. No table name string literal appears anywhere in pipeline code outside of the `Tables` object — enforced by code review.
4. `EventTypes` constants are defined in Section 2.1 and re-exported here for convenience. Includes `EventTypes.DecompositionCompleted = "bill.decomposition.completed"`, `EventTypes.ScoringUserRequested = "scoring.user.requested"`, `EventTypes.ScoringUserCompleted = "scoring.user.completed"`.

**Test Requirements:**
5. Test that every `Tables` constant has a non-empty, non-null string value.
6. Test that no two `Tables` constants have the same string value (no accidental duplicates).
7. Test that the set of table names in `Tables` matches the set of tables created by the migration SQL.
