# 2.2 Pipeline Execution Tracking

> **Design rationale:** Every pipeline run writes a `PipelineRunDO` record and per-item `ProcessingResultDO` records to AlloyDB. These provide observability, debugging, and retry tracking. Maps to `pipeline_runs` and `processing_results` tables.

### Status Enums

| Enum | Values | Notes |
|------|--------|-------|
| `PipelineStatus` | Running, Completed, CompletedWithErrors, Failed | Maps to `pipeline_runs.status` |
| `ResultStatus` | Succeeded, Failed, Skipped | Maps to `processing_results.status` |
| `PipelineTrigger` | Scheduled, Manual, Event | Maps to `pipeline_runs.trigger` |

### DOs

| Class | Table | Key Columns |
|-------|-------|-------------|
| `PipelineRunDO` | `pipeline_runs` | runId (UUID PK), pipelineName, status (PipelineStatus), startedAt, completedAt (Option), itemsProcessed (Int), itemsSucceeded (Int), itemsFailed (Int), errorSummary (Option[String]), snapshotPath (Option[String]), trigger (PipelineTrigger), createdAt |
| `ProcessingResultDO` | `processing_results` | resultId (UUID PK), runId (FK → pipeline_runs), correlationId (UUID), entityType (String), entityId (String — natural key), status (ResultStatus), errorMessage (Option[String]), errorClass (Option[String] — 'Transient' or 'Systemic'), processedAt |

### ProcessingResult (in-memory stream type)

`ProcessingResult` is the per-item result type used in pipeline streams (`Stream[F, ProcessingResult]`). It is distinct from `ProcessingResultDO` (which is the AlloyDB-persisted version). Every pipeline processor returns a stream of these.

```
enum ProcessingResult {
  case Succeeded(entityId: String, eventEmitted: Boolean = false)
  case Failed(entityId: String, reason: String)
  case Skipped(entityId: String, reason: String)
}
```

- `Succeeded.eventEmitted` tracks whether a downstream event was published for this item (e.g., `vote.recorded`). Used by `PipelineRunSummary` to count events.
- `Failed.reason` is a human-readable error description (not a stack trace).
- `Skipped.reason` explains why (e.g., "unchanged", "filtered out").

Helper methods:
- `isSucceeded`, `isFailed`, `isSkipped` — type-check convenience
- `toResultDO(runId: UUID, correlationId: UUID): ProcessingResultDO` — converts to the DB-persisted form

### Pipeline Run Summary (in-memory)

| Class | Fields | Notes |
|-------|--------|-------|
| `PipelineRunSummary` | runId, pipelineName, status, startedAt, completedAt, itemsProcessed, itemsSucceeded, itemsFailed, errorCounts: Map[String, Int] | Computed from `ProcessingResult` stream at end of run. Used to update `PipelineRunDO` before final write. |

### Status Derivation Rules

| Condition | Derived PipelineStatus |
|-----------|----------------------|
| All items Succeeded | Completed |
| Some items Failed, some Succeeded | CompletedWithErrors |
| All items Failed | Failed |
| No items processed (empty run) | Completed |

### Acceptance Criteria — 2.2 Pipeline Execution Tracking

**Functional:**
1. `ProcessingResult` enum has cases `Succeeded`, `Failed`, `Skipped` with the fields specified above.
1a. `ProcessingResult.Succeeded.eventEmitted` defaults to `false`.
1b. `ProcessingResult.toResultDO` correctly maps to `ProcessingResultDO` fields (Succeeded → ResultStatus.Succeeded, etc.).
2. `PipelineRunDO` fields match all columns in the `pipeline_runs` table.
2. `ProcessingResultDO` fields match all columns in the `processing_results` table.
3. `PipelineStatus`, `ResultStatus`, and `PipelineTrigger` enums each have Circe codecs and `fromString` with multi-source alias support.
4. All DOs have Doobie auto-derived `Read` and `Write` instances.
5. `PipelineRunDO.completedAt` is `Option[Instant]` — `None` while running.
6. `ProcessingResultDO.correlationId` is the same UUID that appears in all log lines for that item.
7. `ProcessingResultDO.entityType` is constrained to known values: 'bill', 'vote', 'member', 'amendment', 'analysis', 'score'.
8. `PipelineRunSummary` can be constructed from a `List[ProcessingResultDO]` — counts computed from status values, error counts from `errorClass` grouping.
9. Status derivation follows the rules table above.

**Error Handling:**
10. `PipelineStatus.fromString` and `ResultStatus.fromString` return `Left` with unrecognized value for unknown strings.

**Test Requirements:**
11. Round-trip Doobie `Read`/`Write` test for `PipelineRunDO` and `ProcessingResultDO`.
12. Circe round-trip test for each status enum.
13. Test `PipelineRunSummary` construction from a mixed list of Succeeded/Failed/Skipped results — verify correct counts.
14. Test status derivation: all Succeeded → Completed.
15. Test status derivation: some Failed → CompletedWithErrors.
16. Test status derivation: all Failed → Failed.
17. Test status derivation: empty results → Completed.
