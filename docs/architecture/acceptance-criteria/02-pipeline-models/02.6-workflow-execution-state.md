# 2.6 Workflow Execution State

> **Design rationale:** Workflow execution state is stored in AlloyDB, not in Pub/Sub or the Launcher's memory. Each application updates its own state. The Launcher reads state (read-only) to check dependencies. Two new tables: `workflow_runs` (one row per workflow execution) and `workflow_run_steps` (one row per step per execution).

### New Database Tables

**`workflow_runs`:**

| Column | Type | Notes |
|--------|------|-------|
| `workflow_run_id` | UUID PK | Unique identifier for this workflow execution |
| `workflow_name` | TEXT NOT NULL | e.g., "daily-ingestion" |
| `workflow_definition_path` | TEXT NOT NULL | GCS path to the workflow JSON used for this run |
| `status` | TEXT NOT NULL DEFAULT 'pending' | pending, running, completed, completed_with_errors, failed |
| `triggered_by` | TEXT NOT NULL | e.g., "daily-initializer", "api-server", "manual" |
| `started_at` | TIMESTAMPTZ | Set when first step starts running |
| `completed_at` | TIMESTAMPTZ | Set when all steps complete or workflow fails |
| `created_at` | TIMESTAMPTZ NOT NULL DEFAULT NOW() | |

**`workflow_run_steps`:**

| Column | Type | Notes |
|--------|------|-------|
| `workflow_run_id` | UUID FK → workflow_runs | |
| `step_name` | TEXT NOT NULL | e.g., "members-pipeline" |
| `status` | TEXT NOT NULL DEFAULT 'pending' | pending, running, completed, failed |
| `pipeline_run_id` | UUID FK → pipeline_runs | Nullable — set once the step launches and creates its pipeline run |
| `retry_count` | INT NOT NULL DEFAULT 0 | Incremented on each failure before requeue |
| `max_retries` | INT NOT NULL DEFAULT 3 | |
| `original_message` | TEXT | The Pub/Sub message payload that triggered this step — stored for requeue on failure |
| `started_at` | TIMESTAMPTZ | |
| `completed_at` | TIMESTAMPTZ | |
| `error_message` | TEXT | |
| `created_at` | TIMESTAMPTZ NOT NULL DEFAULT NOW() | |
| `updated_at` | TIMESTAMPTZ NOT NULL DEFAULT NOW() | |
| PRIMARY KEY | (workflow_run_id, step_name) | |

### Status Enums

| Enum | Values | Notes |
|------|--------|-------|
| `WorkflowStepStatus` | Pending, Running, Completed, Failed | Per-step status — step identity is in `step_name` column |
| `WorkflowRunStatus` | Pending, Running, Completed, CompletedWithErrors, Failed | Per-workflow status — derived from step statuses (see derivation rules below) but stored in `workflow_runs.status` column |

Both enums have Circe codecs and `fromString` / `toString` for database mapping.

### Workflow Status Derivation

| Condition | Derived Status |
|-----------|---------------|
| Any step Running | Running |
| All steps Completed | Completed |
| Some steps Completed, some Failed | CompletedWithErrors |
| All non-Pending steps Failed | Failed |
| All steps Pending | Pending |

### DOs

| Class | Table | Key Columns |
|-------|-------|-------------|
| `WorkflowRunDO` | `workflow_runs` | workflowRunId (UUID), workflowName, workflowDefinitionPath, status, triggeredBy, startedAt (Option), completedAt (Option), createdAt |
| `WorkflowRunStepDO` | `workflow_run_steps` | workflowRunId (UUID), stepName, status (WorkflowStepStatus), pipelineRunId (Option[UUID]), retryCount (Int), maxRetries (Int), originalMessage (Option[String]), startedAt (Option), completedAt (Option), errorMessage (Option), createdAt, updatedAt |

### Acceptance Criteria — 2.6 Workflow Execution State

**Functional:**
1. `WorkflowRunDO` fields match all columns in the `workflow_runs` table.
2. `WorkflowRunStepDO` fields match all columns in the `workflow_run_steps` table.
3. `WorkflowStepStatus` and `WorkflowRunStatus` enums have Circe codecs and `fromString`.
4. All DOs have Doobie auto-derived `Read` and `Write` instances.
5. `WorkflowRunStepDO.pipelineRunId` is `Option[UUID]` — `None` until the step is launched and creates its pipeline run.
6. `WorkflowRunStepDO.originalMessage` stores the full Pub/Sub message JSON so the application can requeue on failure without reconstructing it.
7. Workflow status is derived from step statuses following the derivation rules table.
8. When a step fails and `retryCount < maxRetries`, the application increments `retryCount` and requeues `originalMessage` to Pub/Sub.
9. When a step fails and `retryCount >= maxRetries`, the application sets status to `Failed` and the workflow stops executing (no further steps are launched).
10. `WorkflowRunStepDO.pipelineRunId` links to `pipeline_runs` — enables drill-through from workflow → step → pipeline run → processing results.

**State Transitions:**
11. `Failed → Running` is a valid transition — it occurs when a failed step is retried (either via `PipelineFailureHandler` requeue or manual operator intervention). The `retryCount` field tracks how many attempts have been made.
12. Each step has a **single row** in `workflow_run_steps` (PK: `workflow_run_id + step_name`). Status and `retryCount` are updated in place — no new rows are created on retry.

**Error Handling:**
13. `WorkflowStepStatus.fromString` and `WorkflowRunStatus.fromString` return `Left` with unrecognized value for unknown strings.

**Test Requirements:**
14. Round-trip Doobie `Read`/`Write` test for `WorkflowRunDO` and `WorkflowRunStepDO`.
15. Circe round-trip test for `WorkflowStepStatus` and `WorkflowRunStatus`.
16. Test workflow status derivation: all steps Completed → Completed.
17. Test workflow status derivation: one step Failed → CompletedWithErrors (if others completed) or Failed (if all failed).
18. Test workflow status derivation: any step Running → Running.
19. Test retry logic: `retryCount < maxRetries` → requeue allowed.
20. Test retry logic: `retryCount >= maxRetries` → status set to Failed.
21. Test `Failed → Running` transition: failed step retried → status updated to Running, retryCount preserved.
