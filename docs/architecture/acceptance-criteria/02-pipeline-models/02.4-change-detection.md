# 2.4 Change Detection

> **Design rationale:** Each entity type has a defined strategy for detecting whether incoming data represents a change worth persisting and whether to emit an event. Rules from BEHAVIORAL_SPECS.md §1.

### Strategy Enums

| Enum | Values | Notes |
|------|--------|-------|
| `ChangeDetectionStrategy` | UpdateDateComparison, ExistenceCheck, FieldLevelDiff, AlwaysNew | How to determine if an entity changed |
| `PersistenceStrategy` | Upsert, UpsertWithHistory, AppendOnly | How to write the change to the database |

### Change Detection Rules (per entity)

| Entity | Detection Strategy | Persistence Strategy | Emits Event |
|--------|-------------------|---------------------|-------------|
| Bill (metadata) | UpdateDateComparison | Upsert | No |
| Bill (text version) | UpdateDateComparison on `updateDateIncludingText` | AppendOnly (new row in `bill_text_versions`) | `bill.text.available` |
| Vote | UpdateDateComparison | UpsertWithHistory (archive to `vote_history` + `vote_history_positions`) | `vote.recorded` |
| Member | FieldLevelDiff (compare all fields) | Upsert | No |
| Amendment | UpdateDateComparison | Upsert | None |
| Analysis | AlwaysNew (each pass creates new version) | AppendOnly | `analysis.completed` |
| Score | AlwaysNew | Upsert (latest) + AppendOnly (history) | No |

> **Note:** `AmendmentRecordedEvent` was removed per §2.1 -- the amendments pipeline is a pure data recorder. Bill re-analysis is triggered by new bill text (`bill.text.available`), not by amendment events.

### Types

| Class | Fields | Notes |
|-------|--------|-------|
| `ChangeDetectionResult[T]` | entity: T, entityId: String, entityType: String, changed: Boolean, strategy: ChangeDetectionStrategy, persistenceStrategy: PersistenceStrategy, emitEvent: Option[String] | Generic result. `emitEvent` is the event type to publish if `changed == true`, or `None` if no event needed. |
| `EntityChangeConfig` | entityType: String, detectionStrategy: ChangeDetectionStrategy, persistenceStrategy: PersistenceStrategy, eventType: Option[String] | Static configuration per entity type. |

### Acceptance Criteria — 2.4 Change Detection

**Functional:**
1. `ChangeDetectionStrategy` and `PersistenceStrategy` are enums with Circe codecs and `fromString`.
2. `ChangeDetectionResult` is generic over the entity type `T`.
3. `ChangeDetectionResult.emitEvent` is `Some(eventType)` only when `changed == true` AND the entity config has an event type.
4. `ChangeDetectionResult.emitEvent` is `None` when `changed == false` even if the entity config has an event type.
5. `EntityChangeConfig` matches the rules table above for all 7 entity types.
6. Vote change detection uses `UpsertWithHistory` — the old vote data must be archived before the new data is written.

**Test Requirements:**
7. Test `ChangeDetectionResult` with `changed = true` and event config present — verify `emitEvent = Some(...)`.
8. Test `ChangeDetectionResult` with `changed = false` and event config present — verify `emitEvent = None`.
9. Test `ChangeDetectionResult` with `changed = true` and no event config — verify `emitEvent = None`.
10. Verify `EntityChangeConfig` for each of the 7 entity types matches the rules table.
11. Circe round-trip test for `ChangeDetectionStrategy` and `PersistenceStrategy` enums.
