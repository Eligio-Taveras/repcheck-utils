# 2.1 Inter-Pipeline Communication

> **Design rationale:** Pipelines communicate via a single Google Pub/Sub queue. Each message has a typed payload wrapped in a `PipelineEvent` envelope carrying metadata for routing, idempotency, and traceability. The Launcher reads these messages to determine what to launch next. Event types defined in BEHAVIORAL_SPECS.md §4.

### Event Envelope

| Class | Fields | Notes |
|-------|--------|-------|
| `PipelineEvent[T]` | eventType: String, payload: T, timestamp: Instant, eventId: UUID, correlationId: UUID, source: String | Generic envelope. `eventType` is the routing key. `eventId` is unique per emission for idempotency. `correlationId` traces the item through the full pipeline chain. `source` is the pipeline name that emitted it. |

### Acceptance Criteria — 2.1 Inter-Pipeline Communication

**Functional:**
1. `PipelineEvent[T]` has Circe `Encoder` and `Decoder` for any `T` that has its own Circe codec.
2. Every event payload has Circe semi-auto derived `Encoder` and `Decoder`.
3. `PipelineEvent.eventId` is a UUID generated at construction time (default `UUID.randomUUID()`).
4. `PipelineEvent.correlationId` is passed through from the processing context — not auto-generated.
5. `EventTypes` object contains string constants matching the exact event type values used in Pub/Sub routing.
6. Event payloads round-trip through JSON serialization: `decode(encode(event)) == Right(event)`.
7. `BillTextAvailableEvent` includes `versionCode` and `previousVersionCode` (Option — None if first text ever).
8. `VoteRecordedEvent.billId` is `Option[String]` because some procedural votes are not associated with a specific bill.
9. `AmendmentRecordedEvent.sponsorBioguideId` is `Option[String]` because some amendments may not have an identified sponsor.
10. `MemberUpdatedEvent` has a single field `memberId: String` (bioguideId). Minimal payload — consumers look up member details from AlloyDB. Emitted by both `member-profile-pipeline` and `lis-mapping-refresher`.

**Idempotency & Ordering:**

Pub/Sub provides **at-least-once** delivery — consumers may receive the same event more than once and events may arrive out of order. No deduplication table is needed because:
1. **Upsert semantics** — all pipelines use `INSERT ... ON CONFLICT DO UPDATE`, so reprocessing an event produces the same result.
2. **Self-correction on next run** — even if an out-of-order event causes a temporary inconsistency, the next scheduled or event-driven run will re-fetch current data from Congress.gov and correct it.
3. **`eventId`** on `PipelineEvent` is available for consumer-side dedup if a future pipeline needs exactly-once semantics, but current pipelines do not require it.

**Error Handling:**
10. Decoding an unknown `eventType` returns a descriptive `Left` listing valid event types.
11. Decoding a `PipelineEvent` with a malformed payload returns `Left` with the event type and the payload decoding error.

**Test Requirements:**
12. Round-trip codec test for each event payload with all fields populated.
13. Round-trip codec test for `PipelineEvent[BillTextAvailableEvent]` (full envelope + payload).
14. Test that `PipelineEvent` with missing `correlationId` fails decoding (it's required, not auto-generated).
15. Test that each `EventTypes` constant matches its expected string value.
16. Test decoding a `PipelineEvent` with unknown `eventType` — verify descriptive error.
17. Test `BillTextAvailableEvent` with `previousVersionCode = None` (first text) and `Some("IH")` (subsequent version).
18. `BillTextIngestedEvent.versionId` is `UUID` — required field, not optional. It is the `bill_text_versions` PK created by Component 4 §4.8 when text is downloaded and stored. The decomposition pipeline (Component 10) needs this to link sections and concept groups back to the text version.
19. `DecompositionCompletedEvent` has Circe semi-auto derived `Encoder` and `Decoder`.
20. `DecompositionCompletedEvent` round-trips through JSON: `decode(encode(event)) == Right(event)`.
21. `DecompositionCompletedEvent` includes `billId: String`, `versionId: UUID`, `conceptGroupCount: Int`, `sectionCount: Int`.
22. `EventTypes.DecompositionCompleted` constant value is `"bill.decomposition.completed"`.
23. `AnalysisCompletedEvent.passesExecuted` is `List[Int]` (e.g., `[1, 2]`) — replaces the former singular `passCompleted: Int`.
24. `AnalysisCompletedEvent.modelUsed` is `String` — replaces the former `llmModel` field name.
25. `VoteRecordedEvent.billId` is confirmed present as `Option[String]`. The votes pipeline uses this field to update `stance_materialization_status.has_votes` for the bill. When `billId = None` (procedural vote), no stance materialization status update is performed.
26. Round-trip codec test for `VoteRecordedEvent` with `billId = None` (procedural vote) and `billId = Some("118-HR-1234")` (bill-tied vote).
27. `ScoringUserRequestedEvent` has Circe semi-auto derived `Encoder` and `Decoder`.
28. `ScoringUserRequestedEvent` round-trips through JSON: `decode(encode(event)) == Right(event)`.
29. `ScoringUserRequestedEvent.source` is `String` — valid values are `"scheduled"` and `"ad-hoc"`.
30. `ScoringUserRequestedEvent.requestId` is `UUID` — unique per scoring request.
31. `ScoringUserCompletedEvent` has Circe semi-auto derived `Encoder` and `Decoder`.
32. `ScoringUserCompletedEvent` round-trips through JSON: `decode(encode(event)) == Right(event)`.
33. `ScoringUserCompletedEvent.status` is `String` — valid values are `"completed"`, `"partial"`, `"failed"`.
34. `ScoringUserCompletedEvent.memberScoreCount` is `Int` — number of member scores computed for the user.
35. `EventTypes.ScoringUserRequested` constant value is `"scoring.user.requested"`.
36. `EventTypes.ScoringUserCompleted` constant value is `"scoring.user.completed"`.
