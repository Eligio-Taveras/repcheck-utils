# 2.3 Error Handling & Retry

> **Design rationale:** Every pipeline subsystem has an `ErrorClassifier` that categorizes exceptions as `Transient` (safe to retry — network timeouts, rate limits) or `Systemic` (permanent — bad data, schema mismatch). Retry is handled by a centralized `RetryWrapper[F]` with configurable exponential backoff. From BEHAVIORAL_SPECS.md §5.

### Types

| Class/Trait | Type | Fields/Methods |
|-------------|------|----------------|
| `ErrorClass` | Enum | Transient, Systemic |
| `ErrorClassifier` | Trait | `classify(error: Throwable): ErrorClass` — each subsystem provides its own implementation |
| `RetryConfig` | Case class | maxRetries: Int (default 3), initialBackoffMs: Long (default 10), maxBackoffMs: Long (default 60000), backoffMultiplier: Double (default 2.0) |
| `RetryWrapper[F[_]]` | Class | `withRetry[A](config: RetryConfig, classifier: ErrorClassifier, correlationId: UUID)(fa: F[A]): F[A]` — retries on `Transient`, fails immediately on `Systemic`. `correlationId` is included in all retry log messages for traceability. |
| `DeadLetterMessage` | Case class | originalPayload: String, eventType: String, failureReason: String, failureTimestamp: Instant, retryCount: Int |

### Standard ErrorClassifier Implementations

Pipeline-models provides shared classifiers. Individual pipelines can extend these for their specific error types.

| Class | Classifies | Notes |
|-------|-----------|-------|
| `DefaultErrorClassifier` | All unknown exceptions → Systemic | Base fallback implementation |
| `HttpErrorClassifier` | 429, 500, 502, 503, 504 → Transient; 400, 401, 403, 404 → Systemic | Shared by all HTTP-calling pipelines |

### Acceptance Criteria — 2.3 Error Handling & Retry

**Functional:**
1. `ErrorClass` enum has values `Transient` and `Systemic` with Circe codecs and `fromString`.
2. `ErrorClassifier` is a trait with a single method `classify(error: Throwable): ErrorClass`.
3. `RetryConfig` has PureConfig auto-derivation for loading from HOCON/JSON config.
4. `RetryWrapper.withRetry` retries up to `maxRetries` times when `ErrorClassifier` returns `Transient`.
5. `RetryWrapper.withRetry` fails immediately (no retry) when `ErrorClassifier` returns `Systemic`.
6. `RetryWrapper` applies exponential backoff: delay = `min(initialBackoffMs * multiplier^attempt, maxBackoffMs)`.
7. `HttpErrorClassifier` classifies HTTP 429, 500, 502, 503, 504 as `Transient` and all 4xx (except 429) as `Systemic`.
8. `DefaultErrorClassifier` classifies all unrecognized exceptions as `Systemic`.
9. `DeadLetterMessage` has Circe codecs for publishing to dead-letter Pub/Sub topics.

**Error Handling:**
10. `RetryWrapper` logs each retry attempt with: attempt number, max retries, delay before next attempt, error class, and error message.
11. `RetryWrapper` includes the `correlationId` in all retry log messages for traceability.

**Test Requirements:**
12. Test `RetryWrapper` with `Transient` errors — verify retries up to `maxRetries`, then fails.
13. Test `RetryWrapper` with `Systemic` error — verify immediate failure, no retries.
14. Test `RetryWrapper` with `Transient` then success — verify succeeds after retry.
15. Test exponential backoff timing: verify delay doubles between attempts and caps at `maxBackoffMs`.
16. Test `HttpErrorClassifier` with every listed status code — verify correct classification.
17. Test `DefaultErrorClassifier` — verify unknown exception classifies as `Systemic`.
18. Round-trip Circe test for `DeadLetterMessage`.

### Scoping: RetryWrapper vs PipelineFailureHandler

These two mechanisms operate at different levels:

- **`RetryWrapper[F]`** (this section) — wraps **individual operations** (a single API call, a single DB query). It retries transient failures within the operation boundary. Every pipeline uses `RetryWrapper` for its API and DB calls.
- **`PipelineFailureHandler[F]`** (Component 3 §3.7) — wraps an **entire pipeline step**. On failure, it requeues the original Pub/Sub message to retry the step from scratch. Only **event-driven** pipelines (those triggered by Pub/Sub via the Launcher) use `PipelineFailureHandler`. **Scheduled** pipelines (e.g., `bill-metadata-pipeline`, `member-profile-pipeline`) do not use it — they simply fail, and the next scheduled run will re-process.

| Pipeline Type | RetryWrapper | PipelineFailureHandler |
|--------------|-------------|----------------------|
| Scheduled (cron-triggered) | Yes — wraps API/DB calls | No — next scheduled run handles recovery |
| Event-driven (Pub/Sub-triggered) | Yes — wraps API/DB calls | Yes — requeues message on step failure |

### ErrorClassifier Wiring

`HttpErrorClassifier` lives in `pipeline-models` (this component) because it is shared by all HTTP-calling pipelines. Entity-specific classifiers (e.g., one that classifies a specific API's 422 responses as `Transient`) live in each pipeline app and extend `HttpErrorClassifier`. Classifiers are injected via constructor — the pipeline app creates the classifier and passes it to `RetryWrapper` at wiring time.
