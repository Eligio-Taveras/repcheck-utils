# 8.2 Bill Analysis Profiles

> Part of [Component 8 — Prompt Engine Bills](../08-PROMPT-ENGINE-BILLS.md)

## Overview

Defines the prompt profile catalog for bill analysis: which instruction blocks compose each profile, what stage ordering and weights apply, and what the initial block set covers. Profiles and blocks are stored in GCS — this area defines the **contract** (what profiles exist and what they contain), not the prompt text itself.

## Profile Catalog

### Decomposition Profiles

Used by the bill-analysis-pipeline (Component 10) during the text decomposition step for large bills.

#### `concept-simplification`

Simplifies a group of related bill sections into a coherent summary.

| Stage | Blocks | Weight | Purpose |
|-------|--------|--------|---------|
| System | `base-legislative-analyst` | 1.0 | Core analyst role |
| Persona | `general-audience` | 0.9 | Plain language output |
| Context | `concept-group-text` | 1.0 | Placeholder: `{{concept_group_text}}` — the grouped sections to simplify |
| Guardrails | `nonpartisan-constraint`, `accuracy-constraint` | 1.0 | Bias prevention |
| Output | `concept-summary-schema` | 1.0 | JSON schema for concept summary output |

**Expected input context:** A subset of bill text (one concept group — related sections clustered together by Component 10).

**Expected output:** A concise summary of the concept group that preserves key provisions, dollar amounts, affected parties, and policy mechanisms.

#### `section-classification`

Classifies a bill section by topic/policy area to aid concept grouping.

| Stage | Blocks | Weight | Purpose |
|-------|--------|--------|---------|
| System | `base-legislative-analyst` | 1.0 | Core analyst role |
| Context | `section-text` | 1.0 | Placeholder: `{{section_text}}` — the individual section to classify |
| Output | `topic-classification-schema` | 1.0 | JSON schema matching `TopicClassificationOutput` (Component 1 §1.6) |

**Expected input context:** A single section/provision of the bill.

**Expected output:** Topic tags with confidence scores, used by Component 10 to group related sections.

### Analysis Profiles

Used by the bill-analysis-pipeline for the three-pass tiered analysis.

#### `pass1-extraction`

Structured extraction, topic classification, and plain language summary. Runs on all bills (Haiku tier).

| Stage | Blocks | Weight | Purpose |
|-------|--------|--------|---------|
| System | `base-legislative-analyst` | 1.0 | Core analyst role |
| Persona | `general-audience` | 0.9 | 8th-grade reading level |
| Lens | `policy-area-classifier` | 0.8 | Topic/policy area classification |
| Context | `bill-concepts`, `amendments` | 1.0 | `{{bill_concepts}}` = simplified concept summaries (or raw text for short bills); `{{amendments}}` = associated amendment descriptions |
| Guardrails | `nonpartisan-constraint`, `accuracy-constraint` | 1.0 | Bias prevention, factual accuracy |
| Output | `pass1-extraction-schema` | 1.0 | JSON schema for summary + topics + key points |

**Expected output schemas:** `BillSummaryOutput` + `TopicClassificationOutput` (Component 1 §1.6).

#### `pass2-deep-analysis`

Deep analysis: pork detection, impact analysis, stance classification, fiscal estimates. Runs on filtered bills (Sonnet tier).

| Stage | Blocks | Weight | Purpose |
|-------|--------|--------|---------|
| System | `base-legislative-analyst` | 1.0 | Core analyst role |
| Persona | `expert-audience` | 0.8 | More detailed/technical output |
| Lens | `fiscal-lens`, `civil-liberties-lens`, `pork-detector` | 0.8 | Multiple analytical lenses (additive) |
| Context | `bill-concepts`, `amendments`, `pass1-results` | 1.0 | `{{bill_concepts}}`, `{{amendments}}`, `{{pass1_results}}` = Pass 1 output for context |
| Guardrails | `nonpartisan-constraint`, `accuracy-constraint` | 1.0 | Bias prevention |
| Output | `pass2-deep-analysis-schema` | 1.0 | JSON schema for pork + impact + stance + fiscal |

**Expected output schemas:** `PorkDetectionOutput` + `ImpactAnalysisOutput` + `StanceClassificationOutput` + `FiscalEstimateOutput` (Component 1 §1.6).

> **Multiple lenses:** The Lens stage supports multiple blocks. Each lens adds an analytical focus. `ChainAssembler` (Component 1 §1.7) merges them into a single stage section. The `weight` on the stage applies to all lenses equally.

#### `pass3-ambiguity-resolution`

Ambiguity resolution and cross-bill analysis. Runs only on flagged bills (Opus tier).

| Stage | Blocks | Weight | Purpose |
|-------|--------|--------|---------|
| System | `base-legislative-analyst` | 1.0 | Core analyst role |
| Persona | `expert-audience` | 0.9 | Detailed, nuanced output |
| Lens | `ambiguity-resolver`, `cross-reference-analyzer` | 0.9 | Cross-document reasoning |
| Context | `bill-concepts`, `amendments`, `pass1-results`, `pass2-results`, `related-bills` | 1.0 | All prior analysis plus related bills context |
| Guardrails | `nonpartisan-constraint`, `accuracy-constraint` | 1.0 | Bias prevention |
| Output | `pass3-resolution-schema` | 1.0 | JSON schema for ambiguity resolution |

### Utility Profiles

Convenience profiles for targeted analysis (not part of the standard three-pass flow).

| Profile | Purpose | Blocks |
|---------|---------|--------|
| `summary-only` | Generate only a plain language summary (no classification or deep analysis) | System + Persona(general) + Context(bill-concepts) + Output(summary-schema) |
| `pork-detection` | Targeted pork/rider scan without full deep analysis | System + Persona(expert) + Lens(pork-detector) + Context(bill-concepts) + Output(pork-schema) |

## Block Catalog

### Initial Block Set

All blocks are stored in GCS as YAML files. This catalog defines the **contract** — the block names, stages, and purposes that profiles reference. The actual prompt content lives in GCS.

#### System Stage

| Block Name | Purpose |
|------------|---------|
| `base-legislative-analyst` | Core system prompt establishing the analyst role, capabilities, and behavioral baseline |

#### Persona Stage

| Block Name | Purpose |
|------------|---------|
| `general-audience` | Plain language at 8th-grade reading level; avoids jargon |
| `expert-audience` | Technical/detailed output; assumes policy expertise |

#### Lens Stage

| Block Name | Purpose |
|------------|---------|
| `policy-area-classifier` | Topic and policy area classification with confidence scores |
| `fiscal-lens` | Fiscal impact analysis: budgets, revenue, CBO-style estimates |
| `civil-liberties-lens` | Civil liberties and constitutional implications |
| `healthcare-lens` | Healthcare policy impacts |
| `pork-detector` | Earmarks, riders, and unrelated provisions |
| `ambiguity-resolver` | Identifies and resolves ambiguous or contradictory provisions |
| `cross-reference-analyzer` | Cross-bill interaction analysis |

#### Context Stage

Context blocks contain placeholders that are replaced at runtime by `BillContextInjector` (§8.3).

| Block Name | Placeholder | Injected Content |
|------------|-------------|-----------------|
| `bill-concepts` | `{{bill_concepts}}` | Simplified concept summaries (from decomposition) or raw text (short bills) |
| `concept-group-text` | `{{concept_group_text}}` | A single concept group for simplification |
| `section-text` | `{{section_text}}` | A single section for classification |
| `amendments` | `{{amendments}}` | Amendment descriptions for the bill |
| `related-bills` | `{{related_bills}}` | Related bill summaries (for cross-bill analysis) |
| `pass1-results` | `{{pass1_results}}` | Pass 1 extraction results (for Pass 2/3 context) |
| `pass2-results` | `{{pass2_results}}` | Pass 2 analysis results (for Pass 3 context) |

#### Guardrails Stage

| Block Name | Purpose |
|------------|---------|
| `nonpartisan-constraint` | Prohibits partisan language, party-based opinions, political recommendations |
| `accuracy-constraint` | Requires factual citations, distinguishes fact from interpretation, flags uncertainty |

#### Output Stage

| Block Name | References Schema |
|------------|------------------|
| `concept-summary-schema` | Concept group summary (decomposition output) |
| `topic-classification-schema` | `TopicClassificationOutput` (§1.6) |
| `pass1-extraction-schema` | `BillSummaryOutput` + `TopicClassificationOutput` (§1.6) |
| `pass2-deep-analysis-schema` | `PorkDetectionOutput` + `ImpactAnalysisOutput` + `StanceClassificationOutput` + `FiscalEstimateOutput` (§1.6) |
| `pass3-resolution-schema` | Ambiguity resolution output |

## Block Versioning

- All block filenames include semver: `{blockName}-v{major}.{minor}.{patch}.yaml`
- A profile references blocks by name — the version is resolved at load time by `BlockLoader` (§8.1)
- When a block is updated, a new file is created with an incremented version number — old versions are preserved in GCS
- The `BillPromptEngineConfig.defaultVersion` (§8.1) determines which version is loaded unless overridden

## Acceptance Criteria

| Criterion | Test |
|-----------|------|
| Every profile in the catalog has a valid YAML definition in the test fixtures | Load each profile → deserializes to `PromptProfile` without errors |
| Every profile references only blocks that exist in the block catalog | Cross-reference profile `blockNames` against block catalog → no missing references |
| Every block in the catalog has a `stage` matching its directory | Block in `lens/` → `stage == Lens` |
| All context blocks have exactly one placeholder token | Inspect each context block → contains exactly one `{{...}}` token |
| `concept-simplification` profile assembles in correct stage order | Assemble → stages appear as: System, Persona, Context, Guardrails, Output |
| `section-classification` profile has no Persona or Lens stage | Assemble → no Persona or Lens content |
| `pass1-extraction` profile includes all required stages | Assemble → System, Persona, Lens, Context, Guardrails, Output all present |
| `pass2-deep-analysis` profile includes multiple lenses | Assemble → Lens section contains fiscal, civil-liberties, and pork-detector content |
| `pass2-deep-analysis` includes `pass1-results` context block | Profile chain → Context stage includes `pass1-results` |
| `pass3-ambiguity-resolution` includes `pass1-results` and `pass2-results` | Profile chain → Context stage includes both |
| Utility profiles (`summary-only`, `pork-detection`) are valid | Load each → deserializes correctly |
| No two blocks share the same name | Block catalog audit → all names unique |
| No two profiles share the same name | Profile catalog audit → all names unique |
| All weights are in 0.0-1.0 range | Inspect all profile stage weights → all valid |
