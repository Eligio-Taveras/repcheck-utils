# 8.3 Bill Prompt Assembler

> Part of [Component 8 â€” Prompt Engine Bills](../08-PROMPT-ENGINE-BILLS.md)

## Overview

Bill-specific implementation of `ChainAssembler` (Component 1 Â§1.7) that handles context injection â€” serializing structured bill data into prompt-friendly text and injecting it into Context stage blocks at runtime. This is the main entry point consumed by the bill-analysis-pipeline (Component 10).

The assembler does more than simple placeholder replacement. A bill's context is **richly structured** â€” multiple concept groups, multiple amendments, multiple related bills â€” and the assembler is responsible for formatting these structured lists into coherent prompt text that the LLM can reason about.

## Classes

### `BillPromptAssembler[F[_]]` â€” Trait + Impl

```
trait BillPromptAssembler[F[_]] {
  def assembleForAnalysis(
    profileName: String,
    context: BillAnalysisContext,
    version: Option[String] = None     // None â†’ use config default
  ): F[String]

  def assembleForDecomposition(
    profileName: String,
    context: DecompositionContext,
    version: Option[String] = None
  ): F[String]
}
```

Both methods follow the same flow:
1. Resolve version: `version.getOrElse(config.defaultVersion)`
2. Load profile via `ProfileLoader.loadProfile(profileName, v)`
3. Collect all block names from the profile's chain
4. Load all blocks via `BlockLoader.loadBlocksByNames(names, v)`
5. Serialize structured context into prompt text via `BillContextInjector`
6. Delegate to `ChainAssembler.assemble(profile, blocks, contextMap)` (Component 1 Â§1.7)
7. Return the assembled prompt string

**`assembleForAnalysis`** is the primary method used by Component 10 for Pass 1/2/3. It takes `BillAnalysisContext` containing all structured context â€” the assembler serializes and injects only those values referenced by the profile's context blocks.

**`assembleForDecomposition`** is used by Component 10 during the decomposition step. It takes `DecompositionContext` which is simpler (a single concept group or section, no prior pass results).

## Structured Context Types

### `BillAnalysisContext`

Runtime context for analysis pass prompts. A bill's context is multi-dimensional â€” it may have dozens of concept groups, many amendments from different sponsors, and several related bills. Prior pass results are structured outputs (not raw strings) because each pass produces multiple typed analysis artifacts per concept group.

```
final case class BillAnalysisContext(
  billId: String,
  congress: Int,
  concepts: List[ConceptGroup],
  amendments: List[AmendmentSummary],
  relatedBills: List[RelatedBillSummary],
  pass1Results: Option[Pass1Output],         // structured Pass 1 output (for Pass 2/3)
  pass2Results: Option[Pass2Output]          // structured Pass 2 output (for Pass 3)
)
```

### `ConceptGroup`

A cluster of related bill sections that have been simplified by the decomposition step. A large bill may have 10-30 concept groups; a short bill may have 1-3 or use raw text directly.

```
final case class ConceptGroup(
  groupId: String,                           // e.g., "transportation-funding"
  title: String,                             // human-readable concept name
  topics: List[String],                      // from section-classification step
  sectionReferences: List[String],           // which bill sections are in this group
  simplifiedText: String                     // LLM-produced summary of this group
)
```

### `AmendmentSummary`

A summary of one amendment relevant to this bill. A bill may have dozens of amendments across its legislative history â€” House amendments, Senate amendments, committee amendments.

```
final case class AmendmentSummary(
  amendmentId: String,                       // natural key from Component 7
  amendmentType: String,                     // "HAMDT", "SAMDT", "SUAMDT"
  sponsorName: Option[String],               // human-readable sponsor name
  sponsorBioguideId: Option[String],
  description: String,                       // amendment description from Congress.gov
  purpose: Option[String],                   // e.g., "In the nature of a substitute"
  latestActionText: Option[String],          // e.g., "Amendment agreed to by voice vote"
  affectedConcepts: List[String]             // groupIds this amendment touches (if determinable)
)
```

> **`affectedConcepts`** links amendments to the concept groups they modify. This is a best-effort mapping â€” Component 10 may populate it based on textual overlap or leave it empty if the linkage is unclear. The prompt assembler includes it when present to give the LLM cross-reference context.

### `RelatedBillSummary`

A summary of a bill related to the one being analyzed. Relevant for Pass 3 cross-bill analysis, or when amendments reference other legislation.

```
final case class RelatedBillSummary(
  billId: String,                            // natural key
  title: String,
  relationship: String,                      // e.g., "companion bill", "amended by", "supersedes"
  summary: Option[String],                   // Pass 1 summary if available
  topics: List[String]                       // topic overlap with the primary bill
)
```

### `Pass1Output`

Structured output from Pass 1 (Haiku tier). Contains per-concept-group summaries and topic classifications â€” not a single string, because each concept group produces its own summary and classification.

```
final case class Pass1Output(
  conceptSummaries: List[ConceptSummaryResult],
  overallSummary: BillSummaryOutput,         // Component 1 Â§1.6
  topics: TopicClassificationOutput          // Component 1 Â§1.6
)

final case class ConceptSummaryResult(
  groupId: String,                           // matches ConceptGroup.groupId
  summary: BillSummaryOutput,                // per-concept summary
  topics: TopicClassificationOutput          // per-concept topic classification
)
```

> **Why per-concept results?** A large bill with 20 concept groups cannot be meaningfully summarized in a single pass. Pass 1 produces a summary and topic classification *per concept group*, then aggregates into an overall summary. Short bills with a single concept group will have one entry in `conceptSummaries` plus the `overallSummary`.

### `Pass2Output`

Structured output from Pass 2 (Sonnet tier). Deep analysis produces multiple findings per concept group â€” pork detection may flag items in different sections, impact analysis identifies different affected groups per concept.

```
final case class Pass2Output(
  pork: PorkDetectionOutput,                 // Component 1 Â§1.6
  impacts: ImpactAnalysisOutput,             // Component 1 Â§1.6
  stances: StanceClassificationOutput,       // Component 1 Â§1.6
  fiscal: FiscalEstimateOutput,              // Component 1 Â§1.6
  perConceptAnalysis: List[ConceptAnalysisResult]
)

final case class ConceptAnalysisResult(
  groupId: String,                           // matches ConceptGroup.groupId
  pork: PorkDetectionOutput,                 // per-concept pork findings
  impacts: ImpactAnalysisOutput,             // per-concept impact analysis
  stances: StanceClassificationOutput        // per-concept stance classification
)
```

> **Aggregate + per-concept:** The top-level fields (`pork`, `impacts`, `stances`, `fiscal`) are the bill-wide aggregate outputs. `perConceptAnalysis` carries the per-concept breakdowns that produced them. This allows Pass 3 to reference specific concept-level findings.

### `DecompositionContext`

Runtime context for decomposition prompts. Simpler than `BillAnalysisContext` â€” operates on a single section or concept group at a time.

```
final case class DecompositionContext(
  billId: String,
  congress: Int,
  conceptGroupText: Option[String] = None,   // for concept-simplification profile
  sectionText: Option[String] = None,        // for section-classification profile
  sectionId: Option[String] = None           // which section is being classified
)
```

## Vectorization â€” Embedding Generation

Multiple text fields across the context and output types are vectorized (pgvector embeddings) for semantic search. Embedding generation is **not** Component 8's responsibility â€” the bill-analysis-pipeline (Component 10) generates embeddings after each analysis pass and stores them in AlloyDB. However, the types defined here identify which fields are embedding sources.

### Fields That Produce Embeddings

**Bill text layer** (decomposition artifacts â€” tied to text version):

| Source Type | Text Field | Storage Table / Column | DO | Purpose |
|------------|-----------|----------------------|-----|---------|
| Section text | `BillTextSectionDO.content` | `bill_text_sections.embedding` | `BillTextSectionDO` (Â§1.4) | Per-section semantic search across all bills |
| `ConceptGroup` | `simplifiedText` | `bill_concept_groups.embedding` | `BillConceptGroupDO` (Â§1.9) | Semantic search across concept groups within and across bills |

| Source Type | Text Field | Storage Table / Column | DO | Purpose |
|------------|-----------|----------------------|-----|---------|
| `BillSummaryOutput` (Pass 1 overall) | `summary` | `bill_analyses.embedding` | `BillAnalysisDO` (Â§1.9) | Bill-level semantic similarity search |
| `BillSummaryOutput` (per-concept) | `summary` | `bill_concept_summaries.embedding` | `BillConceptSummaryDO` (Â§1.9) | Per-concept analysis semantic search |

### Embedding Rules

- **Embedding model**: Determined by Component 10 configuration (e.g., OpenAI `text-embedding-3-small` at 1536 dimensions, matching the existing schema)
- **When generated**: After each analysis pass completes, Component 10 generates embeddings for all text outputs from that pass
- **Storage**: Embeddings are stored in AlloyDB in two layers: bill text layer (`BillTextSectionDO.embedding`, `BillConceptGroupDO.embedding` â€” Â§1.4/Â§1.9) and analysis layer (`BillAnalysisDO.embedding`, `BillConceptSummaryDO.embedding`, `BillFindingDO.embedding`, `AmendmentFindingDO.embedding` â€” Â§1.9)
- **HNSW indexes**: Defined in the schema (001 + 002 migrations) for cosine-distance search on all embedding columns
- **Component 8 does not generate embeddings** â€” it only assembles prompts. The vectorization concern is documented here because these types define the text fields that become embedding sources

## Context Serialization

### `BillContextInjector` â€” Object

Serializes structured context types into prompt-friendly text and injects them into Context stage blocks. This is more than placeholder replacement â€” it **formats** lists of structured objects into readable prompt sections.

```
object BillContextInjector {
  def serialize(context: BillAnalysisContext): Map[String, String]
  def serialize(context: DecompositionContext): Map[String, String]

  def inject(
    blocks: Map[String, InstructionBlock],
    contextMap: Map[String, String]
  ): Either[String, Map[String, InstructionBlock]]
}
```

**`serialize(BillAnalysisContext)`** produces a map of placeholder â†’ formatted text:

| Placeholder | Serialization |
|-------------|--------------|
| `{{bill_concepts}}` | Each `ConceptGroup` formatted as a titled section: concept title, topics, section refs, simplified text. Groups separated by dividers. |
| `{{amendments}}` | Each `AmendmentSummary` formatted with amendment ID, type, sponsor, description, purpose, status, and affected concepts. Numbered list. |
| `{{related_bills}}` | Each `RelatedBillSummary` formatted with bill ID, title, relationship type, summary, and topic overlap. Numbered list. |
| `{{pass1_results}}` | `Pass1Output` serialized as structured text: overall summary, then per-concept summaries with their topics. Each concept result formatted as a titled section (matching the concept serialization style). |
| `{{pass2_results}}` | `Pass2Output` serialized as structured text: aggregate findings (pork, impacts, stances, fiscal) followed by per-concept breakdowns. Each finding type formatted as a numbered list with severity and descriptions. |

Empty lists serialize to a "None" indicator (e.g., `"No amendments associated with this bill."`) rather than empty string, so the LLM knows the absence is intentional, not a data gap.

**`serialize(DecompositionContext)`** produces:

| Placeholder | Serialization |
|-------------|--------------|
| `{{concept_group_text}}` | Raw section text for the concept group (from `conceptGroupText`) |
| `{{section_text}}` | Raw section text (from `sectionText`) |

**`inject`** behavior:
1. For each block where `stage == Context`:
   - Scan `block.content` for `{{placeholder}}` tokens
   - For each token found, look up the value in `contextMap`
   - Replace the token with the serialized value
2. Non-Context blocks are passed through unchanged
3. Return `Left` if a placeholder token has no matching key in `contextMap` AND the block is referenced by the active profile

### Serialization Format

The serializer produces structured, readable text â€” not JSON. Example for concepts:

```
## Concept 1: Transportation Funding
Topics: infrastructure, federal-highways, public-transit
Sections: Â§101, Â§102, Â§105, Â§201
Summary: Authorizes $550 billion in new federal spending for highways,
bridges, and public transit systems over five years. Establishes a
competitive grant program for major infrastructure projects...

---

## Concept 2: Broadband Infrastructure
Topics: telecommunications, rural-access, digital-divide
Sections: Â§301, Â§302, Â§303
Summary: Allocates $65 billion to expand broadband internet access,

```

This format is LLM-optimized â€” section headers, topic tags, and dividers help the model parse the structure. The exact format is **not hardcoded** â€” it is determined by a `ContextFormatter` strategy that can be swapped or tuned. The default implementation uses the markdown-like format above.

### `ContextFormatter` â€” Trait

```
trait ContextFormatter {
  def formatConcepts(concepts: List[ConceptGroup]): String
  def formatAmendments(amendments: List[AmendmentSummary]): String
  def formatRelatedBills(bills: List[RelatedBillSummary]): String
  def formatPass1Results(results: Pass1Output): String
  def formatPass2Results(results: Pass2Output): String
}
```

Default implementation uses markdown-style formatting. Swappable for testing or if a different format yields better LLM results.

Pass result formatting follows the same markdown-style approach â€” per-concept results appear as titled sections (matching the concept format), with aggregate results in a separate "Overall" section. Example for Pass 1:

```

Summary: [overall BillSummaryOutput.summary]
Key Points: [overall keyPoints]
Topics: [overall topics with confidence scores]

---

---

## Pass 1 â€” Concept: Broadband Infrastructure
Summary: [per-concept summary]
Topics: telecommunications (0.91), rural-access (0.85)
```

## Key Design Decisions

- **Structured context, not strings** â€” `BillAnalysisContext` carries typed lists of concepts, amendments, and related bills rather than pre-formatted strings. This allows the assembler to format them optimally for the LLM and enables testing of the serialization logic independently.
- **`ContextFormatter` is swappable** â€” different LLMs may respond better to different formatting. The formatter is a trait, not a hardcoded implementation. The default is markdown-style.
- **Empty lists produce explicit "None" text** â€” so the LLM distinguishes "no amendments exist" from "amendments were not loaded."
- **Prior pass results are structured, not strings** â€” `Pass1Output` and `Pass2Output` carry per-concept breakdowns (summaries, topics, findings) plus bill-wide aggregates. The assembler serializes them into readable text via `ContextFormatter`, not as raw JSON. This gives the LLM structured context about what each prior pass found per concept group.
- **Vectorization is a downstream concern** â€” the text fields in these types (concept summaries, finding descriptions, etc.) produce embeddings for semantic search, but Component 8 does not generate embeddings. Component 10 handles embedding generation after analysis passes complete.
- **`affectedConcepts` on amendments is best-effort** â€” not all amendments can be linked to specific concept groups. The assembler includes the linkage when present but does not fail if absent.
- **No LLM calls** â€” the assembler produces a string. It never calls the LLM.
- **No bill data fetching** â€” the assembler receives pre-fetched context. Component 10 fetches the data and passes it in.
- **Version override per request** â€” the analysis pipeline can test a new prompt version for a specific bill without changing the default version.
