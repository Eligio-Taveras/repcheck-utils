# Area 11.10 — User-Bill Alignment

> Project: `stance-materializer` | Package: `repcheck.scoring.materializer.alignment`

## Purpose

Scheduled job that pre-computes per-bill, per-topic alignment between each user's stances and the bill's stance findings. Stores results in `user_bill_alignments` and `user_amendment_alignments` for consumption by the scoring orchestrator (§11.7) and score explainer (§11.4). This is Process D from BEHAVIORAL_SPECS.md §3.3.

---

## Types

### `AlignmentReasoningOutput` — LLM Structured Output (11.10)

The LLM structured output type for alignment reasoning. A single user-bill pair can have multiple overlapping topics, and the LLM produces reasoning for each in a single call (one LLM call per (user, bill) pair covering all overlapping topics). The LLM receives the user's stance on each topic, the bill's stance findings per topic, and produces per-topic alignment reasoning.

```scala
case class AlignmentReasoningOutput(
  alignments: List[TopicAlignmentReasoning]
)
case class TopicAlignmentReasoning(
  topic: String,
  reasoning: String,          // LLM explanation of alignment/misalignment
  confidence: Option[Double]  // 0.0-1.0
)
```

### `AlignmentRunSummary` — Run Result (11.10)

```scala
case class AlignmentRunSummary(
  billsProcessed: Int,        // distinct bills aligned
  usersProcessed: Int,        // distinct users aligned (a user processing multiple bills counts once)
  alignmentsCreated: Int      // total user_bill_alignments rows upserted
)
```

## Classes

### `UserBillAligner[F[_]]` — Scheduled Alignment Computation (11.10)
**Package:** `repcheck.scoring.materializer.alignment`

```scala
class UserBillAligner[F[_]: Async](
  xa: Transactor[F],
  stanceAggregator: UserStanceAggregator[F],
  llmProvider: LlmProvider[F],
  retryWrapper: LlmRetryWrapper[F],
  embeddingService: SemanticEmbeddingService[F],
  promptAssembler: UserPromptAssembler[F],
  config: UserBillAlignmentConfig
)
```

#### `alignAll(): F[AlignmentRunSummary]`

Entry point for the scheduled job.

1. **Find bills with changed stances:**
   ```sql
   SELECT bill_id FROM ${Tables.StanceMaterializationStatus}
   WHERE stances_materialized_at IS NOT NULL
     AND stances_materialized_at > COALESCE(last_scoring_run_at, '1970-01-01'::timestamptz)
   ```

2. **Find users with changed Q&A responses** (optional — only if user preferences changed):
   Users whose `last_stance_change_at` indicates new Q&A data since last alignment run.

3. **For each changed bill via `parEvalMap(config.billParallelism)`:**
   a. Fetch all `member_bill_stance_topics` for the bill → get unique topics (`stanceTopics`)
   b. Find users whose topics overlap with the bill's topics
   c. For each overlapping user via `parEvalMap(config.userParallelism)`:
      ```
      userProfile = stanceAggregator.aggregate(userId)  // returns UserTopicProfile from §11.1
      For each overlapping topic:
        userStanceScore = userProfile.topics.find(_.topic == topic).map(_.stanceScore).getOrElse(0.0)
        billStanceDirection = stanceTopic.stanceDirection  // from member_bill_stance_topics
        alignUserBill(userId, billId, topic, userStanceScore, billStanceDirection)
      ```

   Parallelism is two-level: `billParallelism` controls concurrent bills in the outer loop, `userParallelism` controls concurrent users within each bill's inner loop. Total concurrent tasks = `billParallelism * userParallelism`. Configure conservatively to avoid overwhelming the LLM provider rate limits.

4. **Update `last_scoring_run_at`** on processed bills:
   ```sql
   UPDATE ${Tables.StanceMaterializationStatus}
   SET last_scoring_run_at = NOW()
   WHERE bill_id IN ($processedBillIds)
   ```

5. Return `AlignmentRunSummary(billsProcessed, usersProcessed, alignmentsCreated)`

#### `alignUserBill(userId: UUID, billId: String, topic: String, userStanceScore: Double, billStanceDirection: String): F[UserBillAlignmentDO]`

Computes alignment for a single (user, bill, topic) combination.

1. **Compute alignment score:**
   - Convert bill stance to numeric: `Progressive → +1.0, Conservative → -1.0, Neutral → 0.0`
   - `alignmentScore = 1.0 - abs(userStanceScore - billStanceNumeric) / 2.0`
   - Bounded to [0.0, 1.0]

2. **Generate reasoning** (LLM, non-fatal):
   Note: In practice, the LLM call is batched per (user, bill) pair — one call produces `AlignmentReasoningOutput` covering all overlapping topics. The per-topic reasoning is extracted from the batch result:
   ```scala
   val batchReasoning: Option[AlignmentReasoningOutput] = retryWrapper.withRetry {
     llmProvider.completeStructured[AlignmentReasoningOutput](
       LlmRequest(prompt = promptAssembler.assembleAlignmentReasoning(
         userId, billId, overlappingTopics, userProfile, stanceTopics
       ))
     )
   }.map(Some(_)).recover { case _ => None }

   // Extract per-topic reasoning from batch result
   val reasoning: Option[String] = batchReasoning.flatMap(
     _.alignments.find(_.topic == topic).map(_.reasoning)
   )
   ```

3. **Generate reasoning embedding** (non-fatal, DJL/ONNX):
   ```scala
   val reasoningEmbedding: Option[Array[Float]] = reasoning.flatTraverse { r =>
     embeddingService.embed(r).map(Some(_)).recover { case _ => None }
   }
   ```

4. **Find matching finding** for provenance:
   ```sql
   SELECT finding_id FROM ${Tables.BillFindings} bf
   JOIN ${Tables.BillAnalyses} ba ON bf.analysis_id = ba.analysis_id
   JOIN ${Tables.FindingTypes} ft ON bf.finding_type_id = ft.finding_type_id
   WHERE ba.bill_id = $billId AND ft.code = 'stance'
   ORDER BY ba.completed_at DESC LIMIT 1
   ```

5. **Build and upsert:**
   ```scala
   UserBillAlignmentDO(userId, billId, topic, userStanceScore, billStanceDirection,
     alignmentScore, reasoning, reasoningEmbedding, findingId, computedAt = now)
   ```
   ```sql
   INSERT INTO ${Tables.UserBillAlignments} (...)
   ON CONFLICT (user_id, bill_id, topic)
   DO UPDATE SET alignment_score = EXCLUDED.alignment_score,
                 reasoning = EXCLUDED.reasoning, reasoning_embedding = EXCLUDED.reasoning_embedding,
                 finding_id = EXCLUDED.finding_id, computed_at = EXCLUDED.computed_at
   ```

#### `alignUserAmendment(userId: UUID, amendmentId: String, billId: Option[String], topic: String, userStanceScore: Double, amendmentStanceDirection: String): F[UserAmendmentAlignmentDO]`

Same logic as `alignUserBill` but for amendment alignments. Upserts to `user_amendment_alignments`.

`billId` is `Option[String]` because standalone amendments (not attached to a bill) may exist. In practice, most amendments have a parent bill. When `billId` is `None`, the alignment is computed solely from the amendment's own stance findings.

## Key Design Decisions

1. **Changed-bill optimization.** Only process bills where `stances_materialized_at > last_scoring_run_at`. Bills whose stances haven't changed since the last alignment run are skipped.
2. **Batched LLM calls per (user, bill) pair.** One LLM call per (user, bill) pair produces `AlignmentReasoningOutput` covering all overlapping topics. This avoids N separate LLM calls per topic while still providing specific, contextual per-topic explanations.
3. **Reasoning + embedding stored.** Each alignment has reasoning text and a vector(1536) embedding for semantic search ("find bills where the alignment reasoning mentions tax policy").
4. **Finding FK for provenance.** Links back to the `bill_findings` row that determined the bill's stance on that topic.
5. **Upsert semantics.** Re-running alignment for the same (user, bill, topic) overwrites the previous result. This is safe and idempotent.
6. **`last_scoring_run_at` updated after alignment.** This timestamp prevents the same bill from being re-processed on the next run unless new stances arrive.

## Acceptance Criteria

| Criterion | Test |
|-----------|------|
| `alignAll` finds bills with stances newer than last scoring run | `stances_materialized_at > last_scoring_run_at` → bill included |
| `alignAll` skips already-processed bills | `stances_materialized_at < last_scoring_run_at` → bill skipped |
| `alignAll` processes bills in parallel | `billParallelism = 4` → `parEvalMap(4)` |
| `alignAll` processes users within each bill in parallel | `userParallelism = 8` → inner `parEvalMap(8)` |
| Nested parallelism bounded by config | Total concurrent tasks = `billParallelism * userParallelism` |
| `alignUserBill` computes correct alignment score | user stance +0.8, bill "Progressive" (+1.0) → `alignment = 1.0 - abs(0.8 - 1.0)/2.0 = 0.9` |
| Alignment score bounded to [0.0, 1.0] | Extreme inputs → score clamped |
| LLM reasoning batched per (user, bill) pair | One LLM call → `AlignmentReasoningOutput` with multiple `TopicAlignmentReasoning` entries |
| Per-topic reasoning extracted from batch result | `batchReasoning.alignments.find(_.topic == topic)` → per-topic reasoning |
| LLM reasoning failure is non-fatal | LLM throws → `reasoning = None`, alignment still stored |
| Reasoning embedding generated when reasoning succeeds | `reasoning = Some("...")` → embedding generated |
| Reasoning embedding failure is non-fatal | `embeddingService.embed` throws → `reasoningEmbedding = None` |
| Finding FK linked to latest stance finding | `findingId` from latest completed analysis |
| Alignment upserted with ON CONFLICT | Re-run → existing row updated |
| `last_scoring_run_at` updated after processing | Processed bills → `last_scoring_run_at = NOW()` |
| `userStanceScore` sourced from `UserStanceAggregator` | `stanceAggregator.aggregate(userId)` → `userProfile.topics.find(_.topic).stanceScore` |
| Amendment alignment follows same pattern | `alignUserAmendment` → `user_amendment_alignments` upserted |
| Amendment `billId` is optional | `billId = None` → alignment computed from amendment findings only |
| `AlignmentRunSummary` accurately reflects work | 10 bills, 5 users, 30 alignments → summary matches |
| No alignment for non-overlapping topics | User topics = ["healthcare"], bill topics = ["defense"] → no alignment row |
