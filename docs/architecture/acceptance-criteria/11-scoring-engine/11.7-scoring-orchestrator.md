# Area 11.7 — Scoring Orchestrator

> Project: `scoring-pipeline` | Package: `repcheck.scoring.pipeline.orchestration`

## Purpose

Coordinates all scoring steps for a single user across all their paired legislators. Uses a two-phase approach: Phase 1 computes and persists numeric scores (immediately visible), Phase 2 generates and persists LLM explanations (non-blocking). Called by the entry point (§11.8) for each user in the scoring run.

---

## Classes

### `ScoringOrchestrator[F[_]]` — Per-User Coordination (11.7)
**Package:** `repcheck.scoring.pipeline.orchestration`

```scala
class ScoringOrchestrator[F[_]: Async](
  pairingReader: PairingValidator[F],
  stanceAggregator: UserStanceAggregator[F],
  profileBuilder: LegislatorProfileBuilder[F],
  calculator: AlignmentScoreCalculator,
  evidenceFetcher: AlignmentEvidenceFetcher[F],
  explainer: ScoreExplainer[F],
  embeddingService: SemanticEmbeddingService[F],
  scorePersister: ScorePersister[F],
  config: ScoringOrchestratorConfig
)
```

#### `scoreUser(userId: UUID, correlationId: UUID, source: String): F[UserScoringResult]`

> `source` is `"scheduled"` or `"ad-hoc"`. This value is passed through to `ScoreHistoryDO.triggerEvent`.

1. **Read pairings:**
   ```scala
   pairingReader.findPairingsForUser(userId)
   ```
   If empty: return `UserScoringResult.Skipped(userId, correlationId, "no legislator pairings")`

2. **Aggregate user stances:**
   ```scala
   stanceAggregator.aggregate(userId)
   ```
   If `userProfile.topics.isEmpty`: return `UserScoringResult.Skipped(userId, correlationId, "no user Q&A responses")`

3. **Phase 1 — Numeric Scores (parallel):**
   For each pairing via `parEvalMap(config.scoringParallelism)`:
   ```
   attempt {
     a. profileBuilder.buildProfile(memberId, userProfile.topics.map(_.topic))
     b. profileBuilder.buildCongressProfiles(memberId, userProfile.topics.map(_.topic))
     c. calculator.calculate(userProfile, legislatorProfile, congressProfiles)
     d. Check result status:
        - If result.status == "no_overlap": build no-overlap DOs
          nonOverlappingTopics = userProfile.topics.map(_.topic).filterNot(sharedTopics.contains)
        - Else: build scored DOs with result.aggregateScore
     e. For scored results, also compute:
        nonOverlappingTopics = userProfile.topics.map(_.topic).filterNot(result.sharedTopics.contains)
        (partial overlap is common — nonOverlappingTopics is populated for BOTH scored and no_overlap cases)
     f. Build ScoreDO, List[ScoreTopicDO], List[ScoreCongressDO], List[ScoreCongressTopicDO]
        (without reasoning/embedding — Phase 2 fills those in)
     g. Generate scoreId = UUID.randomUUID() for ScoreHistoryDO
        Build ScoreHistoryDO (with triggerEvent = source), List[ScoreHistoryCongressDO],
        List[ScoreHistoryCongressTopicDO] (without reasoning/embedding/highlights — Phase 2 fills those in)
        Include scoreId in the result so Phase 2 can call updateHistoryReasoning(scoreId, ...)
   }.recover {
     case e => LegislatorScoringFailed(memberId, e.getMessage) // counted in membersFailed
   }
   ```
   > Per-legislator failures do not abort other legislators. Each legislator is independently attempted.

4. **Persist all numeric scores:**
   ```scala
   allScoreResults.traverse_ { result =>
     scorePersister.upsertScore(result.score, result.topics, result.congress, result.congressTopics, result.status, result.nonOverlappingTopics) >>
     scorePersister.appendHistory(result.history, result.historyCongress, result.historyCongressTopics, highlights = Nil, status = result.status)
   }
   ```
   > Scores are now visible to the frontend. Reasoning will be backfilled in Phase 2.

5. **Phase 2 — LLM Explanations (parallel):**
   For each scored pairing (skip no-overlap) via `parEvalMap(config.explanationParallelism)`:
   a. Fetch evidence:
      ```scala
      val sharedTopics = result.topics.map(_.topic).toSet
      val evidence = evidenceFetcher.fetchEvidence(memberId, userId, userProfile, sharedTopics)
        .recover { case _ => Nil }
      ```
   b. Fetch user-bill alignments for shared topics
   c. Generate explanation (non-fatal):
      ```scala
      val explanation: Option[AlignmentScoreOutput] =
        explainer.explain(ScoringContext(userId, memberId, userProfile, legislatorProfile, calculationResult = result, evidence, userBillAlignments))
          .map(Some(_))
          .recover { case _ => None }
      ```
   d. Generate reasoning embedding (non-fatal):
      ```scala
      val reasoningEmbedding: Option[Array[Float]] =
        explanation.flatTraverse { e =>
          embeddingService.embed(e.reasoning)
            .map(Some(_))
            .recover { case _ => None }
        }
      ```
   e. Update score with reasoning:
      ```scala
      explanation.traverse_ { e =>
        scorePersister.updateScoreReasoning(userId, memberId, e.reasoning, reasoningEmbedding.getOrElse(Array.empty))
      }
      ```
   f. Update history with reasoning + highlights:
      ```scala
      explanation.traverse_ { e =>
        scorePersister.updateHistoryReasoning(scoreId, e.reasoning, reasoningEmbedding.getOrElse(Array.empty))
      }
      // Highlights are persisted as part of history — appendHistory in Phase 1 passes highlights = Nil,
      // and here we upsert highlights from the LLM explanation:
      explanation.traverse_ { e =>
        scorePersister.appendHistoryHighlights(scoreId, e.highlights)
      }
      ```

6. **Return:**
   ```scala
   UserScoringResult.Succeeded(
     userId, correlationId,
     memberScoreCount = pairings.size,
     membersScored = scoredCount,
     membersFailed = failedCount,
     membersNoOverlap = noOverlapCount
   )
   ```

> **`result.aggregateScore` is authoritative.** The LLM score is not stored as the alignment score. Phase 2 only adds reasoning text and highlights.

> **Explanation failure is non-fatal.** If `ScoreExplainer.explain` throws, `explanation = None`. The score persisted in Phase 1 remains valid with `reasoning = None`. `UserScoringResult.Succeeded` is still returned.

> **Embedding failure is non-fatal.** If `SemanticEmbeddingService.embed` throws, `reasoningEmbedding = None`. The score is persisted without an embedding. In-process DJL/ONNX — failures are rare but possible.

> **Evidence fetch failure is non-fatal.** If `AlignmentEvidenceFetcher.fetchEvidence` throws, the orchestrator catches the error and passes an empty evidence list to the explainer. The LLM can still produce generic reasoning.

> **No-overlap score is actionable.** A member who has no votes on topics the user cares about is persisted with `status = "no_overlap"` and `nonOverlappingTopics`. The frontend uses this to tell the user "This member has not voted on any of your topics" and to suggest additional surveys.

> **Two-phase persistence guarantees fast score visibility.** Phase 1 persists numeric scores immediately. Phase 2 (LLM explanations) can take minutes for all legislators — scores are visible to the user long before explanations complete.

### `UserScoringResult` — Per-User Outcome (11.7)
**Package:** `repcheck.scoring.pipeline.orchestration`

```scala
sealed trait UserScoringResult {
  def userId: UUID
  def correlationId: UUID
}
object UserScoringResult {
  case class Succeeded(
    userId: UUID,
    correlationId: UUID,
    memberScoreCount: Int,       // total legislators paired
    membersScored: Int,          // legislators successfully scored (numeric)
    membersFailed: Int,          // legislators where scoring threw an exception
    membersNoOverlap: Int        // legislators with status = "no_overlap"
  ) extends UserScoringResult
  case class Skipped(userId: UUID, correlationId: UUID, reason: String) extends UserScoringResult
  case class Failed(userId: UUID, correlationId: UUID, reason: String) extends UserScoringResult
}
```

> **Invariant:** `memberScoreCount = membersScored + membersFailed + membersNoOverlap`

## Key Design Decisions

1. **Per-user, not per-pair.** The orchestrator processes all legislators for a single user in one call. This enables batch optimizations (user stance aggregation runs once, shared across all legislators).
2. **Two-phase persistence.** Phase 1 (numeric scores) persists immediately. Phase 2 (LLM explanations) runs after scores are visible. This prevents slow LLM calls from blocking score visibility.
3. **Skip on empty user profile.** A user with no Q&A responses has no preferences to align against. `Skipped` is appropriate — not an error, and no database write is needed.
4. **`no_overlap` status on no shared topics.** If a member has no votes on any topic the user cares about, we persist with `status = "no_overlap"`, `aggregateScore = 0.0`, and `nonOverlappingTopics`.
5. **Explanation failure is non-fatal.** The algorithmic score is the deliverable. The LLM explanation is enrichment.
6. **Reasoning embedding generated in-process.** The `SemanticEmbeddingService` (DJL/ONNX) generates a vector(1536) embedding for each reasoning text. No API cost, ~1ms latency.
7. **`upsertScore` before `appendHistory` in Phase 1.** The current score (frontend reads) is the higher-priority write.
8. **`correlationId` from input event.** Carried from the scheduling/ad-hoc event through the orchestrator and into results.

## Acceptance Criteria

| Criterion | Test |
|-----------|------|
| Empty pairings → `UserScoringResult.Skipped` | No pairings → skipped, no DB writes |
| Empty `UserTopicProfile.topics` → `UserScoringResult.Skipped` | No Q&A responses → skipped, no DB writes, no LLM call |
| Phase 1: numeric scores persisted for all legislators in parallel | 3 legislators → 3 `upsertScore` calls in parallel |
| Phase 1: history appended for all legislators | 3 legislators → 3 `appendHistory` calls (without highlights) |
| Phase 2: LLM explanations generated after Phase 1 | Mock ordering: all `upsertScore` calls before any `explain` call |
| Phase 2: reasoning and embedding updated on existing scores | `updateReasoning` called with explanation text and embedding |
| Phase 2: highlights appended to history | `appendHistoryHighlights` called with LLM highlights |
| No-overlap → score persisted with `status = "no_overlap"`, LLM skipped | `result.status = "no_overlap"` → `explainer.explain` not called |
| No-overlap score has `aggregateScore = 0.0`, `totalBills = 0, totalVotes = 0` | Persisted `ScoreDO` verified |
| No-overlap score carries `nonOverlappingTopics` | User topics = ["healthcare", "economy"] → `nonOverlappingTopics` populated |
| `result.aggregateScore` used for `ScoreDO.aggregateScore` (not LLM score) | `calculationResult.aggregateScore = 0.73` → `ScoreDO.aggregateScore = 0.73` |
| LLM `overallScore` NOT written to `scores.aggregate_score` | `AlignmentScoreOutput.overallScore = 0.68` → not persisted as the score |
| Explanation failure → score still persisted with `reasoning = None` | `ScoreExplainer.explain` throws → `Succeeded`, reasoning = None |
| Explanation success → reasoning stored in both scores and history | Reasoning text → `updateReasoning` and `updateHistoryReasoning` called |
| Highlights from LLM stored as `ScoreHistoryHighlightDO` rows | 3 highlights → 3 rows passed to `appendHistoryHighlights` |
| Evidence fetch failure → explanation still attempted with empty evidence | `evidenceFetcher.fetchEvidence` throws → `explainer.explain` called with `evidence = Nil` |
| Reasoning embedding generated when explanation succeeds | `explanation = Some(...)` → `embeddingService.embed(reasoning)` called |
| Reasoning embedding is `None` when explanation fails | `explanation = None` → `embeddingService.embed` not called |
| Reasoning embedding failure is non-fatal | `embeddingService.embed` throws → `reasoningEmbedding = None`, score persisted |
| Same reasoning embedding written to both scores and history | `reasoningEmbedding = Some(vec)` → both updates carry same vec |
| No-overlap score has `reasoningEmbedding = None` | No-overlap → no embedding generated |
| `ScoreCongressDO` built from `result.perCongressScores` | perCongressScores = {118 → 0.7, 119 → 0.8} → 2 congress rows |
| `UserScoringResult.Succeeded` returned on full success | All steps succeed → `Succeeded` with memberScoreCount |
| `Succeeded` includes `membersScored`, `membersFailed`, `membersNoOverlap` counts | 3 legislators (2 scored, 1 no-overlap) → `membersScored=2, membersFailed=0, membersNoOverlap=1` |
| `memberScoreCount = membersScored + membersFailed + membersNoOverlap` invariant | Counts always sum to `pairings.size` |
| `source` parameter propagated to `ScoreHistoryDO.triggerEvent` | `scoreUser(id, corr, "scheduled")` → history row `trigger_event = "scheduled"` |
| `nonOverlappingTopics` computed for no-overlap case | User topics = ["healthcare", "economy"], no shared → `nonOverlappingTopics = ["healthcare", "economy"]` |
| `nonOverlappingTopics` computed for scored case (partial overlap) | User: healthcare, defense; legislator: healthcare only → `nonOverlappingTopics = ["defense"]` |
| Per-legislator failure does not abort other legislators | Legislator A throws, B and C succeed → `membersScored=2, membersFailed=1` |
| Per-legislator failure counted in `membersFailed` | `profileBuilder.buildProfile` throws for one legislator → `membersFailed` incremented |
| `scoreId` generated per legislator and threaded to Phase 2 | Phase 2 `updateHistoryReasoning` called with same `scoreId` from Phase 1 |
| `UserScoringResult.Failed` returned when top-level step throws | `stanceAggregator.aggregate` throws → `Failed` (not per-legislator) |
| `correlationId` propagated to `UserScoringResult` | Input → same value in returned result |
| `triggerEvent` in history is "scheduled" or "ad-hoc" | History row has `trigger_event` matching source |
| User stance aggregated once, shared across all legislators | `stanceAggregator.aggregate` called exactly once per `scoreUser` |
