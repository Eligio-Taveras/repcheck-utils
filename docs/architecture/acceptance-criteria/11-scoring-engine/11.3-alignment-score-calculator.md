# Area 11.3 ‚Äî Alignment Score Calculator

> Project: `scoring-pipeline` | Package: `repcheck.scoring.pipeline.scoring`

## Purpose

Computes per-topic and aggregate alignment scores by comparing the user's topic stance profile against the legislator's topic stance profile. Pure in-memory computation ‚Äî no effects, no LLM, no I/O. This class produces the **authoritative alignment score** that is written to `scores.aggregate_score`.

## Classes

### `AlignmentScoreCalculator` ‚Äî Pure Alignment Computation (11.3)
**Package:** `repcheck.scoring.pipeline.scoring`

No type parameter ‚Äî pure class with no effects. **Stateless, pure computation class** with no constructor dependencies: no `F[_]`, no database access, no configuration. Instantiate directly with `new AlignmentScoreCalculator`. All inputs arrive via method parameters; all outputs are returned as values.

```scala
class AlignmentScoreCalculator
```

#### `calculate(userProfile: UserTopicProfile, legislatorProfile: LegislatorTopicProfile, congressProfiles: Map[Int, LegislatorTopicProfile]): AlignmentCalculationResult`

> **ID propagation:** `userId` comes from `userProfile.userId` (defined in ¬ß11.1) and `memberId` comes from `legislatorProfile.memberId` (defined in ¬ß11.2). These are passed through to the result: `result.userId = userProfile.userId`, `result.memberId = legislatorProfile.memberId`. If `LegislatorTopicProfile` does not yet include a `memberId: String` field, it must be added (see ¬ß11.2).

1. **Find shared topics:** Intersect topic names in `userProfile.topics` and `legislatorProfile.topics`
2. **Return neutral result if no shared topics:**
   - `AlignmentCalculationResult(userProfile.userId, legislatorProfile.memberId, topics = Nil, aggregateScore = 0.5, perCongressScores = Map.empty)`
   - Note: `perCongressScores` is an empty map ‚Äî no per-congress entries are generated when there are no all-time shared topics.
3. **For each shared topic:**
   - `userEntry = userProfile.topics.find(_.topic == topic).get`
   - `legEntry = legislatorProfile.topics.find(_.topic == topic).get`
   - `rawAlignment = computeAlignment(userEntry.stanceScore, legEntry.stanceScore)`
   - `weightedAlignment = rawAlignment √ó userEntry.importance`
   - Build `PerTopicAlignment(topic, userEntry.stanceScore, legEntry.stanceScore, rawAlignment, weightedAlignment, userEntry.importance, legEntry.billsConsidered, legEntry.votesAnalyzed)`
4. **Compute aggregate score:**
   - `aggregateScore = topicAlignments.map(_.weightedAlignment).sum / userProfile.topics.filter(sharedTopics.contains).map(_.importance).sum`
   - This is the importance-weighted average: sum of (rawAlignment √ó importance) / sum of importance for shared topics
   - **Assumption:** User topic importances are pre-normalized to sum to 1.0 by `UserStanceAggregator` (¬ß11.1). The denominator `sum(importance for shared topics)` may be less than 1.0 when only a subset of user topics overlap with the legislator's. This re-normalization ensures the aggregate score uses the full [0.0, 1.0] range even when overlap is partial.
5. **Compute per-congress scores:**
   - For each `(congress, congressLegProfile)` in `congressProfiles`:
     - Find shared topics between `userProfile.topics` and `congressLegProfile.topics` **for this congress**. Shared topics are determined fresh per congress: a topic that appears in congress 118 but not 119 is included only in the 118 result.
     - If this congress has no shared topics with the user, **omit it from the map** (do not include it with a 0.5 default).
     - Otherwise, apply same formula as steps 3‚Äì4 scoped to this congress's legislator data
     - `perCongressScores(congress) = weightedAverageForCongress`
6. Return `AlignmentCalculationResult`

#### `private def computeAlignment(userStance: Double, legislatorStance: Double): Double`

Linear distance formula converting stance similarity to [0.0, 1.0]:

```
difference = abs(userStance - legislatorStance)
// difference range: [0.0, 2.0] ‚Äî 0.0 = identical, 2.0 = opposite ends of spectrum
alignment = 1.0 - (difference / 2.0)
// alignment: 1.0 = perfect agreement, 0.0 = complete opposition, 0.5 = 1 unit apart
```

Example calculations:
- `userStance = +0.8, legislatorStance = +0.7` ‚Üí `difference = 0.1` ‚Üí `alignment = 0.95`
- `userStance = +1.0, legislatorStance = -1.0` ‚Üí `difference = 2.0` ‚Üí `alignment = 0.0`
- `userStance = +0.5, legislatorStance = 0.0` ‚Üí `difference = 0.5` ‚Üí `alignment = 0.75`
- `userStance = 0.0, legislatorStance = 0.0` ‚Üí `difference = 0.0` ‚Üí `alignment = 1.0`
- `userStance = -0.5, legislatorStance = +0.5` ‚Üí `difference = 1.0` ‚Üí `alignment = 0.5`

## Key Design Decisions

1. **Authoritative score.** This calculator produces the `aggregate_score` stored in the database. The LLM explanation (¬ß11.4) may compute its own `overallScore` for narrative context, but only the algorithmic result from this class is persisted to `scores.aggregate_score`.
2. **Importance-weighted average.** Topics the user cares about most dominate the aggregate. A user passionate about healthcare (importance 0.5) who aligns well with the legislator on healthcare is scored higher than if healthcare were a minor interest.
3. **0.5 neutral on no shared topics.** When a legislator has no votes on topics the user cares about, returning 0.5 (neutral) rather than 0.0 (maximum opposition) is more accurate. There is no evidence of misalignment ‚Äî there is simply no data.
4. **Linear distance is transparent and symmetric.** The formula is easy for users to understand: "how far apart are you on each topic?" Direction of disagreement doesn't change the score magnitude ‚Äî only the degree of difference matters.
5. **Per-congress scores are independent.** Each congress is computed on the legislator data scoped to that congress, using the same formula as the overall aggregate. This enables "how aligned were you in the 118th Congress?" comparisons.

## Acceptance Criteria

| Criterion | Test |
|-----------|------|
| Identical stances ‚Üí alignment 1.0 | `userStance = +0.8, legStance = +0.8` ‚Üí `rawAlignment = 1.0` |
| Opposite stances ‚Üí alignment 0.0 | `userStance = +1.0, legStance = -1.0` ‚Üí `rawAlignment = 0.0` |
| Both neutral stances ‚Üí alignment 1.0 | `userStance = 0.0, legStance = 0.0` ‚Üí `rawAlignment = 1.0` |
| Moderate disagreement ‚Üí alignment 0.75 | `userStance = +0.5, legStance = 0.0` ‚Üí `difference = 0.5` ‚Üí `alignment = 0.75` |
| `computeAlignment` is symmetric | `computeAlignment(a, b) == computeAlignment(b, a)` for all inputs |
| `computeAlignment` output always in [0.0, 1.0] | Edge cases and boundary inputs ‚Üí no value outside range |
| Aggregate score weighted by user importance | healthcare importance 0.8, defense 0.2; alignment 0.9 on healthcare, 0.1 on defense ‚Üí `aggregate = (0.9√ó0.8 + 0.1√ó0.2) / 1.0 = 0.74` |
| No shared topics ‚Üí aggregate 0.5, empty topics list | User topics not in legislator profile ‚Üí `aggregateScore = 0.5`, `topics = Nil` |
| Only shared topics included in `PerTopicAlignment` list | User has healthcare, defense; legislator only has healthcare ‚Üí 1 entry |
| `billsConsidered` and `votesAnalyzed` propagated from legislator entry | Leg entry: `billsConsidered = 5, votesAnalyzed = 7` ‚Üí PerTopicAlignment carries same values |
| `perCongressScores` computed independently per congress | Two congresses with different alignment data ‚Üí two different scores in map |
| Per-congress score uses same formula as overall | Single congress with 2 shared topics ‚Üí `weightedAverage` matches manual calculation |
| Per-congress returns empty map when no congress data | Empty `congressProfiles` ‚Üí `perCongressScores = Map.empty` |
| `weightedAlignment = rawAlignment √ó userImportance` | `rawAlignment = 0.8, importance = 0.6` ‚Üí `weightedAlignment = 0.48` |
| Aggregate score in [0.0, 1.0] for all valid inputs | Exhaustive edge case testing ‚Üí always within bounds |
