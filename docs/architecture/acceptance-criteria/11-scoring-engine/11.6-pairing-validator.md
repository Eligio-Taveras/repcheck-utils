# Area 11.6 ‚Äî Pairing Validator

> Project: `scoring-pipeline` | Package: `repcheck.scoring.pipeline.pairing`

## Purpose

Manages `user_legislator_pairings` ‚Äî the persistent record of which legislators represent each user. Created at signup, validated on schedule, and updated when a user's location changes or a `member.updated` event arrives. Replaces the event-driven `AffectedPairFinder` from the original design.

**Scoping rule: a user is only ever scored against their own representatives.** This invariant is enforced at the pairing level ‚Äî the scoring orchestrator (¬ß11.7) reads pairings and scores only against paired legislators. No per-event pair resolution needed.

class PairingValidator[F[_]: Async](

#### `createPairings(userId: UUID): F[List[UserLegislatorPairingDO]]`

Called at user signup. Creates pairing rows for the user's current legislators.

1. Look up user's state code and district:
   ```sql
   SELECT state, district FROM ${Tables.Users} WHERE user_id = $userId
   ```
2. Find current legislators for that state/district:
   ```sql
   SELECT DISTINCT mt.member_id, m.state, m.district, m.chamber
   FROM ${Tables.MemberTerms} mt
   JOIN ${Tables.Members} m ON mt.member_id = m.member_id
   JOIN ${Tables.UsStates} s ON s.state_name = m.state
   WHERE s.state_code = $stateCode
     AND (m.district IS NULL OR m.district = $userDistrict)
     AND (mt.end_year IS NULL OR mt.end_year >= $currentYear)
   ```
   - `district IS NULL` matches senators (serve entire state)
   - `end_year IS NULL OR end_year >= currentYear` selects current or future-term members
3. For each matched legislator:
   ```sql
   INSERT INTO ${Tables.UserLegislatorPairings}
     (user_id, member_id, state, district, chamber, paired_at, validated_at)
   VALUES ($userId, $memberId, $stateCode, $district, $chamber, NOW(), NOW())
   ON CONFLICT (user_id, member_id) DO UPDATE SET
     state = EXCLUDED.state,
     district = EXCLUDED.district,
     chamber = EXCLUDED.chamber,
     validated_at = NOW()
   ```
4. Return the created/updated pairings

#### `validateAll(): F[ValidationSummary]`

Scheduled job. Scans all users and validates their pairings against current `member_terms`.

Each user is validated in a separate transaction. If validation fails for one user, other users are unaffected. Failures are logged and counted but do not abort the batch.

1. For each user with pairings:
   - Re-run the legislator lookup (same query as `createPairings` step 2)
   - **Add** new legislators not yet paired (newly elected, newly mapped senators) ‚Äî use the same upsert as `createPairings` step 3
   - **Remove** stale pairings where the member no longer serves (term ended):
     ```sql
     DELETE FROM user_legislator_pairings
     WHERE user_id = $userId
       AND member_id NOT IN (
         SELECT m.member_id FROM member_terms mt
         JOIN members m ON m.member_id = mt.member_id
         JOIN us_states s ON s.state_name = m.state
         WHERE s.state_code = $userStateCode
           AND (m.district IS NULL OR m.district = $userDistrict)
           AND (mt.end_year IS NULL OR mt.end_year >= $currentYear)
       )
     ```
   - **Update** `validated_at` on existing valid pairings
2. Return `ValidationSummary(usersChecked, pairingsAdded, pairingsRemoved, pairingsUpdated)`

#### `validateForUser(userId: UUID): F[List[UserLegislatorPairingDO]]`

Called when a user updates their location in the app. Same logic as one iteration of `validateAll` for a single user.

#### `validateForMember(memberId: String): F[Int]`

Called when `member.updated` event arrives. Checks if the member's terms have changed and updates affected pairings.

1. Find all users currently paired with this member
2. For each user: verify the pairing is still valid (member still serves their state/district)
3. Remove invalid pairings, update `validated_at` on valid ones
4. Find users in the member's state/district who are NOT yet paired ‚Üí add new pairings
5. Return count of pairings modified

#### `findPairingsForUser(userId: UUID): F[List[UserLegislatorPairingDO]]`

Read-only lookup used by the scoring orchestrator (¬ß11.7) to get the user's current legislators.

```sql
SELECT * FROM ${Tables.UserLegislatorPairings} WHERE user_id = $userId
```

## Key Design Decisions

1. **Persistent pairings replace per-event pair resolution.** The original design resolved affected pairs on every event. Persisting pairings at signup means the scoring orchestrator simply reads a table ‚Äî no complex joins per scoring run.
2. **User-to-representative join via state name/code.** Members store state as full name (e.g., "Vermont"); users store as state code (e.g., "VT"). The join goes through `us_states` to convert.
3. **Senators matched for all users in the state.** Senators have `district IS NULL` ‚Äî they represent the entire state. The query uses `district IS NULL` to match all users in that state regardless of district.
4. **Validation is scheduled, not real-time.** Legislator terms don't change frequently. A daily or weekly validation job is sufficient to catch term changes, special elections, and redistricting.
5. **`member.updated` handles incremental changes.** Between scheduled validations, the `member.updated` event provides timely pairing updates when a specific member's data changes (e.g., LIS mapping for a new senator).
6. **Upsert semantics on creation.** `ON CONFLICT DO UPDATE SET state, district, chamber, validated_at` ensures idempotent pairing creation ‚Äî calling `createPairings` multiple times for the same user is safe and also updates location data if the user has moved or redistricting has occurred.

## Acceptance Criteria

| Criterion | Test |
|-----------|------|
| `createPairings` finds house member and senators for user | User in VT district 1 ‚Üí 1 house member + 2 senators paired |

| Senators matched for all districts in state | Senator (district=NULL) ‚Üí paired with users in any district |
| Only current-term members paired | Member with `end_year = 2020` ‚Üí not paired (term ended) |
| Members with `end_year IS NULL` are paired | Active member without term end ‚Üí paired |
| `createPairings` is idempotent | Calling twice ‚Üí same pairings, `validated_at` updated |
| Upsert updates state/district/chamber on conflict | User moves from VT to CA ‚Üí `createPairings` updates state/district/chamber on existing pairing |
| `validateAll` adds newly elected legislators | New member added to `member_terms` ‚Üí pairing created |
| `validateAll` removes stale pairings | Member term ended ‚Üí pairing row deleted via NOT IN subquery |
| Stale pairings removed when member no longer serves | Member `end_year` set to past year ‚Üí pairing deleted on next validation |
| `validateAll` updates `validated_at` on valid pairings | Existing valid pairing ‚Üí `validated_at` refreshed |
| `ValidationSummary` counts are accurate | 5 added, 2 removed, 10 updated ‚Üí summary fields match exactly |
| Per-user transaction isolation in `validateAll` | One user fails validation ‚Üí other users still validated, failure logged |
| `validateForUser` re-computes pairings for single user | User moves from VT to CA ‚Üí old pairings removed, CA legislators paired |
| `validateForMember` removes invalid pairings | Member no longer serves ‚Üí affected pairings removed |
| `validateForMember` adds missing pairings | New users in member's district not yet paired ‚Üí pairings created |
| `findPairingsForUser` returns current pairings | User with 3 pairings ‚Üí returns 3 `UserLegislatorPairingDO` |
| No pairings for user with no state/district | User without state ‚Üí empty pairing list |
| `ValidationSummary` field names match type definition | Summary uses `usersChecked`, `pairingsAdded`, `pairingsRemoved`, `pairingsUpdated` |
| Representative from other district not paired | User in district 1 ‚Üí district 2 representative not paired |
