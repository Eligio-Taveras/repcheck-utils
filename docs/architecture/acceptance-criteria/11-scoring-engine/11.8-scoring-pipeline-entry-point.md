# Area 11.8 ÔÇö Scoring Pipeline Entry Point

> Project: `scoring-pipeline` | Package: `repcheck.scoring.pipeline.app`, `repcheck.scoring.pipeline.config`

## Purpose

Cloud Run Job entry point for the scoring pipeline. Operates in two modes: **scheduled** (batch-scores users with changed data) and **ad-hoc** (processes `scoring.user.requested` events from Pub/Sub). Both modes delegate to the `ScoringOrchestrator` (┬ž11.7).

---

## Classes

### `ScoringPipelineApp` ÔÇö IOApp Entry Point (11.8)
**Package:** `repcheck.scoring.pipeline.app`

Extends `IOApp`. Wires all dependencies and runs the pipeline in the configured mode.

```scala
object ScoringPipelineApp extends IOApp
```

#### `run(args: List[String]): IO[ExitCode]`

Pure wiring ÔÇö no domain logic. All pipeline logic extracted into `runPipeline` companion method.

1. Load `ScoringPipelineConfig` via PureConfig
2. Build `Resource` bundle:
   - AlloyDB transactor (Doobie/HikariCP)
   - DJL embedding model (`SemanticEmbeddingService.resource`)
   - LLM provider (Anthropic SDK client)
3. Use resource bundle Ôćĺ call `runPipeline(config, transactor, embeddingService, llmProvider, ...)`
4. Return `ExitCode`

#### `private[app] def runPipeline(config, transactor, embeddingService, llmProvider, ...): IO[ExitCode]`

Testable pipeline logic ÔÇö receives all dependencies as parameters.

1. **Wire components:**
   - `PairingValidator[IO](transactor)`
   - `UserStanceAggregator[IO](transactor, llmProvider, retryWrapper)`
   - `LegislatorProfileBuilder[IO](transactor, config.scoring.voteWeights)`
   - `AlignmentScoreCalculator`
   - `AlignmentEvidenceFetcher[IO](transactor)`
   - `ScoreExplainer[IO](promptAssembler, llmProvider, retryWrapper)`
   - `SemanticEmbeddingService` (passed in)
   - `ScorePersister[IO](transactor)`
   - `ScoringOrchestrator[IO](pairingValidator, stanceAggregator, profileBuilder, calculator, evidenceFetcher, explainer, embeddingService, scorePersister, config.orchestrator)`
   - `ScoreRefreshNotifier[IO](pubsubPublisher, config.pubsub)` (┬ž11.11)

2. **Determine mode** from `config.mode`:

   **Scheduled mode:**
   a. Query users needing re-scoring:
      ```sql
      SELECT DISTINCT u.user_id FROM ${Tables.Users} u
      JOIN ${Tables.UserLegislatorPairings} ulp ON u.user_id = ulp.user_id
      WHERE u.last_stance_change_at > COALESCE(
        (SELECT MAX(sh.computed_at) FROM ${Tables.ScoreHistory} sh
         WHERE sh.user_id = u.user_id), '1970-01-01'::timestamptz
      )
      ```
   b. For each user via `parEvalMap(config.userParallelism)`:
      - Generate `correlationId = UUID.randomUUID()`
      - `orchestrator.scoreUser(userId, correlationId, source = "scheduled")`
   c. Collect all `UserScoringResult`s Ôćĺ `PipelineRunSummary`

   **Ad-hoc mode:**
   a. Subscribe to `scoring.user.requested` Pub/Sub subscription
   b. For each event:
      - Deserialize `PipelineEvent[ScoringUserRequestedEvent]`
      - `orchestrator.scoreUser(event.payload.userId, event.payload.requestId, source = "ad-hoc")`
      - Determine notification status from result:
        ```scala
        result match {
          case Succeeded(_, _, _, membersScored, membersFailed, _) if membersFailed == 0 =>
            notifier.notifyCompleted(userId, requestId, membersScored, "completed")
          case Succeeded(_, _, _, membersScored, membersFailed, _) if membersScored > 0 =>
            notifier.notifyCompleted(userId, requestId, membersScored, "partial")
          case Succeeded(_, _, _, _, _, _) =>
            notifier.notifyCompleted(userId, requestId, 0, "failed")
          case Skipped(_, _, reason) =>
            notifier.notifyCompleted(userId, requestId, 0, "failed")
          case Failed(_, _, reason) =>
            notifier.notifyCompleted(userId, requestId, 0, "failed")
        }
        ```
        The `membersScored` value passed to `notifyCompleted` is the count of legislators *successfully scored*, not total pairings.
      - If `notifyCompleted` throws `NotificationFailed`, log the error but do not fail the pipeline run. The scores are already persisted in the database. Ack the Pub/Sub message regardless.
      - Ack Pub/Sub message
   c. Collect results Ôćĺ `PipelineRunSummary`

3. **Return:** `ExitCode.Success` if no fatal errors, `ExitCode.Error` otherwise

## Key Design Decisions

1. **Two modes in one app.** Scheduled and ad-hoc share all wiring ÔÇö only the user selection and notification differ. A `config.mode` flag selects which path to run. The Cloud Run Job is launched with different args/env for each mode.
2. **Scheduled mode queries for changed users.** `users.last_stance_change_at > max(score_history.computed_at)` identifies users whose paired legislators have had stance changes since their last scoring run. This is the skip-unchanged optimization from BEHAVIORAL_SPECS.md ┬ž3.6.
3. **Ad-hoc mode subscribes to Pub/Sub.** The `scoring.user.requested` event triggers on-demand scoring for a single user. The `ScoreRefreshNotifier` (┬ž11.11) publishes the completion event.
4. **No requeue logic.** The old design requeued `vote.recorded` messages when no analysis existed. The batch model eliminates this ÔÇö stance materialization handles data convergence via DB polling (┬ž11.9).
5. **Testability via companion extraction.** Same pattern as analysis pipeline (┬ž10.10) ÔÇö all logic in a testable method, `run` is wiring only.

## Acceptance Criteria

| Criterion | Test |
|-----------|------|
| Config loads all subsystem configs via PureConfig | Valid `application.conf` Ôćĺ `ScoringPipelineConfig` populated |
| `config.mode = "scheduled"` Ôćĺ scheduled mode runs | Mode flag Ôćĺ scheduled user query executed |
| `config.mode = "ad-hoc"` Ôćĺ ad-hoc mode runs | Mode flag Ôćĺ Pub/Sub subscription created |
| Scheduled mode queries users with changed stances | User with `last_stance_change_at` after last score Ôćĺ included |
| Scheduled mode skips users with no stance changes | User with `last_stance_change_at` before last score Ôćĺ skipped |
| Scheduled mode processes users in parallel | `userParallelism = 4` Ôćĺ `parEvalMap(4)` |
| Ad-hoc mode subscribes to `scoring.user.requested` | Pub/Sub subscription created for configured topic |
| Ad-hoc event triggers `orchestrator.scoreUser` | Event received Ôćĺ `scoreUser(userId, requestId, source = "ad-hoc")` called |
| Scheduled mode passes `source = "scheduled"` | Scheduled user Ôćĺ `scoreUser(userId, correlationId, source = "scheduled")` |
| Ad-hoc `Succeeded` with `failed == 0` Ôćĺ "completed" | All legislators scored Ôćĺ notifier called with status "completed", `membersScored` count |
| Ad-hoc `Succeeded` with `scored > 0` and `failed > 0` Ôćĺ "partial" | Some scored, some failed Ôćĺ notifier called with status "partial", `membersScored` count |
| Ad-hoc `Succeeded` with `scored == 0` Ôćĺ "failed" | No legislators scored Ôćĺ notifier called with status "failed", count 0 |
| Ad-hoc `Skipped` Ôćĺ "failed" status published | User skipped Ôćĺ notifier called with status "failed", count 0 |
| Ad-hoc `Failed` Ôćĺ "failed" status published | Orchestrator failed Ôćĺ notifier called with status "failed", count 0 |
| Notification failure does not fail pipeline run | `notifyCompleted` throws `NotificationFailed` Ôćĺ error logged, Pub/Sub message still acked |
| Pub/Sub message acked after processing | Both success and failure Ôćĺ message acknowledged |
| Pub/Sub message acked even on notification failure | `NotificationFailed` Ôćĺ message still acknowledged (scores already persisted) |
| `PipelineRunSummary` aggregated from results | 3 succeeded, 1 skipped Ôćĺ summary reflects counts |
| Correlation ID generated per user in scheduled mode | Each user gets unique UUID |
| Correlation ID from event in ad-hoc mode | `requestId` from event Ôćĺ used as correlationId |
| Pipeline logic in testable companion method | `runPipeline` callable with mock dependencies |
| `run` method is pure wiring with no domain logic | `run` only creates Resources and calls `runPipeline` |
| Exit code 0 when all succeed, 1 on any failure | All success Ôćĺ `ExitCode.Success`; any failure Ôćĺ `ExitCode.Error` |
| No requeue logic exists | No Pub/Sub re-publish on missing data |
