# Area 11.5 ‚Äî Score Persistence

> Project: `score-cache` | Package: `repcheck.scoring.cache.repository`

## Purpose

Owns all AlloyDB score table operations. Upserts current scores (overwriting on re-score) and appends history records (append-only). The `score-cache` SBT project is a publishable library ‚Äî it has no pipeline or Pub/Sub logic. The API server (Component 12) will depend on `score-cache` for reading scores.

class ScorePersister[F[_]: Async](

#### `upsertScore(score: ScoreDO, topics: List[ScoreTopicDO], congress: List[ScoreCongressDO], congressTopics: List[ScoreCongressTopicDO], status: String, nonOverlappingTopics: List[String]): F[Unit]`

All writes in a single Doobie transaction:

1. **Upsert main score row:**
   ```sql
   INSERT INTO ${Tables.Scores}
     (user_id, member_id, aggregate_score, last_updated, llm_model, total_bills, total_votes, status, non_overlapping_topics)
   VALUES (...)
   ON CONFLICT (user_id, member_id) DO UPDATE SET
     aggregate_score       = EXCLUDED.aggregate_score,
     last_updated          = EXCLUDED.last_updated,
     llm_model             = EXCLUDED.llm_model,
     total_bills           = EXCLUDED.total_bills,
     total_votes           = EXCLUDED.total_votes,
     status                = EXCLUDED.status,
     non_overlapping_topics = EXCLUDED.non_overlapping_topics
   ```

   - `status` is `"scored"` or `"no_overlap"` ‚Äî indicates whether the user's topics overlapped with the member's voting record
   - `nonOverlappingTopics` maps to the `non_overlapping_topics TEXT[]` column ‚Äî lists topics the user cares about but the member has no votes on
   - `reasoning` and `reasoning_embedding` are **not** set by `upsertScore` ‚Äî they are written separately by `updateScoreReasoning` after Phase 2 (LLM explanation) completes
2. **Replace topic scores:** `DELETE FROM ${Tables.ScoreTopics} WHERE user_id = $userId AND member_id = $memberId`, then batch INSERT all `topics` rows
3. **Replace congress scores:** `DELETE FROM ${Tables.ScoreCongress} WHERE user_id = $userId AND member_id = $memberId`, then batch INSERT all `congress` rows
4. **Replace congress-topic scores:** `DELETE FROM ${Tables.ScoreCongressTopics} WHERE user_id = $userId AND member_id = $memberId`, then batch INSERT all `congressTopics` rows

> **Why DELETE + INSERT instead of upsert for sub-tables?** The set of topics and congresses can shrink as well as grow (e.g., a bill's analysis is refined and fewer topics are returned). DELETE + INSERT within the same transaction ensures the stored set exactly reflects the current computation ‚Äî no stale rows from prior runs.

> **Main `scores` row uses ON CONFLICT upsert.** The primary key `(user_id, member_id)` is stable. Upsert is correct here because the row always exists after the first scoring run and we only ever need one current row per pair.

#### `updateScoreReasoning(userId: UUID, memberId: String, reasoning: String, reasoningEmbedding: Array[Float]): F[Unit]`

Updates the reasoning and embedding on an existing score row. Called by ¬ß11.7 Phase 2 after the LLM explanation completes. Separated from `upsertScore` because Phase 1 (numeric scoring) and Phase 2 (explanation generation) happen at different times.

```sql
UPDATE ${Tables.Scores}
SET reasoning = $reasoning, reasoning_embedding = $reasoningEmbedding
WHERE user_id = $userId AND member_id = $memberId
```

> **Why separate from upsertScore?** Phase 1 computes the numeric score and writes it immediately so the frontend can display results. Phase 2 generates the human-readable explanation via LLM (which is slower and may be batched). Keeping them as separate writes avoids blocking the numeric score on LLM latency.

#### `appendHistory(history: ScoreHistoryDO, congress: List[ScoreHistoryCongressDO], congressTopics: List[ScoreHistoryCongressTopicDO], highlights: List[ScoreHistoryHighlightDO], status: String): F[Unit]`

All writes in a single Doobie transaction:

1. INSERT `${Tables.ScoreHistory}` row ‚Äî the `score_id` is a pre-generated UUID passed in via the `ScoreHistoryDO`. The orchestrator generates it via `UUID.randomUUID()` before calling `appendHistory` so that the same `score_id` can be passed to `updateHistoryReasoning` later in Phase 2.
2. The INSERT includes the `status` column (`"scored"` or `"no_overlap"`)
3. Batch INSERT all `congress` rows (FK to `score_id`)
4. Batch INSERT all `congressTopics` rows (FK to `score_id` + `congress`)
5. Batch INSERT all `highlights` rows (FK to `score_id`)

> **Append-only.** History rows are never updated or deleted (except for reasoning/embedding which are back-filled by Phase 2). Each scoring run produces a new `ScoreHistoryDO` with a fresh `score_id`.

> **Separate transaction from `upsertScore`.** History is appended after the current score is committed. If the history write fails, the current score remains valid. Partial history is preferable to discarding a valid score.

#### `updateHistoryReasoning(scoreId: UUID, reasoning: String, reasoningEmbedding: Array[Float]): F[Unit]`

Updates the reasoning and embedding on an existing history row. Called by ¬ß11.7 Phase 2 after the LLM explanation completes, using the same `scoreId` that was passed to `appendHistory`.

```sql
UPDATE ${Tables.ScoreHistory}
SET reasoning = $reasoning, reasoning_embedding = $reasoningEmbedding
WHERE score_id = $scoreId
```

#### `appendHistoryHighlights(scoreId: UUID, highlights: List[ScoreHistoryHighlightDO]): F[Unit]`

Inserts highlight rows for an existing history entry. Called by ¬ß11.7 Phase 2 after the LLM explanation produces highlight candidates, using the same `scoreId` that was passed to `appendHistory`.

```sql
INSERT INTO ${Tables.ScoreHistoryHighlights} (score_id, bill_id, topic, stance, vote, alignment)
VALUES ($scoreId, $billId, $topic, $stance, $vote, $alignment)
```

> **Why separate from `appendHistory`?** In Phase 1, `appendHistory` is called with `highlights = Nil` because the LLM explanation hasn't run yet. Phase 2 generates highlights and appends them to the existing history row.

```sql

```

#### `findCongressScores(userId: UUID, memberId: String): F[List[ScoreCongressDO]]`

```sql
SELECT * FROM ${Tables.ScoreCongress} WHERE user_id = $userId AND member_id = $memberId ORDER BY congress DESC
```

#### `findHistory(userId: UUID, memberId: String, limit: Int): F[List[ScoreHistoryDO]]`

```sql
SELECT * FROM ${Tables.ScoreHistory}
WHERE user_id = $userId AND member_id = $memberId
ORDER BY computed_at DESC LIMIT $limit
```

#### `findHistoryHighlights(scoreId: UUID): F[List[ScoreHistoryHighlightDO]]`

```sql
SELECT * FROM ${Tables.ScoreHistoryHighlights} WHERE score_id = $scoreId
```

## Key Design Decisions

1. **Single transaction for all current score tables.** Scores, topics, congress, and congress-topic rows are written atomically. A partial update would leave the stored score inconsistent ‚Äî e.g., congress scores updated but topic scores stale.
2. **DELETE + INSERT for sub-tables.** Upsert cannot handle row removal. If a topic disappears from the latest scoring run (e.g., the bill's analysis no longer tags it), the old row must be removed. DELETE + INSERT within the same transaction achieves this cleanly.
3. **Separate transaction for history.** History appends are independent of the current score write. If history fails, the current score (which is what the frontend reads) is still valid.
4. **History is append-only with deferred reasoning.** History rows are inserted during Phase 1 without reasoning. Phase 2 back-fills `reasoning` and `reasoning_embedding` via `updateHistoryReasoning`. No other fields are ever updated. This is an audit trail ‚Äî every scoring run's output is preserved for trend chart data.
5. **score-cache is a publishable library.** The API server (Component 12) will consume `ScoreReader` for serving scores to the frontend. Score writes only happen from the scoring pipeline.

## Acceptance Criteria

| Criterion | Test |
|-----------|------|
| `upsertScore` inserts new row on first call | No prior score ‚Üí row created in `scores` |
| `upsertScore` updates `aggregate_score` on re-score | Prior score 0.6 ‚Üí re-score ‚Üí `aggregate_score = new value` |
| `upsertScore` persists `status` column | status `"scored"` ‚Üí read back ‚Üí `"scored"` |
| `upsertScore` persists `non_overlapping_topics` as TEXT[] | `List("healthcare", "defense")` ‚Üí read back ‚Üí same list |
| `upsertScore` updates `status` on re-score | Prior `"no_overlap"` ‚Üí re-score with `"scored"` ‚Üí status updated |
| `upsertScore` updates `non_overlapping_topics` on re-score | Prior `["healthcare"]` ‚Üí re-score with `[]` ‚Üí empty array |
| `upsertScore` does NOT set `reasoning` or `reasoning_embedding` | After `upsertScore` ‚Üí `reasoning` is NULL, `reasoning_embedding` is NULL |
| `upsertScore` replaces topic scores completely | Prior: [healthcare, defense]; re-score with only [healthcare] ‚Üí only healthcare row remains |
| `upsertScore` replaces congress scores completely | Prior: [118, 119]; re-score with only [119] ‚Üí only congress 119 row remains |
| `upsertScore` replaces congress-topic scores completely | Prior congress-topic rows replaced by current run's rows |
| `upsertScore` is transactional ‚Äî all tables updated together | Exception during congress insert ‚Üí main score AND topics also rolled back |
| `updateScoreReasoning` sets reasoning on existing score | After `upsertScore` ‚Üí `updateScoreReasoning` ‚Üí read back ‚Üí reasoning matches |
| `updateScoreReasoning` sets reasoning_embedding on existing score | After `upsertScore` ‚Üí `updateScoreReasoning` ‚Üí read back ‚Üí embedding matches |
| `updateScoreReasoning` is idempotent | Call twice with different reasoning ‚Üí second value wins |
| `appendHistory` inserts a new row on each call | 3 scoring runs ‚Üí 3 `score_history` rows |
| `appendHistory` persists `status` column | status `"no_overlap"` ‚Üí read back ‚Üí `"no_overlap"` |
| `appendHistory` uses caller-provided `score_id` | Pre-generated UUID passed in ‚Üí same UUID readable from `score_history` |
| `appendHistory` never modifies existing history rows (except reasoning) | Prior history rows unchanged after subsequent append |
| `appendHistory` congress sub-rows linked to `score_id` FK | FK enforced ‚Äî `score_id` must exist before congress rows can insert |
| `appendHistory` highlights linked to `score_id` FK | FK enforced ‚Äî highlights reference existing `score_history` row |
| `appendHistory` is transactional | Exception during highlights insert ‚Üí history row also rolled back |
| `updateHistoryReasoning` sets reasoning on existing history row | After `appendHistory` ‚Üí `updateHistoryReasoning` ‚Üí read back ‚Üí reasoning matches |
| `updateHistoryReasoning` sets reasoning_embedding on existing history row | After `appendHistory` ‚Üí `updateHistoryReasoning` ‚Üí read back ‚Üí embedding matches |
| `updateHistoryReasoning` uses same `scoreId` from `appendHistory` | Pre-generated UUID used for both calls ‚Üí reasoning attached to correct history row |
| `findScore` returns `None` when no score exists | Empty `scores` table ‚Üí `None` |
| `findScore` returns current `ScoreDO` after upsert | After `upsertScore` ‚Üí `Some(ScoreDO)` with correct `aggregate_score` |
| `findHistory` returns most recent first | 3 runs at different times ‚Üí ordered `computed_at DESC` |
| `findHistory` respects `limit` parameter | `limit = 2` with 5 history rows ‚Üí only 2 returned |
| `findHistoryHighlights` returns all highlights for a `scoreId` | 3 highlights for `scoreId` ‚Üí 3 rows returned |
| All table names use `Tables` constants | No hardcoded string literals in SQL |
| `ScoreHistoryDO.reasoning = Some("...")` round-trips correctly | Write with reasoning via `updateHistoryReasoning` ‚Üí read ‚Üí same value |
| `ScoreHistoryDO.reasoning = None` before Phase 2 | After `appendHistory` (no `updateHistoryReasoning` yet) ‚Üí read ‚Üí `None` |
| `ScoreDO` fields match all `scores` columns | Write ‚Üí read ‚Üí all fields identical |
| `ScoreCongressDO` fields match all `score_congress` columns | Write ‚Üí read ‚Üí all fields identical |
