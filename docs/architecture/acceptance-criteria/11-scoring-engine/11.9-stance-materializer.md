# Area 11.9 — Stance Materializer

> Project: `stance-materializer` | Package: `repcheck.scoring.materializer`

## Purpose

Scheduled Cloud Run Job that materializes per-topic member bill stances with LLM-generated reasoning and embeddings. Polls `stance_materialization_status` for bills where both votes and analysis are complete, then materializes `member_bill_stances` and `member_bill_stance_topics`. This is Process C from BEHAVIORAL_SPECS.md §3.2.

The stance materializer does NOT subscribe to Pub/Sub events. It polls the DB for readiness, eliminating dual-event coordination complexity.

---

## Classes

### `StanceReasoningOutput` — LLM Structured Output (11.9)
**Package:** `repcheck.scoring.materializer.model`

The LLM receives the full bill analysis context (topics, findings, concept summaries) and the member's vote, and produces per-topic reasoning in a single call. This avoids N separate LLM calls per topic.

```scala
case class StanceReasoningOutput(
  reasonings: List[TopicStanceReasoning]
)

case class TopicStanceReasoning(
  topic: String,
  stanceDirection: String,    // "Progressive", "Conservative", "Neutral"
  reasoning: String,          // LLM explanation of why this vote indicates this stance on this topic
  confidence: Option[Double]  // 0.0-1.0, LLM's confidence in this assessment
)
```

Note: A single bill+vote can have multiple topics. The LLM produces reasoning for each topic in one structured response, so one LLM call per (member, vote) yields all per-topic reasonings.

### `StanceMaterializerApp` — IOApp Entry Point (11.9)
**Package:** `repcheck.scoring.materializer.app`

Extends `IOApp`. Scheduled job — runs, processes all qualifying bills, exits.

```scala
object StanceMaterializerApp extends IOApp
```

#### `run(args: List[String]): IO[ExitCode]`

Pure wiring. Delegates to `runPipeline`.

#### `private[app] def runPipeline(config, transactor, embeddingService, llmProvider, ...): IO[ExitCode]`

1. Wire components: scanner, materializer, user-bill aligner (§11.10)
2. `scanner.findReadyBills()` → `List[String]` (bill IDs)
3. For each bill via `parEvalMap(config.materializationParallelism)`:
   - `materializer.materializeBill(billId)`
4. If `config.runUserBillAlignment` is `true`, invoke `UserBillAligner.alignAll()` as a second phase after all stance materializations complete. When `false`, user-bill alignment runs independently on its own schedule. Both modes are valid. Coupling them ensures alignment is always current after materialization; decoupling allows independent scaling.
5. Return `ExitCode`

#### `findReadyBills(): F[List[String]]`

Polls `stance_materialization_status` for bills where both votes and analysis are complete and stances need materialization:

```sql
SELECT bill_id FROM ${Tables.StanceMaterializationStatus}
WHERE has_votes = true
  AND all_passes_completed = true
  AND (stances_materialized_at IS NULL
       OR stances_materialized_at < GREATEST(votes_updated_at, analysis_completed_at))
```

Returns list of bill IDs ready for materialization.

### `StanceMaterializer[F[_]]` — Per-Bill Materialization (11.9)
**Package:** `repcheck.scoring.materializer.materializer`

```scala
class StanceMaterializer[F[_]: Async](
  xa: Transactor[F],
  embeddingService: SemanticEmbeddingService[F],
  llmProvider: LlmProvider[F],
  retryWrapper: LlmRetryWrapper[F],
  promptAssembler: UserPromptAssembler[F],
  config: StanceMaterializerConfig
)
```

#### `materializeBill(billId: String): F[MaterializationResult]`

1. **Fetch vote positions** for the bill:
   ```sql
   SELECT vp.member_id, v.vote_id, vp.position, v.vote_type, v.vote_date, v.congress, v.amendment_id
   FROM ${Tables.VotePositions} vp
   JOIN ${Tables.Votes} v ON vp.vote_id = v.vote_id
   WHERE v.bill_id = $billId
   ```

2. **Fetch analysis findings** — stance findings from the latest analysis. The `stanceDirection` comes from `bill_findings` where `finding_type.code = 'stance_detection'`. The finding's `summary` field contains a structured assessment including the stance direction. For the LLM call, pass the finding summary as context and let the LLM determine the per-topic stance direction (since a bill finding covers the whole bill, but individual topics within it may have different stance implications depending on the member's vote position):
   ```sql
   SELECT bf.finding_id, bf.summary, bf.concept_group_id, ft.code, ba.topics
   FROM ${Tables.BillFindings} bf
   JOIN ${Tables.BillAnalyses} ba ON bf.analysis_id = ba.analysis_id
   JOIN ${Tables.FindingTypes} ft ON bf.finding_type_id = ft.finding_type_id
   WHERE ft.code = 'stance_detection'
     AND ba.bill_id = $billId
     AND ba.analysis_id = (
       SELECT analysis_id FROM ${Tables.BillAnalyses}
       WHERE bill_id = $billId AND status = 'completed'
       ORDER BY completed_at DESC LIMIT 1
     )
   ```

3. **Fetch amendment findings** (if vote has `amendment_id`). When `amendment_id` is set on `member_bill_stances`, the materializer queries `amendment_findings` instead of `bill_findings`. If multiple stance findings exist, take the most recent:
   ```sql
   SELECT af.finding_id, af.summary, ft.code
   FROM ${Tables.AmendmentFindings} af
   JOIN ${Tables.FindingTypes} ft ON af.finding_type_id = ft.finding_type_id
   WHERE af.amendment_id = $amendmentId AND ft.code = 'stance_detection'
   ORDER BY af.created_at DESC LIMIT 1
   ```

4. **For each (member, vote):**
   a. Determine stance source: amendment findings if `amendment_id` is set, otherwise bill findings
   b. Fetch concept summary for context (from `bill_concept_summaries` for the concept group)
   c. **Generate per-topic reasoning** (single LLM call per member+vote, produces all topic reasonings):
      ```scala
      val reasoningOutput: Option[StanceReasoningOutput] = retryWrapper.withRetry {
        llmProvider.completeStructured[StanceReasoningOutput](
          LlmRequest(prompt = promptAssembler.assembleStanceReasoning(
            memberPosition = position,
            findingSummary = finding.summary,
            topics = topics,
            conceptSummary = conceptSummary
          ))
        )
      }.recover { case _ => None }
      ```
      The LLM receives the full bill analysis context (topics, finding summary, concept summaries) and the member's vote position. It returns a `StanceReasoningOutput` containing per-topic `TopicStanceReasoning` entries, each with `stanceDirection`, `reasoning`, and `confidence`.
   d. **For each `TopicStanceReasoning` in the output:**
      i. **Generate reasoning embedding** (non-fatal, in-process DJL/ONNX):
         ```scala
         val reasoningEmbedding: Option[Array[Float]] =
           embeddingService.embed(topicReasoning.reasoning).map(Some(_)).recover { case _ => None }
         ```
      ii. Build `MemberBillStanceTopicDO(id = UUID.randomUUID(), memberId, billId, voteId, topic = topicReasoning.topic, stanceDirection = topicReasoning.stanceDirection, reasoning = Some(topicReasoning.reasoning), reasoningEmbedding, confidence = topicReasoning.confidence, findingId = Some(finding.findingId), conceptSummary, createdAt = now)`

5. **Upsert parent `member_bill_stances`:**
   ```sql
   INSERT INTO ${Tables.MemberBillStances}
     (member_id, bill_id, vote_id, amendment_id, position, vote_type, vote_date, congress, topics)
   VALUES (...)
   ON CONFLICT (member_id, bill_id, vote_id)
   DO UPDATE SET position = EXCLUDED.position, vote_type = EXCLUDED.vote_type,
                 topics = EXCLUDED.topics, amendment_id = EXCLUDED.amendment_id
   ```

6. **Replace children `member_bill_stance_topics`:**
   The DELETE of existing `member_bill_stance_topics` rows and the batch INSERT of new rows happen in a single Doobie `ConnectionIO` transaction. If the insert fails, the delete is rolled back. This prevents data loss from partial writes.
   ```sql
   DELETE FROM ${Tables.MemberBillStanceTopics}
   WHERE member_id = $memberId AND bill_id = $billId AND vote_id = $voteId
   ```
   Then batch insert all new `MemberBillStanceTopicDO` rows within the same transaction.

7. **Update `stances_materialized_at`:**
   ```sql
   UPDATE ${Tables.StanceMaterializationStatus}
   SET stances_materialized_at = NOW()
   WHERE bill_id = $billId
   ```

8. **Update `users.last_stance_change_at`** for all users paired with affected members:
   ```sql
   UPDATE ${Tables.Users} u
   SET last_stance_change_at = NOW()
   FROM ${Tables.UserLegislatorPairings} ulp
   WHERE ulp.user_id = u.user_id
     AND ulp.member_id IN ($affectedMemberIds)
   ```

9. Return `MaterializationResult(billId, membersProcessed, topicRowsCreated)`

## Key Design Decisions

1. **DB polling, not event-driven.** The scanner queries `stance_materialization_status` for bills where both `has_votes = true` AND `all_passes_completed = true`. This eliminates the race condition where events arrive in unpredictable order.
2. **Per-topic reasoning via LLM.** Each `(member, vote, topic)` combination gets a brief LLM-generated reasoning explaining why the member's vote indicates a particular stance on that topic. This reasoning is stored and reused by the score explainer (§11.4).
3. **Per-topic reasoning embedding.** The reasoning text is embedded via DJL/ONNX for semantic search ("find legislators whose healthcare stance reasoning mentions Medicare").
4. **Finding FK as proof.** Each `MemberBillStanceTopicDO` links back to the `bill_findings` or `amendment_findings` row that determined the stance direction. This provides provenance.
5. **Delete + insert for children.** `member_bill_stance_topics` are replaced entirely per (member, bill, vote) — simpler than upsert logic for child rows with generated UUIDs.
6. **`last_stance_change_at` update cascades to users.** When stances are materialized for a bill, all users paired with the affected members have their `last_stance_change_at` updated. The scoring pipeline (§11.8) uses this timestamp to determine which users need re-scoring.
7. **Amendment stance lookup.** When a vote is on an amendment (`amendment_id` is set), the materializer looks up amendment-specific findings rather than parent bill findings for stance determination.

## Acceptance Criteria

| Criterion | Test |
|-----------|------|
| Scanner finds bills with both votes and analysis | `has_votes = true, all_passes_completed = true` → bill included |
| Scanner excludes bills without votes | `has_votes = false` → bill excluded |
| Scanner excludes bills without completed analysis | `all_passes_completed = false` → bill excluded |
| Scanner finds bills needing re-materialization | `stances_materialized_at < analysis_completed_at` → bill included |
| Scanner skips already-materialized bills | `stances_materialized_at > GREATEST(votes_updated_at, analysis_completed_at)` → excluded |
| Scanner returns empty list when nothing ready | No qualifying rows → empty list |
| `materializeBill` fetches vote positions for the bill | Bill with 3 members voting → 3 position rows fetched |
| `materializeBill` fetches stance findings from latest analysis | Two analyses → findings from newer one used |
| Stance findings use `finding_type.code = 'stance_detection'` | Query filters on `ft.code = 'stance_detection'`, not `'stance'` |
| Amendment vote → amendment findings used | `amendment_id` set → `amendment_findings` queried instead of `bill_findings` |
| Amendment findings take most recent when multiple exist | Two amendment stance findings → most recent by `created_at` used |
| Non-amendment vote → bill findings used | `amendment_id` NULL → `bill_findings` queried |
| Stance direction determined by LLM per topic | Finding summary passed as context → LLM returns per-topic `stanceDirection` |
| `StanceReasoningOutput` has multiple reasonings per LLM call | Bill with 3 topics → single LLM call returns `StanceReasoningOutput` with 3 `TopicStanceReasoning` entries |
| One LLM call per (member, vote), not per topic | 3 topics × 1 member → 1 LLM call, not 3 |
| LLM reasoning failure is non-fatal | LLM throws → `reasoning = None`, materialization continues |
| Reasoning embedding generated when reasoning succeeds | `reasoning = Some("...")` → `embeddingService.embed` called |
| Reasoning embedding failure is non-fatal | `embeddingService.embed` throws → `reasoningEmbedding = None` |
| `findingId` linked to source finding | `MemberBillStanceTopicDO.findingId = Some(finding.findingId)` |
| Parent `member_bill_stances` upserted | New vote → inserted; re-run → updated via ON CONFLICT |
| Children `member_bill_stance_topics` replaced atomically | Delete + insert in single Doobie `ConnectionIO` transaction → insert failure rolls back delete |
| `stances_materialized_at` updated after materialization | `stance_materialization_status.stances_materialized_at = NOW()` |
| `users.last_stance_change_at` updated for paired users | 5 users paired with affected members → all 5 updated |
| Materialization is idempotent | Running twice with same data → same result |
| Config includes `scanInterval` | `scanInterval = 15.minutes` parsed from config |
| Config includes `runUserBillAlignment` flag | `runUserBillAlignment = true` → `UserBillAligner.alignAll()` invoked after materialization; `false` → skipped |
| Bills processed in parallel per config | `materializationParallelism = 2` → at most 2 concurrent bills |
| `MaterializationResult` accurately reflects work done | 10 members, 25 topic rows → result reflects counts |
| Pipeline logic in testable companion method | `runPipeline` callable with mock dependencies |
| `run` method is pure wiring with no domain logic | `run` only creates Resources and calls `runPipeline` |
