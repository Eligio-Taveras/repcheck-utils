# Area 11.2 ‚Äî Legislator Profile Construction

> Project: `scoring-pipeline` | Package: `repcheck.scoring.pipeline.profile`

## Purpose

Constructs a per-topic stance profile for a legislator from pre-materialized data in `member_bill_stance_topics`. No LLM calls, no finding lookups ‚Äî all stance data is pre-computed by the stance materializer (¬ß11.9) and stored with reasoning, embeddings, and finding provenance.

## Classes

### `LegislatorProfileBuilder[F[_]]` ‚Äî Pre-Materialized Profile Query (11.2)
**Package:** `repcheck.scoring.pipeline.profile`

Reads from `member_bill_stance_topics` (pre-populated by the stance materializer ¬ß11.9) to construct the legislator's per-topic stance profile. No LLM calls. No finding lookups ‚Äî all data is pre-materialized.

```scala
class LegislatorProfileBuilder[F[_]: Async](
  xa: Transactor[F],
  voteWeights: VoteWeightConfig
)
```

#### `buildProfile(memberId: String, relevantTopics: List[String]): F[LegislatorTopicProfile]`

Only computes stance for topics in `relevantTopics` (the user's topics from ¬ß11.1).

1. **Fetch pre-materialized per-topic stances:**
   ```sql
   SELECT mbst.*, mbs.vote_type, mbs.position
   FROM ${Tables.MemberBillStanceTopics} mbst
   JOIN ${Tables.MemberBillStances} mbs
     ON mbst.member_id = mbs.member_id AND mbst.bill_id = mbs.bill_id AND mbst.vote_id = mbs.vote_id
   WHERE mbst.member_id = $memberId
     AND mbst.topic IN ($relevantTopics)
   ```
2. **For each `MemberBillStanceTopicDO` row:**
   - `stanceDir = stanceDirection match { case "Progressive" => +1.0; case "Conservative" => -1.0; case _ => 0.0 }`
     Valid values for `stanceDirection` in `MemberBillStanceTopicDO` are `"Progressive"`, `"Conservative"`, and `"Neutral"`. These are the only three values produced by the stance materializer (¬ß11.9). Any other value is treated as 0.0 (same as Neutral) by the wildcard case.
   - `memberDir = position match { case "Yea" => +1.0; case "Nay" => -1.0; case _ => 0.0 }`
     Valid values for `position` in `member_bill_stances` are `"Yea"`, `"Nay"`, `"Present"`, and `"Not Voting"`. Only `"Yea"` (+1.0) and `"Nay"` (-1.0) produce non-zero contributions. `"Present"` and `"Not Voting"` map to 0.0, meaning these votes are skipped (zero contribution is excluded per the skip rule).
   - `voteWeight = voteWeights.forType(voteType)`
   - `contribution = stanceDir √ó memberDir √ó voteWeight`
   - If `contribution == 0.0`: skip (Neutral stance or abstention ‚Äî no signal). When `contribution == 0.0`, the vote is skipped entirely ‚Äî it does NOT increment `topicCounts` and is NOT added to `topicBills`. Only votes with non-zero contributions count toward the topic average and bill/vote counts.
   - Accumulate: `topicContribs(topic) += contribution`, `topicCounts(topic) += 1`
   - Track: `topicBills(topic) add billId` (for `billsConsidered` distinct count)
3. **Per topic:** `stanceScore = topicContribs(topic) / topicCounts(topic)` (average contribution across all contributing votes)
4. **Clamp** each `stanceScore` to [-1.0, 1.0]
5. Return `LegislatorTopicProfile(memberId, entries)` ‚Äî only topics with at least one non-zero contributing vote

> **Only user-relevant topics computed.** The `relevantTopics` parameter filters which topics to build. Computing all possible topics on every scoring run would be wasteful ‚Äî bill topics are diverse and most won't overlap with any given user's interests.

> **Zero-contribution votes excluded from averaging.** Neutral stances and Present/Not Voting positions are skipped entirely ‚Äî they provide no directional signal. `topicCounts` counts only votes that contributed (non-zero `contribution`), so the average is over meaningful votes only.

> **No finding lookups needed.** Unlike the previous event-driven design that had to join `bill_findings` to get stance direction, the stance materializer (¬ß11.9) has already resolved stance direction and stored it in `member_bill_stance_topics.stanceDirection`. Profile construction is now a simple aggregation query.

#### `buildCongressProfiles(memberId: String, relevantTopics: List[String]): F[Map[Int, LegislatorTopicProfile]]`

Builds per-congress profiles for the aggregate/per-congress score split.

1. Same query as `buildProfile` but includes `mbs.congress` in the SELECT and result. `congress` comes from the `member_bill_stances` table (column `congress`), joined via the parent FK. It is never NULL ‚Äî it's a NOT NULL column in the schema.
2. Partition rows by `congress` value
3. For each congress: run the same aggregation logic as `buildProfile`
4. Return `Map[congressNumber, LegislatorTopicProfile]`

## Key Design Decisions

1. **Reads from pre-materialized `member_bill_stance_topics`.** The stance materializer (¬ß11.9) has already resolved vote positions against bill findings and stored per-topic stance direction, reasoning, and embeddings. Profile construction is now a simple aggregation query ‚Äî no multi-table joins, no finding lookups.
2. **vote √ó stance direction √ó weight = contribution.** This formula captures three dimensions: what the bill does to a topic (stance direction from pre-materialized data), how the member voted (member direction), and how important that vote type is (weight). Multiplying gives the signed contribution to the member's stance profile.
3. **Average, not sum.** Dividing by contributing vote count normalizes across members with different numbers of votes. A senator with 20 healthcare votes and a representative with 3 are scored on the same scale.
4. **Skip zero-contribution votes from the average.** Neutral stances and abstentions provide no signal. Including them in the vote count denominator would dilute the average without adding information.
5. **Only user-relevant topics computed.** The `relevantTopics` parameter keeps computation bounded to what matters for the current scoring run.

| Conservative stance + Nay ‚Üí positive stanceScore contribution | `stanceDirection = "Conservative"`, `position = "Nay"` ‚Üí `contribution = +voteWeight` |
| Neutral stance ‚Üí zero contribution, excluded from average | `stanceDirection = "Neutral"` ‚Üí contribution skipped, `topicCounts` not incremented |
| "Not Voting" position ‚Üí zero contribution, excluded from average | `position = "Not Voting"` ‚Üí contribution skipped |
| Passage votes use weight 1.0 | `voteType = "Passage"` ‚Üí `voteWeight = 1.0` in contribution |
| Cloture votes use weight 0.8 | `voteType = "Cloture"` ‚Üí `voteWeight = 0.8` |
| Amendment votes use weight 0.5 | `voteType = "Amendment"` ‚Üí `voteWeight = 0.5` |

| stanceScore clamped to [-1.0, 1.0] | Rare extreme accumulation ‚Üí clamped at boundary |
| `billsConsidered` counts distinct bills | Same bill, two votes ‚Üí `billsConsidered = 1` |
| `votesAnalyzed` counts contributing vote positions | 3 non-zero contributions ‚Üí `votesAnalyzed = 3` |

| `VoteWeightConfig.forType` returns `other` for unknown type | Unrecognized voteType string ‚Üí `voteWeights.other` returned |
