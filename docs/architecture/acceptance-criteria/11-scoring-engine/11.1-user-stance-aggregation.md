# Area 11.1 — User Stance Aggregation

> Project: `scoring-pipeline` | Package: `repcheck.scoring.pipeline.aggregation`

## Purpose

Extracts per-topic political stances and importance weights from a user's Q&A responses using the pre-tagged question bank. Multiple-choice answers are processed algorithmically (no LLM). Custom fill-in text answers require a single batched LLM call via the `preference-interpretation` profile (Component 9 §9.2).

---

## Types

### `UserTopicProfile` — Aggregated User Stance (11.1)
**Package:** `repcheck.scoring.pipeline.aggregation`

Result of aggregating all Q&A responses for one user. Passed to `AlignmentScoreCalculator` (§11.3) and injected into the scoring prompt context (§11.4). This is the return type of `aggregate(userId: UUID): F[UserTopicProfile]`.

```scala
case class UserTopicProfile(
  userId: UUID,
  topics: List[UserTopicEntry]
)
```

- `topics` — sorted by `importance DESC`. Only topics with at least one Q&A response are included.

### `UserTopicEntry` — Per-Topic User Position (11.1)
**Package:** `repcheck.scoring.pipeline.aggregation`

```scala
case class UserTopicEntry(
  topic: String,
  stanceScore: Double,    // [-1.0, +1.0]
  importance: Double       // [0.0, 1.0], all importances sum to 1.0
)
```

- `stanceScore` — signed: positive = progressive, negative = conservative, 0.0 = neutral or balanced across questions
- `importance` — normalized so all topics in the profile sum to 1.0; sourced from explicit user-provided weights in `user_topic_priorities`, not from Q&A signal accumulation

## Classes

### `UserStanceAggregator[F[_]]` — Stance Extraction (11.1)
**Package:** `repcheck.scoring.pipeline.aggregation`

```scala
class UserStanceAggregator[F[_]: Async](
  xa: Transactor[F],
  promptAssembler: UserPromptAssembler[F]
)
```

#### `aggregate(userId: UUID): F[UserTopicProfile]`

1. **Fetch responses:** Call `fetchResponses(userId)` → `List[QaUserResponseDO]`
2. **Return empty** if the list is empty: `UserTopicProfile(userId, Nil)`
3. **Fetch question metadata:** Call `fetchQuestionMappings(questionIds)` and `fetchAnswerOptions(questionIds)` using the unique question IDs from responses
4. **Partition** responses into `multipleChoice` (`selectedOption.isDefined`) and `customText` (`customText.isDefined`)
   > The `chk_answer_type` constraint in migration 004 guarantees exactly one of `selectedOption` or `customText` is non-NULL per response. Responses violating this invariant cannot exist in the database.
5. **Process multiple-choice** (no LLM):
   - For each response and each `QaQuestionTopicDO` mapping for its question:
     - `agreeDir = if (mapping.agreeStance == "Progressive") +1.0 else -1.0`
     - `multiplier = answerOptions((response.questionId, response.selectedOption.get)).stanceMultiplier`
     - `stanceContrib = agreeDir × multiplier`
     - Accumulate into two mutable maps: `topicStanceNumer(topic) += stanceContrib × mapping.weight`, `topicStanceDenom(topic) += mapping.weight`
6. **Process custom fill-in** (LLM via Component 9):
   - If `customText` responses are non-empty: batch all into a single `promptAssembler.assemble("preference-interpretation", context)` call
   - Parse LLM response → stance-only adjustments `List[(topic, stanceAdj)]`
   - Merge each as a weight-1.0 contribution: `topicStanceNumer(topic) += stanceAdj`, `topicStanceDenom(topic) += 1.0`
   - If `customText` is empty: skip LLM call entirely
7. **Compute stanceScore** per topic: `stanceScore(topic) = topicStanceNumer(topic) / topicStanceDenom(topic)`
   > If `topicStanceDenom(topic) == 0` for any topic, that topic is excluded from results (no contributing responses).
8. **Fetch topic priorities:** Call `fetchTopicPriorities(userId, topicStanceNumer.keySet)` → `Map[String, Double]`
9. **Assign importance:**
   - For each topic in `topicStanceNumer`: `rawWeight = priorities.getOrElse(topic, 0.0)`
   - If all `rawWeight` values are 0.0 (user never completed the prioritization screen): fall back to equal weights (`rawWeight = 1.0` for each topic)
   - Normalize: `importance(topic) = rawWeight / rawWeights.values.sum`
10. **Normalize importance weights:** `importance(topic) = rawWeight(topic) / sum(allRawWeights)`. The resulting importance values sum to 1.0 and are used directly by §11.3 without further normalization.
11. Return `UserTopicProfile(userId, topics.sortBy(-_.importance))`

> **No LLM for pure multiple-choice flows.** When a user has answered only multiple-choice questions, `aggregate` makes zero LLM calls. This is the common case.

> **Importance is explicit, not inferred.** Topic weights come from `user_topic_priorities`, set by the user on the "Prioritize your topics" screen after Q&A. The aggregator does not infer importance from answer count, `importanceSignal` values, or any other Q&A signal.

> **LLM call is a single batch.** All custom fill-in responses for all questions are sent in one `promptAssembler.assemble` call, not one per response.

#### `fetchResponses(userId: UUID): F[List[QaUserResponseDO]]`

`private[aggregation]` — testable helper.

1. Query: `SELECT * FROM ${Tables.QaUserResponses} WHERE user_id = $userId`
2. Return `List[QaUserResponseDO]`

#### `fetchQuestionMappings(questionIds: List[String]): F[Map[String, List[QaQuestionTopicDO]]]`

`private[aggregation]` — testable helper.

1. Query: `SELECT * FROM ${Tables.QaQuestionTopics} WHERE question_id IN ($questionIds)`
2. Group by `questionId` → `Map[String, List[QaQuestionTopicDO]]`

#### `fetchAnswerOptions(questionIds: List[String]): F[Map[(String, String), QaAnswerOptionDO]]`

`private[aggregation]` — testable helper.

1. Query: `SELECT * FROM ${Tables.QaAnswerOptions} WHERE question_id IN ($questionIds)`
2. Key by `(questionId, optionValue)` → `Map[(String, String), QaAnswerOptionDO]`

> **`importanceSignal` is no longer used.** `QaAnswerOptionDO` may still carry this column but `UserStanceAggregator` does not read it — importance is sourced entirely from `user_topic_priorities`.

#### `fetchTopicPriorities(userId: UUID, topics: Set[String]): F[Map[String, Double]]`

`private[aggregation]` — testable helper.

1. Query:
   ```sql
   SELECT topic, priority_weight
   FROM ${Tables.UserTopicPriorities}
   WHERE user_id = $userId AND topic IN ($topics)
   ```
2. Return `Map[String, Double]` keyed by topic; absent topics are not included (caller applies `getOrElse(topic, 0.0)`)

## Key Design Decisions

1. **Intelligence is in the question design, not the aggregator.** The `agreeStance` + `stanceMultiplier` metadata encodes all political meaning. The aggregator performs arithmetic — no political judgement. This mirrors how professional political survey platforms (iSideWith, VoteSmart) work.
2. **Weighted average, not sum, for stanceScore.** Per-topic stance is `Σ(stanceContrib_i × weight_i) / Σ(weight_i)` — a weighted mean. Since `agreeDir ∈ {-1.0, +1.0}`, `stanceMultiplier ∈ [-1.0, +1.0]`, and `weight > 0`, each `stanceContrib_i ∈ [-1.0, +1.0]` and the weighted average of bounded values is bounded. The result is mathematically guaranteed to be within [-1.0, +1.0] — no clamping is needed. A question with `weight = 2.0` influences the topic score proportionally more than one with `weight = 0.5`, but cannot push the result out of range.
3. **LLM batching.** All custom fill-in responses are sent in a single prompt context, amortizing latency and token cost.
4. **No LLM for the common case.** Designing users toward multiple-choice responses minimizes LLM cost at scale. Custom fill-in is opt-in via `allowCustom = true` on the question.
5. **Explicit topic importance.** Topic weights come from the "Prioritize your topics" screen (`user_topic_priorities` table), not from answer frequency or `importanceSignal` values on questions. This makes weighting transparent and intentional — users directly control what matters to them rather than having importance inferred from engagement patterns. The `importanceSignal` column on `qa_answer_options` is no longer consumed by this aggregator.
6. **Equal-weight fallback.** If a user's `user_topic_priorities` rows are absent (e.g., they skipped the prioritization screen), all topics default to equal weight. This prevents silent exclusion of stances when priority data is missing.
7. **Contradiction detection deferred.** Detecting and resolving contradictory answers across questions (e.g., "expand Medicare" + "cut federal spending") is a future enhancement. Current implementation produces a weighted average — contradictions yield stanceScore near 0.0, which is the correct neutral behavior.

## Acceptance Criteria

| Criterion | Test |
|-----------|------|
| "Strongly agree" on Progressive question → stanceScore near +1.0 | `agreeStance = "Progressive"`, `stanceMultiplier = 1.0`, `weight = 1.0` → `topicStance += +1.0` |
| "Strongly disagree" on Progressive question → stanceScore near -1.0 | `agreeStance = "Progressive"`, `stanceMultiplier = -1.0`, `weight = 1.0` → `topicStance += -1.0` |
| "Agree" on Conservative question → negative stanceScore contribution | `agreeStance = "Conservative"`, `stanceMultiplier = 0.6` → `agreeDir = -1.0` → contribution = -0.6 |
| "Neutral" answer → zero stanceScore contribution | `stanceMultiplier = 0.0` → zero contribution to topic |
| `weight = 0.5` on topic mapping has proportionally less influence on stanceScore | Same answer applied twice — once with `weight = 0.5`, once with `weight = 1.0` — the higher-weight response pulls the weighted average more |
| stanceScore stays within [-1.0, +1.0] via weighted average (mathematically guaranteed) | Single response with `stanceMultiplier = 1.0`, `agreeDir = +1.0` → `stanceScore = 1.0`; multiple mixed responses → result remains within range |
| User with no responses returns empty `UserTopicProfile` | No rows in `qa_user_responses` for userId → `UserTopicProfile(userId, Nil)` |
| One question with two topic mappings produces two independent weighted averages | `qa_question_topics` has healthcare (weight 1.0) and spending (weight 0.6) → both topics get separate numer/denom accumulators |
| Importance sourced from `user_topic_priorities`, not from answer count or `importanceSignal` | User answers 5 healthcare questions, 1 economy question; `user_topic_priorities` weights economy higher → economy has higher importance |
| All topics' importance values sum to 1.0 | 3 topics with priority weights 4.0, 2.0, 1.0 → normalized 4/7, 2/7, 1/7 |
| Topics sorted by importance (from priorities) descending | Topic with highest priority_weight → `topics.head` |
| Topic absent from `user_topic_priorities` → weight 0.0 | Topic has stances but no priority row → excluded from scoring (importance = 0.0) |
| All priorities absent → equal weight fallback | No rows in `user_topic_priorities` for userId → all touched topics receive equal importance |
| Pure multiple-choice user: zero LLM calls | All `selectedOption` responses → `promptAssembler.assemble` never called |
| Custom fill-in user: LLM called exactly once with all custom responses batched | 3 custom responses → single `promptAssembler.assemble("preference-interpretation", ...)` call |
| Custom fill-in returns stance adjustments only (no importanceAdj) | LLM response parsed as `List[(topic, stanceAdj)]`; no importance value consumed from LLM output |
| Custom fill-in stance adjustments merged with MC accumulations | MC: `stanceContrib = +0.4`, `weight = 1.0` → `numer += 0.4`, `denom += 1.0`; LLM: `stanceAdj = +0.2`, `weight = 1.0` → `numer += 0.2`, `denom += 1.0`; final `stanceScore = 0.6 / 2.0 = 0.3` |
| `fetchResponses` returns only rows for the specified userId | userId = A inserted alongside userId = B → only A's responses returned |
| `fetchQuestionMappings` returns empty map for unknown question IDs | Non-existent question IDs → empty result |
| `fetchAnswerOptions` returns options keyed by (questionId, optionValue) | 3 questions × 5 options = 15 entries returned and keyed correctly |
| `fetchTopicPriorities` returns only rows for the specified userId and topic set | userId = A with topics {healthcare, economy}; userId = B rows present → only A's rows for those topics returned |
| `fetchTopicPriorities` returns empty map when no priority rows exist | No rows in `user_topic_priorities` for userId → empty map returned (caller applies equal-weight fallback) |
| Importance values sum to 1.0 after normalization | 3 topics with raw weights 4.0, 2.0, 1.0 → normalized importances 4/7 + 2/7 + 1/7 = 1.0 |
| Topic with `topicStanceDenom == 0` is excluded from results | Topic appears in mappings but has zero accumulated weight → topic absent from `UserTopicProfile.topics` |
| Return type of `aggregate` is `F[UserTopicProfile]` | `aggregate(userId)` returns `UserTopicProfile` with `userId` and `List[UserTopicEntry]` containing per-topic `stanceScore` and `importance` |
