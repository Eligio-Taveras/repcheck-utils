# Area 11.4 — Score Explainer

> Project: `scoring-pipeline` | Package: `repcheck.scoring.pipeline.scoring`

## Purpose

Generates human-readable alignment explanations and highlights using the LLM. Receives the pre-computed `AlignmentCalculationResult` and scoring context, assembles a prompt via Component 9's `UserPromptAssembler`, calls the LLM via Component 10's `LlmProvider`, and returns an `AlignmentScoreOutput`. Evidence is sourced from pre-materialized `member_bill_stance_topics` and `user_bill_alignments`. Explanation failure is non-fatal — the orchestrator (§11.7) falls back to persisting the algorithmic score with no reasoning.

---

## Types

### `ScoringContext` — LLM Input Bundle (11.4)
**Package:** `repcheck.scoring.pipeline.scoring`

All information needed to assemble the LLM scoring prompt. Built by the orchestrator (§11.7) and passed to `ScoreExplainer`.

```scala
case class ScoringContext(
  userId: UUID,
  memberId: String,
  userProfile: UserTopicProfile,        // from §11.1
  legislatorProfile: LegislatorTopicProfile,  // from §11.2
  calculationResult: AlignmentCalculationResult,  // from §11.3
  evidence: List[AlignmentEvidence],    // from AlignmentEvidenceFetcher
  userBillAlignments: List[UserBillAlignmentDO]  // from §11.10, pre-computed
)
```

> **No `triggerEvent` field.** In the batch model, profile selection is not driven by event type. A default profile is used, or score-delta-based selection (big change → richer profile). See design decision 2.

### `AlignmentEvidence` — Pre-Materialized Evidence (11.4)
**Package:** `repcheck.scoring.pipeline.scoring`

```scala
case class AlignmentEvidence(
  billId: String,
  topic: String,
  memberPosition: String,      // "Yea", "Nay", etc.
  stanceDirection: String,     // "Progressive", "Conservative", "Neutral"
  reasoning: Option[String],   // LLM reasoning from stance materializer
  conceptSummary: Option[String],
  userAlignmentScore: Option[Double],
  userReasoning: Option[String]
)
```

> Evidence is built from pre-materialized data: `member_bill_stance_topics` (legislator stance + reasoning) + `user_bill_alignments` (user alignment + reasoning). No multi-table join against raw `bill_findings` needed.

> **Evidence fetch failure handling:** If `fetchEvidence` raises an exception, the orchestrator (§11.7) catches it and calls `explain` with `evidence = Nil`. The explainer proceeds with empty evidence — the LLM still generates an explanation based on the numeric scores and profile data.

## Classes

### `AlignmentEvidenceFetcher[F[_]]` — Pre-Materialized Evidence Reader (11.4)
**Package:** `repcheck.scoring.pipeline.scoring`

Reads from `member_bill_stance_topics` and `user_bill_alignments` to build evidence for the LLM explanation prompt. Replaces the previous multi-table join approach.

```scala
class AlignmentEvidenceFetcher[F[_]: Async](
  xa: Transactor[F]
)
```

#### `fetchEvidence(memberId: String, userId: UUID, userProfile: UserTopicProfile, sharedTopics: Set[String], maxPerCategory: Int = 3): F[List[AlignmentEvidence]]`

1. **Fetch evidence in a single query** joining stance topics with positions and user alignments:
   ```sql
   SELECT mbst.*, mbs.position, uba.alignment_score AS user_alignment_score, uba.reasoning AS user_reasoning
   FROM ${Tables.MemberBillStanceTopics} mbst
   JOIN ${Tables.MemberBillStances} mbs ON mbs.member_id = mbst.member_id AND mbs.bill_id = mbst.bill_id AND mbs.vote_id = mbst.vote_id
   LEFT JOIN ${Tables.UserBillAlignments} uba ON uba.user_id = $userId AND uba.bill_id = mbst.bill_id AND uba.topic = mbst.topic
   WHERE mbst.member_id = $memberId AND mbst.topic IN ($sharedTopics)
   ```
2. **Build `AlignmentEvidence`** from each result row:
   - `stanceDirection` from `mbst.stance_direction`
   - `reasoning` from `mbst.reasoning` (legislator stance reasoning)
   - `conceptSummary` from `mbst.concept_summary`
   - `memberPosition` from `mbs.position` (Yea, Nay, etc.)
   - `userAlignmentScore` from `uba.alignment_score` (nullable — LEFT JOIN)
   - `userReasoning` from `uba.reasoning` (nullable — LEFT JOIN)
3. **Select top evidence:** Take up to `maxPerCategory` most aligned + `maxPerCategory` least aligned (by `userAlignmentScore`) per topic
4. Return combined evidence list

### `ScoreExplainer[F[_]]` — LLM Explanation Layer (11.4)
**Package:** `repcheck.scoring.pipeline.scoring`

```scala
class ScoreExplainer[F[_]: Async](
  promptAssembler: UserPromptAssembler[F],
  llmProvider: LlmProvider[F],
  retryWrapper: LlmRetryWrapper[F]
)
```

#### `explain(context: ScoringContext): F[AlignmentScoreOutput]`

1. **Select profile** — default profile for all scoring runs, or score-delta-based:
   - Default: `"full-alignment"` (Sonnet — thorough, with detailed per-topic reasoning)
   - If score delta is small (< config threshold): `"quick-score"` (Haiku — aggregate score with minimal explanation)
   - Score delta = `abs(newScore - previousScore)` if a previous score exists, otherwise treated as large
2. **Select highlight candidates** via `selectHighlightCandidates(context)` → `List[AlignmentHighlight]` (up to 6)
3. **Assemble prompt** via `promptAssembler.assemble(profileName, contextBlock)` where `contextBlock` contains:
   - User's topic stances and importance weights (from `context.userProfile`)
   - Legislator's topic stances and vote counts (from `context.legislatorProfile`)
   - Pre-computed per-topic alignment scores (from `context.calculationResult.topics`)
   - Pre-materialized evidence with reasoning (from `context.evidence`)
   - Pre-computed user-bill alignments (from `context.userBillAlignments`)
   - Highlight candidates for the LLM to select from (from step 2)
   - Pre-computed `aggregateScore` (the algorithmic score — so the LLM understands the numeric context)
4. **Call LLM:**
   ```scala
   retryWrapper.withRetry {
     llmProvider.completeStructured[AlignmentScoreOutput](LlmRequest(prompt = assembledPrompt))
   }
   ```
5. **Handle LLM failure:** If `retryWrapper.withRetry` fails after all retries, catch the exception and raise `ExplanationFailed(userId, memberId, cause.getMessage)`. This ensures the orchestrator receives a typed error rather than an opaque retry-exhaustion exception:
   ```scala
   .adaptError { case cause =>
     ExplanationFailed(context.userId, context.memberId, cause.getMessage)
   }
   ```
6. Return `AlignmentScoreOutput`

> **LLM receives the pre-computed score.** The `calculationResult.aggregateScore` from §11.3 is injected into the prompt. The LLM's task is to write reasoning and select highlights — not to compute the score.

> **Evidence includes pre-materialized reasoning.** The LLM receives stance reasoning from the materializer and alignment reasoning from the user-bill aligner, enabling richer explanations without the LLM needing to re-derive stance positions from raw data.

> **Highlight candidates bounded.** The prompt contains at most 6 candidate bills (top 3 most aligned + top 3 least aligned). The LLM selects which to include as `AlignmentHighlight` entries in its output.

#### `private[scoring] def selectHighlightCandidates(context: ScoringContext): List[AlignmentHighlight]`

Testable helper. Selects the top 3 most aligned and top 3 least aligned topics from `context.calculationResult.topics`:

1. **Sort** `topics` by `abs(rawAlignment - 0.5)` descending — topics furthest from neutral are the most noteworthy
2. **Partition** the sorted list into strong alignment (`rawAlignment > 0.5`) and weak alignment (`rawAlignment < 0.5`). Take up to 3 from each partition (up to 6 total). Topics with `rawAlignment == 0.5` are excluded — they convey no alignment signal.
3. **Populate `billId`** for each selected topic: from `context.evidence`, find the bill with the highest absolute stance contribution for that topic (i.e., the evidence entry whose `stanceDirection` weight had the most impact on the topic's alignment). This is the most representative bill for that topic. Do NOT hardcode `billId` to empty string.
4. **Build** `AlignmentHighlight(billId = representativeBillId, topic, stance = userStanceLabel, vote = legVoteLabel, alignment = rawAlignment)` for each selected entry

## Key Design Decisions

1. **Algorithmic score is authoritative.** The LLM cannot override the `aggregateScore` from §11.3. It provides narrative explanation and selects highlights. This prevents hallucination from affecting stored scores.
2. **Profile selection by score delta, not event type.** In the batch model, there is no trigger event to drive profile selection. Instead, the default profile is `"full-alignment"` (Sonnet). If the score delta from the previous run is small, `"quick-score"` (Haiku) is used to save cost. This replaces the event-type mapping (`analysis.completed` → `full-alignment`, `vote.recorded` → `topic-breakdown`, `user.profile.updated` → `quick-score`).
3. **Explanation failure is non-fatal.** If the LLM call fails (rate limit, timeout, structured output parse failure), the orchestrator catches the error and persists the algorithmic score with `reasoning = None`. A missing explanation is never a reason to discard a valid alignment score.
4. **Evidence from pre-materialized tables.** `AlignmentEvidenceFetcher` reads from `member_bill_stance_topics` and `user_bill_alignments` rather than performing multi-table joins against `bill_findings`, `bill_concept_summaries`, etc. This is simpler, faster, and provides richer evidence (includes pre-generated reasoning).
5. **Single LLM call per pair.** Each `(userId, memberId)` pair produces exactly one LLM call in the explainer. There is no per-topic LLM call — the full profile is sent in one request.
6. **Bounded highlight candidates.** Sending all topics to the LLM could produce very large prompts for legislators who vote on many topics. Pre-selecting 6 candidates keeps the context focused and the prompt within bounds.
