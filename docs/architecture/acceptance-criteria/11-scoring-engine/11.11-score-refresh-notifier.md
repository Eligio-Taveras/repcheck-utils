# Area 11.11 — Score Refresh Notifier

> Project: `scoring-pipeline` | Package: `repcheck.scoring.pipeline.notification`

## Purpose

Publishes `scoring.user.completed` events to Pub/Sub when ad-hoc scoring runs complete. This is Process F from BEHAVIORAL_SPECS.md §3.5. The API server reads the return queue to notify the user that their scores are ready.

#### `notifyCompleted(userId: UUID, requestId: UUID, memberScoreCount: Int, status: String): F[Unit]`

Publishes a `ScoringUserCompletedEvent` to the return queue.

`memberScoreCount` is the number of legislators that were *successfully scored* (not total pairings). When `status` is `"failed"`, `memberScoreCount` is always 0.

1. Build `ScoringUserCompletedEvent(userId, requestId, memberScoreCount, status)`
2. Wrap in `PipelineEvent[ScoringUserCompletedEvent]` with `correlationId = requestId`
3. Serialize to JSON via Circe
4. Publish to `config.scoringCompletedTopic`
5. Log completion with requestId for traceability

> **`status` values and decision table:**
>
> | `membersScored` | `membersFailed` | Status        |
> |-----------------|-----------------|---------------|
> | all             | 0               | `"completed"` |
> | some > 0        | some > 0        | `"partial"`   |
> | 0               | all or skipped  | `"failed"`    |
>
> - `"completed"` — all legislators scored successfully
> - `"partial"` — some legislators scored, some failed
> - `"failed"` — no legislators scored (all failed or user skipped)
>
> **Skipped users DO trigger notification** in ad-hoc mode (with status `"failed"`) so the API server can inform the user why scoring did not happen.

### Error Types (11.11)

```scala
case class NotificationFailed(
  userId: UUID,
  requestId: UUID,
  reason: String
) extends Exception(s"Failed to publish scoring completion for user $userId (request $requestId): $reason")
```

## Key Design Decisions

1. **Return queue pattern.** The API server publishes `scoring.user.requested` to a request queue; the scoring pipeline publishes `scoring.user.completed` to a return queue. This decouples the request from the response without polling.
2. **`requestId` as correlation.** The `requestId` from the request event is echoed in the completion event, allowing the API server to match requests to responses.
3. **Three status values.** `"completed"`, `"partial"`, and `"failed"` give the API server enough information to decide what to show the user (e.g., "Your scores are ready" vs "Some scores are still processing").
4. **Notification failure is non-fatal to scoring.** If the Pub/Sub publish fails, the scores are still in the database. The API server can poll for score existence as a fallback.
5. **Only for ad-hoc mode.** Scheduled mode does not publish completion events — there is no caller waiting for a response.

## Acceptance Criteria

| Criterion | Test |
|-----------|------|
| `notifyCompleted` publishes `ScoringUserCompletedEvent` | Mock publisher → verify event published |
| Event has correct `userId` and `requestId` | Input values → echoed in published event |
| `memberScoreCount` reflects successfully scored legislators | 3 of 5 legislators scored → `memberScoreCount = 3` (not 5) |
| `memberScoreCount = 0` when status is "failed" | All failed or skipped → `memberScoreCount = 0` |
| `status = "completed"` when all legislators scored, none failed | `membersScored = 5, membersFailed = 0` → status "completed" |
| `status = "partial"` when some scored and some failed | `membersScored = 3, membersFailed = 2` → status "partial" |
| `status = "failed"` when no legislators scored | `membersScored = 0` → status "failed" |
| Skipped users trigger notification in ad-hoc mode | `Skipped` result → notifier called with status "failed", count 0 |
| Event published to configured topic | `config.scoringCompletedTopic` → publish target |
| Event wrapped in `PipelineEvent` envelope | Published event has `eventType`, `eventId`, `correlationId` |
| `correlationId` set to `requestId` | `requestId` from input → `PipelineEvent.correlationId` |
| Event serialized correctly via Circe | Event → JSON → deserialize → same event |
| Publish failure raises `NotificationFailed` | Mock publisher throws → `NotificationFailed` propagated |
| Notification is not called in scheduled mode | Scheduled scoring → `notifyCompleted` never called |
