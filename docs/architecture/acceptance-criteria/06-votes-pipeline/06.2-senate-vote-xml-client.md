# 6.2 Senate Vote XML Client

> Part of [Component 6 — Votes Pipeline](../06-VOTES-PIPELINE.md)

## Overview

Fetches and parses Senate roll call vote XML from `senate.gov`. Senate votes are not available through the Congress.gov JSON API — they come from the Senate's own XML feed. Each vote is a separate XML document at a predictable URL. This client also handles LIS-to-bioguide resolution using the `lis_member_mapping` table.

## Classes

### `SenateVoteXmlClient[F[_]]`

```
class SenateVoteXmlClient[F[_]: Async](
  httpClient: Client[F],
  retryWrapper: RetryWrapper[F],
  config: SenateVoteXmlConfig,
  logger: PipelineLogger[F]
) {
  def fetchVote(congress: Int, session: Int, voteNumber: Int): F[SenateVoteXmlDTO]

  def fetchVoteIndex(congress: Int, session: Int): F[List[SenateVoteIndexEntry]]
}
```

**`fetchVote` behavior:**

1. Builds URL: `https://www.senate.gov/legislative/LIS/roll_call_votes/vote{congress}{session}/vote_{congress}_{session}_{voteNumber:05d}.xml`
2. Fetches XML document via HTTP GET
3. Parses XML → `SenateVoteXmlDTO` (contains `congress`, `session`, `voteNumber`, `question`, `voteDate`, `result`, `members: List[SenateVoteMemberXmlDTO]`)
4. On HTTP error: classifies via error classifier, retries transient errors
5. On XML parse failure: wraps in `SenateVoteFetchFailed` with context

**`fetchVoteIndex` behavior:**

1. Builds URL: `https://www.senate.gov/legislative/LIS/roll_call_lists/vote_menu_{congress}_{session}.xml`
2. Fetches and parses the vote index (list of all votes in a session with vote numbers and dates)
3. Returns `List[SenateVoteIndexEntry]` for the pipeline to iterate over
4. The index provides `voteNumber`, `voteDate`, and `question` for each vote — used for change detection before fetching full XML

```
  baseUrl: String = "https://www.senate.gov",

```

> **Lower parallelism and higher delay** than House API client — senate.gov is less tolerant of high request rates.

### `LisResolver[F[_]]`

Resolves Senate LIS member IDs to bioguide IDs using the `lis_member_mapping` table.

> **No dependency on Component 5.** The votes-pipeline does NOT import `LisMappingRepository` from the members-pipeline. Instead, `LisResolver` uses its own read-only Doobie query against the `lis_member_mapping` table (which is shared infrastructure populated by Component 5). This avoids an SBT dependency between the two pipeline projects.

```
class LisResolver[F[_]: Sync](
  xa: Transactor[F],
  logger: PipelineLogger[F]
) {
  def resolve(senateVote: SenateVoteXmlDTO): F[LisResolutionResult]
  def findByLisIds(lisIds: List[String]): F[Map[String, String]]
}
```

**`LisResolutionResult`:**

```
final case class LisResolutionResult(
  resolvedMembers: List[VoteResultDTO],
  unresolvedLisIds: List[String]
)
```

**`findByLisIds` behavior:**

- `SELECT lis_member_id, member_id FROM {Tables.LisMemberMapping} WHERE lis_member_id IN (...)`
- Returns `Map[String, String]` (lisMemberId → bioguideId)
- Single batch query — critical for votes with 100 senators

**`resolve` behavior:**

1. Extract all `lisMemberId` values from `senateVote.members`
2. Call `findByLisIds` to batch-fetch mappings
3. For each `SenateVoteMemberXmlDTO`:
   - If `lisMemberId` has a mapping → create `VoteResultDTO` with `memberId = bioguideId`
   - If no mapping → add to `unresolvedLisIds` list, log at warn level
4. Return `LisResolutionResult` — caller decides how to handle unresolved IDs
5. The resolved `VoteResultDTO` list is used to construct a unified `VoteMembersDTO` (same shape as House)

### Error Types

```
final case class SenateVoteFetchFailed(
  congress: Int,
  session: Int,
  voteNumber: Int,
  detail: String,
  cause: Option[Throwable] = None
) extends Exception(
  s"Failed to fetch Senate vote $congress-$session-$voteNumber: $detail",
  cause.orNull
)

final case class LisResolutionFailed(
  voteId: String,
  unresolvedCount: Int,
  unresolvedIds: List[String],
  detail: String
) extends Exception(
  s"LIS resolution incomplete for vote $voteId: $unresolvedCount unresolved IDs: ${unresolvedIds.mkString(", ")}"
)
```

## Acceptance Criteria

| Criterion | Test |
|-----------|------|
| `fetchVote` parses valid Senate vote XML | WireMock returns vote XML → `SenateVoteXmlDTO` with correct fields |
| `fetchVote` URL constructed correctly with 5-digit zero-padded vote number | Vote 7 → URL ends with `vote_{congress}_{session}_00007.xml` |
| `fetchVoteIndex` parses vote menu XML | WireMock returns index XML → list of `SenateVoteIndexEntry` |
| `fetchVote` HTTP error → `SenateVoteFetchFailed` with context | Mock 404 → error includes congress/session/voteNumber |
| `fetchVote` XML parse error → `SenateVoteFetchFailed` | Mock malformed XML → error wraps parse exception |
| HTTP 429/503 classified as Transient and retried | Mock 429 → retried → succeeds |
| Request delay respected between votes | 1s delay, 3 votes → elapsed ≥ 2s |
| `LisResolver.resolve` maps all members when all mappings exist | 100 senators, all mapped → 100 `VoteResultDTO`, 0 unresolved |
| `LisResolver.resolve` handles partial mappings | 100 senators, 2 unmapped → 98 resolved, 2 in `unresolvedLisIds` |
| `LisResolver.findByLisIds` uses single batch query | Verify single `WHERE IN` query, not 100 individual queries |
| `LisResolver.resolve` logs warn for each unresolved LIS ID | 2 unresolved → 2 warn log entries |
| Unresolved positions do not block vote ingestion | 2 unresolved out of 100 → vote still persisted with 98 positions |
| `SenateVoteFetchFailed` includes congress/session/vote context | Verify error message fields |
| `LisResolutionFailed` lists all unresolved IDs | 3 unresolved → error message contains all 3 IDs |
