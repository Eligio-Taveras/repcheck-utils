# 6.4 Vote Change Detection

> Part of [Component 6 â€” Votes Pipeline](../06-VOTES-PIPELINE.md)

## Overview

Vote-specific change detection that determines whether a vote is new, unchanged, or updated â€” and critically, whether **positions changed** (which gates event emission). This lives in the votes-pipeline project, not in `ingestion-common`, because it requires vote-specific comparison logic.

The generic `ChangeDetector` (Component 3 Â§3.3) is unsuitable for votes because:
1. It compares individual fields via `Product` reflection â€” but vote positions are a separate entity collection, not fields on `VoteDO`
2. Event emission for votes depends specifically on **position changes**, not just any field change
3. The comparison logic needs access to stored positions from the repository

## Classes

### `VoteChangeDetector[F[_]]`

```
class VoteChangeDetector[F[_]: Sync](
  voteRepo: VoteRepository[F],
  positionRepo: VotePositionRepository[F],
  logger: PipelineLogger[F]
) {
  def detect(
    incoming: VoteConversionResult,
    correlationId: UUID
  ): F[VoteChangeReport]
}
```

**`detect` behavior:**

1. Fetch stored vote: `voteRepo.findByVoteId(incoming.vote.voteId)`
2. **If `None` (new vote)**:
   - Return `VoteChangeReport.New(incoming)`
3. **If `Some(stored)` â€” compare `updateDate`**:
   - If `incoming.vote.updateDate <= stored.updateDate` â†’ return `VoteChangeReport.Unchanged`
   - If `incoming.vote.updateDate > stored.updateDate` â†’ proceed to position comparison
4. **Position comparison**:
   - Fetch stored positions: `positionRepo.findByVoteId(voteId)`
   - Compare incoming positions against stored positions:
     - Match by `(voteId, memberId)` composite key
     - Detect added positions (member not in stored set)
     - Detect removed positions (member not in incoming set)
     - Detect changed positions (same member, different `position` value)
   - If any position differences found â†’ `VoteChangeReport.Updated(positionsChanged = true, diffs)`
   - If no position differences â†’ `VoteChangeReport.Updated(positionsChanged = false, diffs)` (metadata changed, positions same)

### `VoteChangeReport`

```
enum VoteChangeReport {
  case New(data: VoteConversionResult)
  case Updated(positionsChanged: Boolean, diffs: List[VotePositionDiff])
  case Unchanged
}
```

### `VotePositionDiff`

```
enum VotePositionDiff {
  case Added(memberId: String, position: String)
  case Removed(memberId: String, position: String)
  case Changed(memberId: String, oldPosition: String, newPosition: String)
}
```

> **Why track individual diffs?** The diffs are logged for observability (e.g., "Member B000944 changed from Nay to Yea"). They are not persisted â€” only the final position set is stored. The `positionsChanged` boolean is what gates event emission.

### Event Emission Decision Matrix

| Change Report | Archive? | Upsert? | Emit `vote.recorded`? |
|--------------|----------|---------|----------------------|
| `New` | No (nothing to archive) | Yes | Yes (`isUpdate = false`) |
| `Updated(positionsChanged = true)` | Yes | Yes | Yes (`isUpdate = true`) |
| `Updated(positionsChanged = false)` | Yes | Yes | No (metadata-only change) |
| `Unchanged` | No | No | No |

> **Rationale**: Scoring is position-driven. If a vote's metadata changes (e.g., corrected `result` field) but positions are identical, there is nothing for the scoring engine to re-process. Archiving still happens to preserve the audit trail of the metadata change.
>
> **Intentional divergence from bills pipeline**: The bills pipeline (Component 4) does not emit events at all from `bill-metadata-pipeline` â€” event emission is the responsibility of `bill-text-availability-checker`. The votes pipeline emits `vote.recorded` directly because there is no separate "text availability" step for votes. The position-gated emission logic is vote-specific and has no equivalent in the bills pipeline.

## Acceptance Criteria

| Criterion | Test |
|-----------|------|
| New vote (not in DB) â†’ `New` | `findByVoteId` returns `None` â†’ `VoteChangeReport.New` |
| Same `updateDate` â†’ `Unchanged` | Stored and incoming have same date â†’ `Unchanged` |
| Incoming `updateDate` older than stored â†’ `Unchanged` | Regression â†’ `Unchanged`, warning logged |
| Newer `updateDate` + position changes â†’ `Updated(positionsChanged = true)` | Member switched Yeaâ†’Nay â†’ `Updated` with `positionsChanged = true` |
| Newer `updateDate` + no position changes â†’ `Updated(positionsChanged = false)` | Metadata changed but positions same â†’ `positionsChanged = false` |
| Added position detected | New member not in stored â†’ `Added` diff |
| Removed position detected | Stored member not in incoming â†’ `Removed` diff |
| Changed position detected | Same member, Yeaâ†’Nay â†’ `Changed` diff |
| Position comparison is order-independent | Same positions in different order â†’ no diffs |
| Diffs logged at info level | Updated with 3 diffs â†’ 3 log entries with member IDs and positions |
| Correlation ID included in all log messages | Verify `correlationId` in logger context |
| `Updated(positionsChanged = true)` â†’ event emitted | Verify downstream event emission decision |
| `Updated(positionsChanged = false)` â†’ no event | Verify no event for metadata-only change |
