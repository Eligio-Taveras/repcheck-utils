# 6.5 Vote Processing Pipeline

> Part of [Component 6 — Votes Pipeline](../06-VOTES-PIPELINE.md)

## Overview

FS2 streaming pipeline that orchestrates the full vote ingestion flow for both chambers. Fetches votes from House (Congress.gov JSON) and Senate (senate.gov XML), detects changes, archives history, upserts to AlloyDB, creates placeholder members, and emits `vote.recorded` events.

## Classes

### `VoteProcessor[F[_]]`

```
class VoteProcessor[F[_]: Async](
  houseClient: HouseVotesApiClient[F],
  senateClient: SenateVoteXmlClient[F],
  lisResolver: LisResolver[F],
  voteRepo: VoteRepository[F],
  positionRepo: VotePositionRepository[F],
  historyArchiver: VoteHistoryArchiver[F],
  changeDetector: VoteChangeDetector[F],
  placeholderCreator: PlaceholderCreator[F],
  memberRepository: EntityRepository[F, MemberDO],
  billRepository: EntityRepository[F, BillDO],
  eventPublisher: IngestionEventPublisher[F],
  config: VotePipelineConfig,
  logger: PipelineLogger[F]
) {
  def streamAll(correlationId: UUID): Stream[F, ProcessingResult]
  def processHouseVotes(correlationId: UUID): Stream[F, ProcessingResult]
  def processSenateVotes(correlationId: UUID): Stream[F, ProcessingResult]
  def processVote(voteMembersDTO: VoteMembersDTO, correlationId: UUID): F[ProcessingResult]
}
```

### `VotePipelineConfig`

```
final case class VotePipelineConfig(
  congress: Int,
  session: Int,
  houseParallelism: Int = 4,
  senateParallelism: Int = 2,
  lookbackDays: Int = 7,
  house: HouseVotesConfig,       // §6.1
  senate: SenateVoteXmlConfig    // §6.2
)
```

> `house` nests the `HouseVotesConfig` from §6.1; `senate` nests the `SenateVoteXmlConfig` from §6.2. Top-level `houseParallelism`/`senateParallelism` override the nested parallelism values for the streaming pipeline (nested configs set client-level defaults).

### `streamAll` behavior

1. Run `processHouseVotes` and `processSenateVotes` concurrently via `Stream.merge`
2. Collect `ProcessingResult` per vote from both streams
3. Each chamber stream handles its own errors independently — a failure in one chamber does not halt the other

### `processHouseVotes` behavior

1. Compute lookback: `fromDateTime = Instant.now.minus(config.lookbackDays, ChronoUnit.DAYS)`
2. `houseClient.fetchAll(FetchParams(congress = Some(config.congress), fromDateTime = Some(fromDateTime)))` → `Stream[F, VoteListItemDTO]`
3. For each vote via `parEvalMap(config.houseParallelism)`:
   - `houseClient.fetchMembers(congress, session, rollCallNumber)` → `VoteMembersDTO`
   - `processVote(voteMembersDTO, correlationId)`

### `processSenateVotes` behavior

1. `senateClient.fetchVoteIndex(config.congress, config.session)` → `List[SenateVoteIndexEntry]`
2. Filter entries by lookback window: `voteDate >= config.lookbackDays ago`
3. For each entry via `parEvalMap(config.senateParallelism)`:
   - `senateClient.fetchVote(congress, session, voteNumber)` → `SenateVoteXmlDTO`
   - `lisResolver.resolve(senateVote)` → `LisResolutionResult(resolvedMembers, unresolvedLisIds)`
   - If `unresolvedLisIds` is non-empty: log warn with unresolved IDs, include count in `ProcessingResult`
   - Construct unified `VoteMembersDTO` from `resolvedMembers` + vote metadata from `SenateVoteXmlDTO`
   - `processVote(voteMembersDTO, correlationId)`

> **Senate conversion flow:** `SenateVoteXmlDTO` → `LisResolver.resolve` (maps LIS IDs to bioguide IDs, produces `VoteResultDTO` list) → assemble `VoteMembersDTO` with resolved members → `VoteMembersDTO.toDO` (same as House path). The `SenateVoteXmlDTO.toDO` method defined in Component 1 §1.2 accepts a `Map[String, String]` (lisMemberId → bioguideId) and produces a `VoteMembersDTO`. The `LisResolver` produces this map.

### `processVote` behavior

Shared processing for both chambers after convergence on `VoteMembersDTO`:

1. **Convert:** `voteMembersDTO.toDO` → `Either[String, VoteConversionResult]`
   - `Left(reason)` → log warning, return `ProcessingResult.Failed(voteId, reason)`
   - `Right(VoteConversionResult(voteDO, positions))` → continue

2. **Detect change:** `changeDetector.detect(conversionResult, correlationId)` → `VoteChangeReport`
   - `Unchanged` → return `ProcessingResult.Skipped(voteId, "unchanged")`
   - `New` or `Updated` → continue

3. **Placeholder members:** For each position's `memberId`:
   - `placeholderCreator.ensureExists[MemberDO](memberId, memberRepository)`

4. **Placeholder bill:** If `voteDO.billId` is `Some(billId)`:
   - `placeholderCreator.ensureExists[BillDO](billId, billRepository)`

5. **Archive + Upsert (single transaction):** Steps 5-7 execute in a **single Doobie transaction**. If any step fails, the entire transaction rolls back.
   - If `Updated` (not `New`): `historyArchiver.archiveVote(voteId)`
6. `voteRepo.upsert(voteDO)`
7. `positionRepo.replaceAll(voteId, positions)`

8. **Update stance materialization status (outside transaction, before event):**
   - If `New` or `Updated(positionsChanged = true)`, and `voteDO.billId` is `Some(billId)`:
     ```sql
     INSERT INTO stance_materialization_status (bill_id, has_votes, votes_updated_at)
     VALUES ($billId, true, NOW())
     ON CONFLICT (bill_id) DO UPDATE SET has_votes = true, votes_updated_at = NOW()
     ```
   - If `voteDO.billId` is `None` (procedural vote): skip status update
   - If `Updated(positionsChanged = false)`: skip status update (metadata-only change does not affect stances)

9. **Emit event (outside transaction):**
   - If `New`: emit `VoteRecordedEvent(voteId = voteDO.voteId, billId = voteDO.billId, chamber = voteDO.chamber, date = voteDO.voteDate, congress = voteDO.congress, isUpdate = false)`
   - If `Updated(positionsChanged = true)`: emit `VoteRecordedEvent(..., isUpdate = true)`
   - If `Updated(positionsChanged = false)`: do **not** emit (metadata-only change)
   - Event publishing uses `RetryWrapper`. If retries exhausted, log at error level (triggers alert for operator intervention). DB state is correct; only downstream notification missing.

10. Return `ProcessingResult.Succeeded(voteId)`

### `summarize(results: List[ProcessingResult]): PipelineRunSummary`

Aggregates individual results:

```
PipelineRunSummary(
  totalProcessed = results.size,
  succeeded = results.count(_.isSucceeded),
  skipped = results.count(_.isSkipped),
  failed = results.count(_.isFailed),
  eventsEmitted = results.count { case s: Succeeded => s.eventEmitted; case _ => false },
  errors = results.collect { case f: Failed => f.reason }
)
```

### Key Design Decisions

- **Single project, dual sources** — House and Senate vote processing share change detection, repositories, history archival, and event emission. Only the fetch + parse layer differs.
- **Stream.merge for concurrent chamber processing** — both chamber streams run simultaneously, improving throughput.
- **Single vote failure does not halt the stream** — errors are captured per-vote as `ProcessingResult.Failed`. The stream continues.
- **Archive before upsert** — history captured before data overwritten, preserving prior state.
- **Placeholders before upsert** — member and bill placeholders created before vote upsert so FK constraints are satisfied.
- **Position-gated event emission** — only position changes trigger `vote.recorded`. Metadata-only changes are persisted and archived but do not notify downstream (scoring is position-driven).
- **Unresolved LIS IDs do not block ingestion** — Senate votes with partial resolution are still persisted with available positions. Missing positions are logged for operator visibility.

## Acceptance Criteria

| Criterion | Test |
|-----------|------|
| `streamAll` processes both House and Senate votes concurrently | Mock both sources → results from both chambers collected |
| House failure does not halt Senate processing | House API error → Senate votes still processed |
| Senate failure does not halt House processing | Senate XML error → House votes still processed |
| House votes fetched with lookback window | `lookbackDays = 7` → `fromDateTime` is 7 days ago |
| Senate votes filtered by lookback window | Entries older than lookback → skipped |
| `processVote` converts DTO to DO correctly | Valid `VoteMembersDTO` → `VoteConversionResult` with vote + positions |
| New vote → upserted + event emitted | Vote not in DB → upsert + `vote.recorded` with `isUpdate = false` |
| Updated vote with position changes → archived + upserted + event emitted | Positions changed → archive + upsert + event with `isUpdate = true` |
| Updated vote with metadata-only changes → archived + upserted, no event | Positions same → archive + upsert, no event emitted |
| Unchanged vote → skipped | Same `updateDate` → `ProcessingResult.Skipped` |
| Placeholder members created for unknown voters | 3 positions, 1 unknown member → 1 placeholder created |
| Placeholder bill created for unknown bill | Vote references `billId` not in DB → placeholder created |
| No placeholder for null `billId` | Procedural vote (no bill) → no bill placeholder |
| History archived before upsert for updated votes | Updated vote → `archiveVote` called before `upsert` |
| History NOT archived for new votes | New vote → `archiveVote` not called |
| Positions replaced alongside vote | Vote with 100 positions → all 100 persisted |
| Archive + upsert + replace in single transaction | DB failure mid-transaction → all rolled back |
| Event published outside transaction | Transaction commits → then event published |
| Event publish failure retried via RetryWrapper | Publish fails → retried → succeeds on attempt 2 |
| Event publish exhausted → error logged, DB state intact | All retries fail → error log, vote still in DB |
| DTO-to-DO conversion failure → `ProcessingResult.Failed` | Invalid DTO → failure result with error detail |
| Single vote failure does not halt stream | First vote fails, second succeeds → 1 Failed, 1 Succeeded |
| `PipelineRunSummary` accurately counts results | 5 succeeded, 3 skipped, 2 failed, 3 events → summary matches |
| Senate vote with partial LIS resolution persisted | 98/100 resolved → vote persisted with 98 positions |
| Unresolved LIS IDs logged and included in `ProcessingResult` | 2 unresolved → warn logs + failure detail |
| House parallelism configurable | `houseParallelism = 4` → at most 4 concurrent House votes |
| Senate parallelism configurable | `senateParallelism = 2` → at most 2 concurrent Senate votes |
| Correlation ID flows through all operations | All log messages and results carry the correlation ID |
| New vote with `billId` → `stance_materialization_status` updated | New vote on bill → `has_votes = true, votes_updated_at = NOW()` upserted |
| Updated vote with position changes → `stance_materialization_status` updated | Positions changed → `votes_updated_at` refreshed |
| Updated vote with metadata-only changes → status NOT updated | `positionsChanged = false` → no `stance_materialization_status` write |
| Procedural vote (no billId) → status NOT updated | `billId = None` → no `stance_materialization_status` write |
| `stance_materialization_status` upsert uses `ON CONFLICT (bill_id) DO UPDATE` | Row already exists → updates `has_votes`, `votes_updated_at` |
| Status update happens before event publish | `stance_materialization_status` upserted → then `vote.recorded` published |
