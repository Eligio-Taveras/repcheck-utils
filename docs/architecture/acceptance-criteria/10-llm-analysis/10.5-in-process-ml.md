# Area 10.5 ŌĆö In-Process ML (DJL + Smile)

> Project: `bill-decomposition-pipeline` | Package: `repcheck.decomposition.pipeline.ml`

## Purpose

Local, in-process machine learning for section embedding and semantic clustering. No external API calls ŌĆö all computation runs on the Cloud Run Job's allocated CPU/memory.

## Classes

### `SectionEmbedder[F[_]]` ŌĆö DJL/ONNX Embedding (10.5)
**Package:** `repcheck.decomposition.pipeline.ml`

Generates 384-dim embeddings for clustering using DJL + ONNX Runtime.

```scala
class SectionEmbedder[F[_]: Async](
  predictor: Predictor[String[], float[][]]
)
```

**Factory:**
```scala
object SectionEmbedder {
  def resource[F[_]: Async]: Resource[F, SectionEmbedder[F]]
}
```

The `resource` factory loads the all-MiniLM-L6-v2 model at startup and releases it on shutdown.

#### `embedBatch(texts: List[String]): F[List[Array[Float]]]`

1. Validate `texts` is non-empty ŌĆö return empty list if so
2. Convert `texts` to `String[]` array
3. Execute `Async[F].blocking(predictor.predict(textArray))` ŌĆö CPU-bound, must not starve compute pool
4. Convert `float[][]` result to `List[Array[Float]]`
5. Validate output dimensionality: each vector must be 384-dim
6. On DJL failure ŌåÆ raise `EmbeddingGenerationFailed(error.getMessage, texts.size)`

#### `embedSingle(text: String): F[Array[Float]]`

1. Delegate to `embedBatch(List(text))`
2. Return single result

### `ConceptClusterer[F[_]]` ŌĆö Smile Clustering (10.5)
**Package:** `repcheck.decomposition.pipeline.ml`

Groups sections into concept clusters using semantic similarity.

```scala
class ConceptClusterer[F[_]: Async](
  config: ClusteringConfig
)
```

#### `cluster(embeddings: List[Array[Float]], sections: List[SectionParseResult]): F[ClusteringResult]`

1. Validate `embeddings.size == sections.size` ŌĆö raise `ClusteringFailed` if mismatch
2. If single section ŌåÆ return single-group `ClusteringResult` (no clustering needed)
3. Convert embeddings to `Array[Array[Double]]` (Smile uses Double)
4. Execute clustering in `Async[F].blocking`:
   - If `config.algorithm == "dbscan"`:
     a. Run `DBSCAN.fit(data, config.dbscanMinPoints, config.dbscanEpsilon)`
     b. Extract cluster labels (ŌłÆ1 = noise)
     c. Collect noise points (label ŌłÆ1)
   - If `config.algorithm == "kmeans"`:
     a. Determine k: `min(config.kmeansMaxK, sections.size / config.minClusterSize)`
     b. Run `KMeans.fit(data, k)`
     c. No noise points for k-means
5. Group sections by cluster label ŌåÆ create `ConceptGroupAssignment` for each cluster:
   - `groupId` ŌåÉ derived from cluster centroid hash (stable, deterministic)
   - `title` ŌåÉ heading of the section closest to centroid (or first section heading if none)
   - `sectionIndexes` ŌåÉ section indexes assigned to this cluster
6. Handle noise points: for each noise point, find nearest cluster centroid and assign to that group
7. Enforce `config.minClusterSize`: merge small clusters into nearest neighbor
8. Return `ClusteringResult(groups, noisePoints)`
9. On Smile failure ŌåÆ raise `ClusteringFailed(error.getMessage, sections.size)`

## Key Design Decisions

1. **Model loaded as Resource.** DJL model loading is expensive (~2-5s). Load once at startup, share across all bills in the run. `Resource` ensures cleanup.
2. **DBSCAN preferred over k-means.** Bills vary enormously in size. DBSCAN adapts to the data without requiring k. K-means is available as a fallback.
3. **Ephemeral embeddings.** The 384-dim clustering vectors are never persisted. Search embeddings (generated by `SemanticEmbeddingService`, area 10.8) are separate and are persisted.
4. **Blocking wrappers.** Both DJL and Smile are CPU-intensive Java libraries. All operations wrap in `Async[F].blocking` to avoid starving the CE compute pool.
5. **Noise handling.** DBSCAN may produce noise points (sections that don't fit any cluster). These are assigned to the nearest cluster rather than dropped.
