# Area 10.4 — Decomposition Orchestrator

> Project: `bill-decomposition-pipeline` | Package: `repcheck.decomposition.pipeline.orchestration`, `repcheck.decomposition.pipeline.persistence`

## Purpose

Orchestrates the full decomposition pipeline: Ollama section parsing → DJL search embedding → persistence → DJL clustering embedding → Smile clustering → concept group persistence → Haiku simplification → DJL concept embedding → final persistence.

---

## Classes

### `DecompositionOrchestrator[F[_]]` — Pipeline Orchestration (10.4)
**Package:** `repcheck.decomposition.pipeline.orchestration`

The central coordinator for bill decomposition. Called by the pipeline entry point for each bill.

```scala
class DecompositionOrchestrator[F[_]: Async](
  ollamaProvider: LlmProvider[F],
  sectionEmbedder: SectionEmbedder[F],
  conceptClusterer: ConceptClusterer[F],
  embeddingService: SemanticEmbeddingService[F],
  claudeProvider: LlmProvider[F],
  persister: DecompositionPersister[F],
  config: DecompositionConfig,
  logger: PipelineLogger[F]
)
```

#### `decompose(billId: String, versionId: UUID, correlationId: UUID): F[DecompositionResult]`

Primary entry point for decomposing a single bill.

1. **Reuse check:** `persister.existsForVersion(versionId)` → if `true`, return `DecompositionResult(billId, versionId, 0, 0, alreadyExisted = true)`
2. **Fetch bill text:** `persister.fetchBillText(versionId)` → `String` (from `bill_text_versions`)
3. **Parse sections:**
   - Build `LlmRequest(systemPrompt = sectionParsingPrompt, userPrompt = billText, modelTier = Local, ...)`
   - Call `ollamaProvider.completeStructured[List[SectionParseResult]](request)`
   - On failure (catch `LlmCallFailed` / `StructuredOutputParseFailed`) → log warning, wrap as `DecompositionFailed` context, treat entire text as single `SectionParseResult(0, None, None, billText)`
   - Enforce `config.maxSectionsPerBill` — if more sections returned, truncate with warning
4. **Generate search embeddings for sections:** `embeddingService.embedBatch(sections.map(_.content))` → `List[Array[Float]]`
5. **Persist sections + embeddings:** `persister.insertSections(versionId, sections, searchEmbeddings)` → inserts `bill_text_sections` rows
6. **Generate clustering embeddings:** `sectionEmbedder.embedBatch(sections.map(_.content))` → `List[Array[Float]]` (384-dim, ephemeral)
7. **Cluster sections:** `conceptClusterer.cluster(clusteringEmbeddings, sections)` → `ClusteringResult`
8. **Persist concept groups:** `persister.insertConceptGroups(versionId, clusteringResult)` → inserts `bill_concept_groups` + `bill_concept_group_sections`
9. **Simplify concept groups:** For each concept group via `parEvalMap(config.simplificationParallelism)`:
   - Concatenate section contents belonging to this group
   - Build `LlmRequest(systemPrompt = simplificationPrompt, userPrompt = concatenatedText, modelTier = config.simplificationModelTier, ...)`
   - Call `claudeProvider.completeStructured[SimplifiedConceptOutput](request)`
   - On per-group failure → log error, use first 500 chars of concatenated text as fallback
10. **Generate search embeddings for concept groups:** `embeddingService.embedBatch(simplifiedTexts)` → `List[Array[Float]]`
11. **Persist simplified text + embeddings:** `persister.updateConceptGroups(versionId, simplifiedTexts, conceptEmbeddings)`
12. **Return:** `DecompositionResult(billId, versionId, sections.size, conceptGroups.size, alreadyExisted = false)`

> **Transaction boundaries:** Steps 5 and 8 execute in a single transaction (section + group creation are sequential writes with no compute between them — step 6-7 happens BEFORE step 8, not between 5 and 8). Step 11 is a separate update transaction after simplification completes.

### `SimplifiedConceptOutput` — Haiku Simplification Output (10.4)
**Package:** `repcheck.decomposition.pipeline.orchestration`

The structured output type returned by Haiku when simplifying a concept group. Requires Circe semi-auto `Decoder`.

```scala
case class SimplifiedConceptOutput(
  simplifiedText: String,
  keyTopics: List[String]
)

object SimplifiedConceptOutput {
  implicit val decoder: Decoder[SimplifiedConceptOutput] = deriveDecoder
}
```

- `simplifiedText` — concise summary of all sections in the concept group, suitable as input to the analysis pipeline
- `keyTopics` — topic keywords extracted during simplification (stored on `bill_concept_groups` for searchability)

### `DecompositionPersister[F[_]]` — AlloyDB Persistence (10.4)
**Package:** `repcheck.decomposition.pipeline.persistence`

Handles all AlloyDB writes for decomposition artifacts.

```scala
class DecompositionPersister[F[_]: Async](
  xa: Transactor[F]
)
```

#### `existsForVersion(versionId: UUID): F[Boolean]`

1. Query `SELECT COUNT(*) FROM ${Tables.BillTextSections} WHERE version_id = $versionId`
2. Return `count > 0`

#### `fetchBillText(versionId: UUID): F[String]`

1. Query `SELECT text_content FROM ${Tables.BillTextVersions} WHERE version_id = $versionId`
2. Return text content
3. If not found → raise `DecompositionFailed(versionId.toString, "fetch", "bill text version not found")`

#### `insertSections(versionId: UUID, sections: List[SectionParseResult], embeddings: List[Array[Float]]): F[Unit]`

1. Zip `sections` with `embeddings`
2. Batch insert into `${Tables.BillTextSections}`:
   - `version_id`, `section_index`, `section_identifier`, `heading`, `content`, `embedding`
3. Use Doobie `Update[SectionRow].updateMany(rows).transact(xa)`

#### `insertConceptGroups(versionId: UUID, clusteringResult: ClusteringResult): F[List[UUID]]`

1. For each group in `clusteringResult.groups`:
   - Generate `UUID` for `concept_group_id`
   - Insert into `${Tables.BillConceptGroups}`: `concept_group_id`, `version_id`, `group_id`, `title`
2. For each group, insert junction rows into `${Tables.BillConceptGroupSections}`:
   - `concept_group_id`, `section_index`
3. Handle noise points: assign to nearest cluster's concept group
4. Return list of generated concept group UUIDs
5. All inserts in a single transaction

#### `updateConceptGroups(versionId: UUID, simplifiedTexts: List[(UUID, String)], embeddings: List[(UUID, Array[Float])]): F[Unit]`

1. For each `(conceptGroupId, simplifiedText)`:
   - `UPDATE ${Tables.BillConceptGroups} SET simplified_text = $text WHERE concept_group_id = $id`
2. For each `(conceptGroupId, embedding)`:
   - `UPDATE ${Tables.BillConceptGroups} SET embedding = $embedding WHERE concept_group_id = $id`
3. All updates in a single transaction

## Key Design Decisions

1. **Reuse check is first.** If `bill_text_sections` already has rows for this `version_id`, skip the entire pipeline. Decomposition is tied to text version, not analysis run.
2. **Graceful degradation on Ollama failure.** If section parsing fails, treat the whole bill as one section. Clustering will produce one concept group. Analysis still proceeds. The `LlmCallFailed` from the provider is caught and logged, not propagated.
3. **Corrected transaction boundary.** Steps 5 (insert sections) and 8 (insert groups + junctions) are a single transaction — steps 6-7 (clustering) happen BEFORE step 8, so there is no compute during the open transaction. Step 11 (update simplified text + embeddings) is a separate transaction after Haiku simplification.
4. **Simplification is the expensive step.** Everything else is local/free. Parallelism config controls how many Haiku calls run concurrently to balance cost vs. throughput.
5. **Two LlmProvider instances.** `ollamaProvider` for section parsing (local, free), `claudeProvider` for Haiku simplification (API cost). Both implement the same `LlmProvider[F]` trait.
6. **Per-group simplification failure is not fatal.** If a single concept group fails Haiku simplification, a truncated fallback is used. The bill still gets decomposed.

## Acceptance Criteria

| Criterion | Test |
|-----------|------|
| Existing decomposition is reused (skip all steps) | `existsForVersion` returns true → `DecompositionResult.alreadyExisted = true`, no LLM calls |
| Bill text fetched from AlloyDB by version ID | Mock persister → verify `fetchBillText(versionId)` called |
| Ollama parses bill text into sections | WireMock Ollama → returns 5 sections → 5 `SectionParseResult` items |
| Ollama failure degrades to single section | WireMock Ollama → 500 error → entire text becomes 1 section, `LlmCallFailed` caught |
| Max sections per bill enforced | Ollama returns 200 sections, `maxSectionsPerBill = 100` → truncated to 100 |
| Search embeddings generated for each section | 5 sections → `embeddingService.embedBatch` called with 5 texts |
| Sections + embeddings persisted to `bill_text_sections` | 5 sections → 5 rows inserted with embeddings |
| Clustering embeddings generated (384-dim, ephemeral) | 5 sections → `sectionEmbedder.embedBatch` called, results not persisted |
| Sections clustered into concept groups | 5 sections → `conceptClusterer.cluster` → `ClusteringResult` with N groups |
| Concept groups + junction rows persisted | 3 groups from 5 sections → 3 `bill_concept_groups` rows + 5 junction rows |
| Noise points assigned to nearest cluster | DBSCAN produces 1 noise point → assigned to a group |
| Simplification parallelized per config | `simplificationParallelism = 3` → up to 3 concurrent Haiku calls |
| Simplification uses Haiku tier | Verify `LlmRequest.modelTier = Haiku` for simplification calls |
| `SimplifiedConceptOutput` decoded from Haiku response | Valid JSON → `simplifiedText` and `keyTopics` populated |
| Per-group simplification failure uses fallback | 1 of 3 groups fails → fallback text (first 500 chars), others succeed |
| Search embeddings generated for concept groups | 3 groups → `embeddingService.embedBatch` called with 3 simplified texts |
| Simplified text + embeddings persisted | 3 groups → 3 `bill_concept_groups` rows updated |
| `DecompositionResult` has correct counts | 5 sections, 3 groups → `sectionCount = 5`, `conceptGroupCount = 3` |
| Full pipeline runs sequentially within a bill | Steps 1→12 execute in order for a single bill |
| `DecompositionFailed` raised when bill text not found | Version ID not in DB → `DecompositionFailed` with step "fetch" |
| All persistence uses `Tables` constants | Verify no hardcoded table name strings in SQL |
| Sections + groups persisted in single transaction | Steps 5 + 8 → one `transact(xa)` call |
| Simplification update in separate transaction | Step 11 → separate `transact(xa)` call |
