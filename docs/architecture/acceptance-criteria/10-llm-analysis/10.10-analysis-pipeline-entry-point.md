# Area 10.10 — Analysis Pipeline Entry Point

> Project: `bill-analysis-pipeline` | Package: `repcheck.analysis.pipeline.app`, `repcheck.analysis.pipeline.config`

## Purpose

Cloud Run Job entry point for the analysis pipeline. Subscribes to `bill.decomposition.completed` events, orchestrates multi-pass LLM analysis via the `AnalysisOrchestrator`, and emits `analysis.completed` events.

---

## Classes

### `BillAnalysisPipelineApp` — IOApp Entry Point (10.10)
**Package:** `repcheck.analysis.pipeline.app`

Extends `IOApp`. Wires all dependencies and runs the pipeline.

```scala
object BillAnalysisPipelineApp extends IOApp
```

#### `run(args: List[String]): IO[ExitCode]`

Pure wiring — no domain logic. All pipeline logic extracted into `runPipeline` companion method.

1. Load `AnalysisPipelineConfig` via PureConfig
2. Build `Resource` bundle:
   - AlloyDB transactor (Doobie/HikariCP)
   - DJL search embedding model (`SemanticEmbeddingService.resource`)
   - Anthropic SDK client (from API key in environment)
3. Use resource bundle → call `runPipeline(config, transactor, embeddingService, ...)`
4. Return `ExitCode`

#### `private[app] def runPipeline(config, transactor, embeddingService, anthropicClient, ...): IO[ExitCode]`

Testable pipeline logic — receives all dependencies as parameters.

1. **Wire providers:**
   - `StructuredOutputEnforcer[IO](config.llm.structuredOutputMaxRetries)`
   - `ClaudeProvider[IO](anthropicClient, config.llm.claude, enforcer)` — for all analysis passes
   - `LlmRetryWrapper[IO](config.llm.retry, LlmErrorClassifier())` — wraps provider
2. **Wire components:**
   - `AnalysisRunCreator[IO](transactor)`
   - `Ref[IO].of(None: Option[List[AnalysisFindingType]])` → `findingTypeCache`
   - `AnalysisResultPersister[IO](transactor, runCreator, findingTypeCache)`
   - `BillPromptAssembler` — initialized from Component 8 with GCS prompt blocks
   - `PassExecutor[IO](claudeProvider, promptAssembler, config.analysis.passExecution)`
   - `PipelineLogger[IO]` — initialized with correlation context
   - `PassRouter[IO](config.analysis.passRouting, logger)`
   - `AnalysisOrchestrator[IO](passExecutor, passRouter, persister, embeddingService, config.analysis, logger)`
   - `AnalysisEventPublisher[IO](pubsubPublisher, config.pubsub)`
3. **Subscribe to Pub/Sub:** Pull messages from `bill.decomposition.completed` subscription
4. **Stream processing:** For each event via `parEvalMap(config.parallelism)`:
   a. Deserialize `PipelineEvent[DecompositionCompletedPayload]`
   b. Extract `payload.billId` and `payload.versionId`
   c. Call `orchestrator.analyze(payload.billId, payload.versionId, event.correlationId)`
   d. On success → upsert `stance_materialization_status`:
      ```sql
      INSERT INTO stance_materialization_status (bill_id, has_analysis, all_passes_completed, analysis_completed_at)
      VALUES ($billId, true, true, NOW())
      ON CONFLICT (bill_id) DO UPDATE SET has_analysis = true, all_passes_completed = true, analysis_completed_at = NOW()
      ```
      Then `eventPublisher.publish(analysisResult, event.correlationId)` → `ProcessingResult.Succeeded`
   e. On failure → log error, `ProcessingResult.Failed`
   f. Ack Pub/Sub message
5. **Aggregate:** Collect all `ProcessingResult`s → `PipelineRunSummary`
6. **Return:** `ExitCode.Success` if no fatal errors, `ExitCode.Error` otherwise

### `AnalysisEventPublisher[F[_]]` — Event Emission (10.10)
**Package:** `repcheck.analysis.pipeline.app`

Publishes `analysis.completed` events to Pub/Sub.

```scala
class AnalysisEventPublisher[F[_]: Sync](
  publisher: PubSubPublisher[F],
  config: PubSubConfig
)
```

#### `publish(result: AnalysisResult, correlationId: UUID): F[Unit]`

1. Build `AnalysisCompletedPayload` from `result`:
   - `billId` ← `result.billId`
   - `analysisId` ← `result.analysisId`
   - `topics` ← `result.topics`
   - `modelUsed` ← `result.highestModelUsed`
   - `passesExecuted` ← `result.passesExecuted`
2. Wrap in `PipelineEvent[AnalysisCompletedPayload]` with `correlationId`
3. Serialize to JSON via Circe
4. Publish to `config.outputTopic`

### `AnalysisCompletedPayload` — Event Payload (10.10)
**Package:** `repcheck.analysis.pipeline.app`

```scala
case class AnalysisCompletedPayload(
  billId: String,
  analysisId: UUID,
  topics: List[String],
  modelUsed: String,
  passesExecuted: List[Int]
)
```

> **Cross-component note: Event contract update required.** Component 2 currently defines `AnalysisCompletedEvent` with `passCompleted: Int` (singular) and `llmModel: String`. This Component 10 spec uses `passesExecuted: List[Int]` (all passes that ran) and `modelUsed: String`. Component 2's event definition must be updated to match this spec, since multi-pass analysis produces multiple passes per run. The Component 10 definition is authoritative.

## Key Design Decisions

1. **No sidecar dependency.** Unlike the decomposition pipeline, the analysis pipeline has no Ollama sidecar. It only needs network access to the Claude API. `LlmAdapterConfig.ollama` is `None`.
2. **Component 8 integration.** The prompt assembler from Component 8 is initialized at startup with GCS block loading. Profile selection happens per-pass in the orchestrator.
3. **Finding type caching via `Ref`.** Finding type definitions are loaded from AlloyDB on first access and cached in a `Ref` for the entire run. This is passed to the persister constructor.
4. **Testability via companion extraction.** Same pattern as decomposition pipeline — all logic in a testable method, `run` is wiring only.
5. **Event payload includes passes executed.** Downstream consumers read `stance_materialization_status` to determine readiness. The `analysis.completed` event payload includes `passesExecuted` for observability and is still emitted for possible future consumers, but stance materialization is triggered by the DB status scanner (§11.9), not by consuming this event directly.
6. **Enforcer maxRetries from config.** `StructuredOutputEnforcer` receives `config.llm.structuredOutputMaxRetries`.

## Acceptance Criteria

| Criterion | Test |
|-----------|------|
| Config loads all subsystem configs via PureConfig | Valid `application.conf` → `AnalysisPipelineConfig` populated |
| `config.llm.ollama` is `None` for analysis pipeline | No ollama section in config → `None` |
| AlloyDB transactor created as `Resource` | Transactor acquired and released on shutdown |
| DJL embedding service loaded as `Resource` | Model loaded at startup, released on shutdown |
| `ClaudeProvider` wired with Anthropic SDK and config | Provider receives SDK client and `config.llm.claude` |
| `StructuredOutputEnforcer` receives `structuredOutputMaxRetries` from config | `config.llm.structuredOutputMaxRetries` → enforcer constructor |
| No Ollama provider initialized | Analysis pipeline → no `OllamaProvider` created |
| Prompt assembler initialized with GCS blocks | Component 8 `BillPromptAssembler` wired with `config.promptEngine` |
| Finding type cache `Ref` created and passed to persister | `Ref[IO].of(None)` → `AnalysisResultPersister` constructor |
| `PipelineLogger` wired into orchestrator and router | Logger created and passed to both constructors |
| Pipeline subscribes to `bill.decomposition.completed` | Pub/Sub subscription created for configured topic |
| Input event contains `billId` and `versionId` | `DecompositionCompletedPayload` deserialized → both fields available |
| Each event triggers analysis orchestrator | Event received → `orchestrator.analyze()` called with `billId` + `versionId` |
| Successful analysis emits `analysis.completed` | `orchestrator.analyze` succeeds → event published |
| Failed analysis produces `ProcessingResult.Failed` | `orchestrator.analyze` throws → `Failed` result, no event |
| Pub/Sub message acked after processing | Both success and failure → message acknowledged |
| Bills processed in parallel per config | `parallelism = 4` → `parEvalMap(4)` |
| `PipelineRunSummary` aggregated from results | 3 succeeded, 1 failed → summary reflects counts |
| Correlation ID propagated from input event to output event | Input event correlation ID → output event has same ID |
| `AnalysisCompletedPayload` includes `passesExecuted` as list | Pass 1 + 2 ran → `passesExecuted = [1, 2]` |
| `AnalysisCompletedPayload` includes highest model used | Pass 2 with Sonnet → `modelUsed = "claude-3-sonnet-..."` |
| `AnalysisCompletedPayload` includes extracted topics | 5 topics from analysis → `topics` has 5 entries |
| `AnalysisCompletedPayload` serializes correctly | Payload → JSON → deserialize → same payload |
| Pipeline logic in testable companion method | `runPipeline` callable with mock dependencies |
| `run` method is pure wiring with no domain logic | `run` only creates Resources and calls `runPipeline` |
| Exit code 0 when all succeed, 1 on any failure | All success → `ExitCode.Success`; any failure → `ExitCode.Error` |
| Successful analysis upserts `stance_materialization_status` | Analysis succeeds → `has_analysis = true, all_passes_completed = true, analysis_completed_at = NOW()` |
| `stance_materialization_status` upsert uses `ON CONFLICT (bill_id) DO UPDATE` | Row already exists → updates `has_analysis`, `all_passes_completed`, `analysis_completed_at` |
| `stance_materialization_status` upsert happens before event publish | DB status updated first → event publish second (status is authoritative, event is informational) |
