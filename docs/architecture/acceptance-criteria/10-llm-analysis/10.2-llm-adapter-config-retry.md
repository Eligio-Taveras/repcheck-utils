# Area 10.2 — LLM Adapter Configuration & Retry

> Project: `llm-adapter` | Package: `repcheck.llm.adapter.config`, `repcheck.llm.adapter.retry`

## Purpose

Configuration, retry logic, and rate limiting for LLM API calls. Ensures resilient communication with Claude API including handling of rate limits, transient failures, and provider unavailability.

---

## Classes

### `LlmAdapterConfig` — Top-Level Config (10.2)
**Package:** `repcheck.llm.adapter.config`

PureConfig auto-derived configuration for the entire llm-adapter library.

```scala
case class LlmAdapterConfig(
  claude: ClaudeProviderConfig,
  ollama: Option[OllamaProviderConfig],
  retry: LlmRetryConfig,
  rateLimiting: RateLimitConfig,
  structuredOutputMaxRetries: Int = 2
)
```

- `claude` — always present (analysis pipeline always needs Claude)
- `ollama` — `Some` in decomposition pipeline (has sidecar), `None` in analysis pipeline (no sidecar). When `None`, no `OllamaProvider` is created.

### `ClaudeProviderConfig` — Provider-Specific Config (10.2)
**Package:** `repcheck.llm.adapter.config`

```scala
case class ClaudeProviderConfig(
  apiKey: String,
  modelMappings: Map[ModelTier, String],
  defaultMaxTokens: Int,
  defaultTemperature: Double
)
```

- `apiKey` resolved from `ANTHROPIC_API_KEY` environment variable at startup, never in config files
- `modelMappings` maps `ModelTier` enum values to model IDs (e.g., `Haiku → "claude-3-haiku-20240307"`)

### `LlmRetryConfig` — Retry Settings (10.2)
**Package:** `repcheck.llm.adapter.config`

Controls transport-level retries (HTTP failures, rate limits, timeouts). Does NOT control structured output re-prompting (that's `structuredOutputMaxRetries` on `LlmAdapterConfig`).

```scala
case class LlmRetryConfig(
  maxRetries: Int = 3,
  initialBackoff: FiniteDuration = 1.second,
  backoffMultiplier: Double = 2.0,
  maxBackoff: FiniteDuration = 60.seconds
)
```

> **Deviation from universal defaults:** The 1s initial backoff (vs. 10ms system-wide default) is intentional for LLM APIs. Rate limits recover on minute boundaries — rapid retries waste quota and hit rate limits again.

### `RateLimitConfig` — Rate Limit Settings (10.2)
**Package:** `repcheck.llm.adapter.config`

```scala
case class RateLimitConfig(
  requestsPerMinute: Int,
  tokensPerMinute: Int,
  burstAllowance: Int
)
```

> **Future use.** `RateLimitConfig` is defined and loaded for forward compatibility. The initial implementation enforces rate limits solely through the `LlmRetryWrapper`'s `Retry-After` header handling (reactive). Proactive rate limiting (preemptive throttling based on these config values) is a future enhancement. No `RateLimiter` class is needed for the initial implementation.

### `LlmRetryWrapper[F[_]]` — Retry with LLM-Specific Classification (10.2)
**Package:** `repcheck.llm.adapter.retry`

Wraps the centralized `RetryWrapper[F]` pattern with LLM-specific error classification. This is an independent implementation following the same pattern as Component 2's `RetryWrapper` but with LLM-specific retry timing, `Retry-After` awareness, and the `LlmErrorClassifier`.

```scala
class LlmRetryWrapper[F[_]: Temporal](
  retryConfig: LlmRetryConfig,
  classifier: LlmErrorClassifier
)
```

#### `withRetry[A](operation: F[A], operationName: String): F[A]`

1. Execute `operation`
2. On success → return `A`
3. On failure → classify error via `classifier.classify(error)`
4. If `Systemic` → raise `ProviderUnavailable(operationName, error.getMessage)` immediately (no retry)
5. If `Transient`:
   a. Check `classifier.extractRetryAfter(error)` → if `Some(duration)`, use that duration
   b. Otherwise compute backoff: `min(initialBackoff * multiplier^attempt, maxBackoff)` + jitter
   c. Sleep for computed duration
   d. Decrement retry counter, re-execute `operation`
6. If retries exhausted → raise `LlmCallFailed(operationName, retryConfig.maxRetries, error.getMessage)`

### `LlmErrorClassifier` — Error Classification (10.2)
**Package:** `repcheck.llm.adapter.retry`

Implements `ErrorClassifier` for LLM provider errors. Has an additional `extractRetryAfter` method beyond the base `ErrorClassifier` trait — this is why `LlmRetryWrapper` takes the concrete `LlmErrorClassifier` type, not the `ErrorClassifier` trait.

```scala
class LlmErrorClassifier extends ErrorClassifier
```

#### `classify(error: Throwable): ErrorSeverity`

1. Match on Anthropic SDK exception types:
   - `RateLimitException` (429) → `Transient`
   - `InternalServerException` (5xx) → `Transient`
   - `ConnectionException` / timeout → `Transient`
   - `AuthenticationException` (401) → `Systemic`
   - `BadRequestException` (400) → `Systemic`
   - `NotFoundException` (404, model not found) → `Systemic`
2. For Ollama HTTP errors:
   - Status 5xx or connection refused → `Transient`
   - Status 4xx → `Systemic`
3. Default unknown exceptions → `Transient` (safer to retry than halt)

#### `extractRetryAfter(error: Throwable): Option[FiniteDuration]`

1. If error is `RateLimitException` and response contains `Retry-After` header → parse to `FiniteDuration`
2. Otherwise → `None`

### Error Types (10.2)
**Package:** `repcheck.llm.adapter.errors`

```scala
case class LlmCallFailed(
  operationName: String,
  attempts: Int,
  lastError: String
) extends Exception(s"LLM call failed ($operationName) after $attempts attempts: $lastError")

case class ProviderUnavailable(
  provider: String,
  reason: String
) extends Exception(s"Provider $provider unavailable: $reason")
```

## Retry Scope Summary

Three separate retry/re-prompt mechanisms exist. They do NOT overlap:

| Mechanism | Scope | Config | Trigger |
|-----------|-------|--------|---------|
| `LlmRetryWrapper.withRetry` | Transport failures (HTTP errors, rate limits, timeouts) | `LlmRetryConfig.maxRetries` (default 3) | 429, 5xx, connection errors |
| `StructuredOutputEnforcer.enforce` | Parse failures (malformed JSON from LLM) | `LlmAdapterConfig.structuredOutputMaxRetries` (default 2) | Circe decode failure |
| `OllamaProviderConfig.maxRetries` | **Removed** — Ollama transport retries use `LlmRetryWrapper` like Claude | N/A | N/A |

---

## Key Design Decisions

1. **LLM-specific retry timing.** Initial backoff is 1s (not 10ms like REST APIs) because LLM rate limits recover on minute boundaries. Rapid retries waste quota.
2. **Rate limit awareness.** The retry wrapper reads `Retry-After` headers and waits exactly that long, rather than using generic exponential backoff for 429s.
3. **Proactive rate limiting deferred.** `RateLimitConfig` is defined for forward compatibility. Initial implementation uses reactive rate limiting only (retry on 429 with `Retry-After`).
4. **API key from environment.** Never in config files. Always from `ANTHROPIC_API_KEY` environment variable, resolved at startup.
5. **Concrete classifier type.** `LlmRetryWrapper` takes `LlmErrorClassifier` (not `ErrorClassifier` trait) because it needs the `extractRetryAfter` method that's not on the base trait.

## Acceptance Criteria

| Criterion | Test |
|-----------|------|
| `LlmAdapterConfig` loads from `application.conf` via PureConfig | Valid config → all fields populated |
| `LlmAdapterConfig.ollama` is `None` when not in config | Analysis pipeline config (no ollama section) → `ollama = None` |
| `LlmAdapterConfig.ollama` is `Some` when present | Decomposition pipeline config → `ollama = Some(OllamaProviderConfig(...))` |
| `ClaudeProviderConfig.apiKey` read from environment variable | `ANTHROPIC_API_KEY` set → apiKey populated |
| `ClaudeProviderConfig.modelMappings` keyed by `ModelTier` | Config with `haiku = "claude-3-haiku-..."` → `Map(Haiku -> "claude-3-haiku-...")` |
| `LlmRetryConfig` defaults are 3 retries, 1s initial, 2x multiplier, 60s cap | Default config → verify all defaults |
| Rate limit 429 is classified as `Transient` | `RateLimitException` → `Transient` |
| Server error 5xx is classified as `Transient` | `InternalServerException` → `Transient` |
| Network timeout is classified as `Transient` | `ConnectionException` → `Transient` |
| Auth failure 401 is classified as `Systemic` | `AuthenticationException` → `Systemic` |
| Bad request 400 is classified as `Systemic` | `BadRequestException` → `Systemic` |
| Transient error retries with exponential backoff | Mock fails 2x then succeeds → 3 calls total, increasing delays |
| Systemic error raises immediately without retry | `Systemic` error → `ProviderUnavailable` raised, 1 call only |
| `Retry-After` header respected for 429s | 429 with `Retry-After: 5` → next retry after ~5s |
| Backoff capped at `maxBackoff` | Many retries → delay never exceeds 60s |
| `LlmCallFailed` raised after exhausting retries | 4 consecutive failures (3 retries) → `LlmCallFailed` with `operationName` |
| Jitter applied to backoff | Multiple retries → delays not exactly exponential (randomized) |
| Ollama connection refused classified as `Transient` | HTTP connection refused → `Transient` |
| Ollama 4xx classified as `Systemic` | HTTP 400 from Ollama → `Systemic` |
| Unknown exceptions default to `Transient` | `RuntimeException` → `Transient` |
