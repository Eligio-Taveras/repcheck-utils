# Area 10.7 — Analysis Result Persistence

> Project: `bill-analysis-pipeline` | Package: `repcheck.analysis.pipeline.persistence`

## Purpose

Maps LLM output schemas (§1.6) to domain objects (§1.9) and persists them to AlloyDB analysis layer tables. Called after each pass completes.

---

## Classes

### `AnalysisResultPersister[F[_]]` — AlloyDB Analysis Persistence (10.7)
**Package:** `repcheck.analysis.pipeline.persistence`

Translates typed LLM outputs into database rows and writes them transactionally.

```scala
class AnalysisResultPersister[F[_]: Async](
  xa: Transactor[F],
  runCreator: AnalysisRunCreator[F],
  findingTypeCache: Ref[F, Option[List[AnalysisFindingType]]]
)
```

#### `createAnalysisRun(billId: String, versionId: UUID): F[UUID]`

1. Delegate to `runCreator.create(billId, versionId)`
2. Return generated `analysisId`

#### `fetchConceptGroups(versionId: UUID): F[List[AnalysisConceptGroup]]`

1. Query `SELECT concept_group_id, group_id, title, simplified_text FROM ${Tables.BillConceptGroups} WHERE version_id = $versionId`
2. Map to `List[AnalysisConceptGroup]` (type defined in 10.6)
3. Return results

#### `fetchFindingTypes(): F[List[AnalysisFindingType]]`

1. Check `findingTypeCache` — if `Some`, return cached value
2. Query `SELECT finding_type_id, code, name, description FROM ${Tables.FindingTypes}`
3. Map to `List[AnalysisFindingType]` (type defined in 10.6)
4. Store in `findingTypeCache`
5. Return results

> **Cross-component note:** The `finding_types` table needs a `code` column (e.g., "pork_barrel", "stance", "civil_liberties") in addition to the existing `finding_type_id` and `display_name` columns. This requires a Component 1 migration update. The `code` column is the lookup key used by this persister — never hardcode numeric IDs.

#### `persistPassResults(analysisId: UUID, passNumber: Int, outputs: List[ConceptGroupPassOutput]): F[Unit]`

All writes for a single pass execute in a single Doobie transaction:

1. **Map BillSummaryOutput** → update `${Tables.BillAnalyses}` row:
   - `summary` ← `output.summary.text`
   - `reading_level` ← `output.summary.readingLevel`
   - Only the first concept group's summary is used for the bill-level summary (or concatenated)
2. **Map TopicClassificationOutput** → insert `${Tables.BillAnalysisTopics}` rows:
   - One row per topic: `analysis_id`, `topic_code`, `confidence`, `pass_number`, `concept_group_id`
3. **Map StanceClassificationOutput** → insert `${Tables.BillFindings}` rows:
   - `finding_type_id` ← lookup by code "stance" from `AnalysisFindingType` list
   - One row per topic-stance pair: `analysis_id`, `finding_type_id`, `summary`, `confidence`, `pass_number`, `concept_group_id`
4. **Map PorkDetectionOutput** → insert `${Tables.BillFindings}` rows:
   - `finding_type_id` ← lookup by code "pork_barrel"
   - One row per detection: `analysis_id`, `finding_type_id`, `summary`, `confidence`, `pass_number`, `concept_group_id`
5. **Map ImpactAnalysisOutput** → insert `${Tables.BillFindings}` rows:
   - `finding_type_id` ← lookup by impact category code (e.g., "civil_liberties", "environmental")
   - One row per impact: `analysis_id`, `finding_type_id`, `summary`, `severity`, `pass_number`, `concept_group_id`
6. **Map FiscalEstimateOutput** → upsert `${Tables.BillFiscalEstimates}` row:
   - `analysis_id`, `estimated_cost`, `timeframe`, `confidence`, `pass_number`, `concept_group_id`
7. **Map per-concept results** → insert `${Tables.BillConceptSummaries}` rows:
   - `analysis_id`, `concept_group_id`, `summary`, `topics` (from output), `reading_level`, `key_points`, `pass_number`
8. All operations use `Tables` constants, never hardcoded strings
9. Finding type lookups use `code` strings, resolved from the `AnalysisFindingType` list

> **Cross-component note: DO updates required in Component 1 §1.9.**
> - `BillAnalysisDO` needs: `status`, `versionId`, `createdAt`, `completedAt`, `passesExecuted`, `highestModelUsed`, `failureReason`, `highProfileScore`, `mediaCoverageLevel`, `appropriationsEstimate`, `stanceConfidence`, `routingReasoning`, `overallConfidence`, `crossConceptContradictionScore`, `expectedVoteContention`, `contradictionDetails`, `routingReasoningPass2`
> - `BillFindingDO` needs: `passNumber` field
> - `BillAnalysisTopicDO` needs: `passNumber` field
> - `BillConceptSummaryDO` needs: `passNumber`, `topics`, `readingLevel`, `keyPoints` fields
> - `BillFiscalEstimateDO` needs: `passNumber` field; rename `timeframe` to match 10.7 usage
> - `FindingTypeDO` needs: `code` field

#### `persistPass1RoutingScores(analysisId: UUID, scores: Pass1RoutingScores): F[Unit]`

1. Update `${Tables.BillAnalyses}` row:
   - `high_profile_score` ← `scores.highProfileScore`
   - `media_coverage_level` ← `scores.mediaCoverageLevel`
   - `appropriations_estimate` ← `scores.appropriationsEstimate`
   - `stance_confidence` ← `scores.stanceConfidence`
   - `routing_reasoning` ← `scores.routingReasoning`

#### `persistPass2RoutingScores(analysisId: UUID, scores: Pass2RoutingScores): F[Unit]`

1. Update `${Tables.BillAnalyses}` row:
   - `overall_confidence` ← `scores.overallConfidence`
   - `cross_concept_contradiction_score` ← `scores.crossConceptContradictionScore`
   - `expected_vote_contention` ← `scores.expectedVoteContention`
   - `contradiction_details` ← `scores.contradictionDetails`
   - `routing_reasoning_pass2` ← `scores.routingReasoning`

#### `persistFindingEmbeddings(analysisId: UUID, embeddings: List[(UUID, Array[Float])]): F[Unit]`

1. For each `(findingId, embedding)`:
   - `UPDATE ${Tables.BillFindings} SET embedding = $embedding WHERE finding_id = $findingId`
2. Batch update in a single transaction

#### `completeAnalysisRun(analysisId: UUID, passesExecuted: List[Int], highestModel: String): F[Unit]`

1. Delegate to `runCreator.complete(analysisId, passesExecuted, highestModel)`

#### `failAnalysisRun(analysisId: UUID, reason: String): F[Unit]`

1. Delegate to `runCreator.fail(analysisId, reason)`

### `AnalysisRunCreator[F[_]]` — Analysis Run Lifecycle (10.7)
**Package:** `repcheck.analysis.pipeline.persistence`

Manages the `bill_analyses` row lifecycle.

```scala
class AnalysisRunCreator[F[_]: Async](
  xa: Transactor[F]
)
```

#### `create(billId: String, versionId: UUID): F[UUID]`

1. Generate `analysisId: UUID`
2. Insert into `${Tables.BillAnalyses}`:
   - `analysis_id`, `bill_id`, `version_id`, `status = "in_progress"`, `created_at = Instant.now`
3. Return `analysisId`

#### `complete(analysisId: UUID, passesExecuted: List[Int], highestModel: String): F[Unit]`

1. Update `${Tables.BillAnalyses}`:
   - `status = "completed"`
   - `passes_executed` ← `passesExecuted` (stored as integer array)
   - `highest_model_used` ← `highestModel`
   - `completed_at = Instant.now`

#### `fail(analysisId: UUID, reason: String): F[Unit]`

1. Update `${Tables.BillAnalyses}`:
   - `status = "failed"`
   - `failure_reason` ← `reason`
   - `completed_at = Instant.now`

## Key Design Decisions

1. **Finding type lookup by code, not ID.** The `finding_types` table is seeded data. The persister looks up `finding_type_id` by `code` (e.g., "pork_barrel") to avoid coupling to auto-generated IDs.
2. **Per-pass transactional writes.** Each pass's results are written in one transaction. If Pass 2 fails to persist, Pass 1 results are still in the database.
3. **Append-only findings.** Pass 2/3 findings are appended alongside Pass 1 findings, not replacements. The `pass_number` on each finding indicates which pass produced it, and consumers can choose the highest-pass version.
4. **Routing scores as denormalized columns.** `Pass1RoutingScores` and `Pass2RoutingScores` fields are stored as individual columns on `bill_analyses`, not as JSONB. This makes routing decisions queryable and auditable without JSON parsing.
5. **Embeddings written separately.** The persister writes the core data first, then the embedding service generates vectors, then a second write adds embeddings. This avoids blocking the main persistence on embedding generation.
6. **Finding types cached.** `fetchFindingTypes()` uses a `Ref` to cache results after the first call since finding types are seeded migration data that doesn't change during a pipeline run.
7. **`concept_group_id` links analysis to decomposition.** Every finding, topic, concept summary, and fiscal estimate row carries `concept_group_id` to trace which concept group produced it.

## Acceptance Criteria

| Criterion | Test |
|-----------|------|
| Analysis run created with status "in_progress" | `create()` → row in `bill_analyses` with `status = "in_progress"` |
| Analysis run ID is UUID | `create()` → valid UUID returned |
| Concept groups fetched as `AnalysisConceptGroup` | 3 groups in DB → returns 3 with `conceptGroupId`, `groupId`, `title`, `simplifiedText` |
| Finding types fetched with `code` field | 10 finding types in DB → returns 10 `AnalysisFindingType` with `code` populated |
| Finding types cached after first call | Two calls → single DB query executed |
| `BillSummaryOutput` mapped to `bill_analyses` columns | Summary + reading level → row updated |
| `TopicClassificationOutput` mapped to `bill_analysis_topics` rows | 5 topics → 5 rows with confidence, pass_number, concept_group_id |
| `StanceClassificationOutput` mapped to `bill_findings` with "stance" type | Finding type looked up by code "stance" → rows inserted with pass_number |
| `PorkDetectionOutput` mapped to `bill_findings` with "pork_barrel" type | Finding type looked up by code "pork_barrel" → rows inserted |
| `ImpactAnalysisOutput` mapped to `bill_findings` per impact category | 3 impacts → 3 rows with appropriate finding type codes |
| `FiscalEstimateOutput` mapped to `bill_fiscal_estimates` | Cost estimate → row upserted with pass_number |
| Per-concept results mapped to `bill_concept_summaries` | 3 groups → 3 rows with `concept_group_id`, `summary`, `topics`, `pass_number` |
| Pass 1 results persist separately from Pass 2 results | Pass 1 findings have `pass_number = 1`, Pass 2 have `pass_number = 2` |
| Pass 2 findings appended, not replacing Pass 1 | After Pass 2 → both Pass 1 and Pass 2 findings in DB |
| Pass 1 routing scores stored as columns | `high_profile_score`, `media_coverage_level`, etc. on `bill_analyses` |
| Pass 2 routing scores stored as columns (if ran) | `overall_confidence`, `cross_concept_contradiction_score`, `expected_vote_contention` etc. |
| Pass 2 routing score columns NULL when Pass 2 didn't run | Pass 1 only → Pass 2 columns are NULL |
| Finding type lookup uses `code`, not ID | "pork_barrel" → correct `finding_type_id` resolved from cache |
| All writes per pass are transactional | Doobie `.transact(xa)` wraps all inserts for a pass |
| Partial pass failure rolls back that pass's writes | Exception during topic insert → entire pass transaction rolled back |
| Finding embeddings written separately after generation | `persistFindingEmbeddings` called independently from `persistPassResults` |
| Analysis run completed with correct metadata | `complete()` → `status = "completed"`, `passes_executed = [1,2]`, model recorded |
| Analysis run failed with reason | `fail()` → `status = "failed"`, `failure_reason` populated |
| `failAnalysisRun` prevents stuck "in_progress" runs | Orchestrator catches error → calls `failAnalysisRun` |
| All table names from `Tables` constants | Verify no hardcoded table strings in SQL |
| Every finding row has `concept_group_id` | Findings linked back to source concept group |
