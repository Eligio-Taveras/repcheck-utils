# Area 10.6 — Multi-Pass Analysis Orchestrator

> Project: `bill-analysis-pipeline` | Package: `repcheck.analysis.pipeline.analysis`

## Purpose

Orchestrates the multi-pass LLM analysis chain (Pass 1/2/3) on decomposed bills. Uses prompt profiles from Component 8, LLM providers from 10.1, and finding type descriptions to guide analysis. Includes pass routing logic driven entirely by LLM-produced routing scores.

---

## Types Used Across This Area

Several types are referenced by the orchestrator, executor, and persister. They are defined here for clarity:

### `AnalysisConceptGroup` — Analysis-Layer Concept Group (10.6)
**Package:** `repcheck.analysis.pipeline.analysis`

Query result type representing a concept group ready for analysis. Fetched from AlloyDB by `AnalysisResultPersister.fetchConceptGroups`. Distinct from Component 8's `ConceptGroup` (which has `topics` and `sectionReferences` for prompt assembly) — this is the persistence-layer representation.

```scala
case class AnalysisConceptGroup(
  conceptGroupId: UUID,
  groupId: String,
  title: String,
  simplifiedText: String
)
```

- `conceptGroupId` — UUID primary key from `bill_concept_groups`
- `groupId` — stable semantic identifier (e.g., "transportation-funding") from clustering
- `title` — human-readable name
- `simplifiedText` — Haiku-simplified summary from decomposition

### `AnalysisFindingType` — Finding Type for Prompt Injection (10.6)
**Package:** `repcheck.analysis.pipeline.analysis`

Query result from `finding_types` table. Used to inject finding type descriptions into LLM prompts so the model classifies results along established categories.

```scala
case class AnalysisFindingType(
  findingTypeId: Int,
  code: String,
  name: String,
  description: String
)
```

- `code` — snake_case identifier used for lookups (e.g., "pork_barrel", "stance", "civil_liberties"). This is the column that 10.7 uses for finding type resolution.

> **Cross-component note:** Component 1 §1.9 `FindingTypeDO` needs a `code` field added (currently has `findingTypeId` and `displayName` only). The `code` column must be added to the `finding_types` table migration in Component 1.

### `ConceptGroupPassOutput` — Per-Concept-Group LLM Output (10.6)
**Package:** `repcheck.analysis.pipeline.analysis`

The structured output type returned by the LLM for a single concept group's analysis. This is what `completeStructured[ConceptGroupPassOutput]` returns. Each concept group produces one of these per pass. Requires Circe semi-auto `Decoder`.

```scala
case class ConceptGroupPassOutput(
  conceptGroupId: UUID,
  summary: BillSummaryOutput,
  topics: TopicClassificationOutput,
  stance: StanceClassificationOutput,
  pork: PorkDetectionOutput,
  impact: ImpactAnalysisOutput,
  fiscal: FiscalEstimateOutput,
  routingAssessment: Option[RoutingAssessment]
)

object ConceptGroupPassOutput {
  implicit val decoder: Decoder[ConceptGroupPassOutput] = deriveDecoder
}
```

- Contains all 6 output schemas (§1.6) for one concept group
- `routingAssessment` — optional because Pass 3 does not produce routing scores
- `BillSummaryOutput`, `TopicClassificationOutput`, etc. are defined in Component 1 §1.6 (`repcheck-shared-models`)

### `RoutingAssessment` — Per-Concept-Group Routing Data (10.6)
**Package:** `repcheck.analysis.pipeline.analysis`

Raw routing data from a single concept group's LLM response. Aggregated across concept groups into `Pass1RoutingScores` or `Pass2RoutingScores` by the orchestrator.

```scala
case class RoutingAssessment(
  highProfileScore: Option[Double],
  mediaCoverageLevel: Option[Double],
  appropriationsEstimate: Option[BigDecimal],
  stanceConfidence: Option[Double],
  overallConfidence: Option[Double],
  crossConceptContradictionScore: Option[Double],
  expectedVoteContention: Option[Double],
  reasoning: String
)

object RoutingAssessment {
  implicit val decoder: Decoder[RoutingAssessment] = deriveDecoder
}
```

- Pass 1 prompts request: `highProfileScore`, `mediaCoverageLevel`, `appropriationsEstimate`, `stanceConfidence`
- Pass 2 prompts request: `overallConfidence`, `crossConceptContradictionScore`, `expectedVoteContention`
- Fields are `Option` because each pass only requests a subset

## Classes

### `AnalysisOrchestrator[F[_]]` — Analysis Pipeline Coordinator (10.6)
**Package:** `repcheck.analysis.pipeline.analysis`

The central coordinator for bill analysis. Called by the pipeline entry point for each decomposed bill.

```scala
class AnalysisOrchestrator[F[_]: Async](
  passExecutor: PassExecutor[F],
  passRouter: PassRouter[F],
  persister: AnalysisResultPersister[F],
  embeddingService: SemanticEmbeddingService[F],
  config: AnalysisConfig,
  logger: PipelineLogger[F]
)
```

#### `analyze(billId: String, versionId: UUID, correlationId: UUID): F[AnalysisResult]`

Primary entry point for analyzing a single decomposed bill.

1. **Create analysis run:** `persister.createAnalysisRun(billId, versionId)` → `analysisId: UUID`
2. **Fetch concept groups:** `persister.fetchConceptGroups(versionId)` → `List[AnalysisConceptGroup]`
3. **Fetch finding types:** `persister.fetchFindingTypes()` → `List[AnalysisFindingType]` (cached after first call)
4. **Execute Pass 1** (always):
   a. `passExecutor.executePass(1, conceptGroups, findingTypes, priorResults = None)` → `List[ConceptGroupPassOutput]`
   b. Aggregate routing assessments → `Pass1RoutingScores`
   c. `persister.persistPassResults(analysisId, 1, pass1Outputs)`
   d. Generate finding embeddings: `embeddingService.embedFindings(pass1Outputs)` → persist via `persister.persistFindingEmbeddings`
   e. `persister.persistPass1RoutingScores(analysisId, pass1Scores)`
5. **Evaluate Pass 2 routing:** `passRouter.shouldRunPass2(pass1Scores)` → `Boolean`
6. **Execute Pass 2** (if routed):
   a. `passExecutor.executePass(2, conceptGroups, findingTypes, priorResults = Some(pass1Outputs))` → `List[ConceptGroupPassOutput]`
   b. Aggregate routing assessments → `Pass2RoutingScores`
   c. `persister.persistPassResults(analysisId, 2, pass2Outputs)`
   d. Generate finding embeddings → persist
   e. `persister.persistPass2RoutingScores(analysisId, pass2Scores)`
7. **Evaluate Pass 3 routing:** `passRouter.shouldRunPass3(pass2Scores)` → `Boolean`
8. **Execute Pass 3** (if routed):
   a. `passExecutor.executePass(3, conceptGroups, findingTypes, priorResults = Some(pass1Outputs ++ pass2Outputs))`
   b. `persister.persistPassResults(analysisId, 3, pass3Outputs)`
   c. Generate finding embeddings → persist
9. **Update analysis run:** `persister.completeAnalysisRun(analysisId, passesExecuted, highestModel)`
10. **Return:** `AnalysisResult` with summary

> **Error handling:**
> - If Pass 1 fails entirely (`AnalysisPassFailed`) → mark analysis run as failed via `persister.failAnalysisRun(analysisId, reason)`, propagate error
> - If Pass 2/3 fails entirely → log error, mark analysis run as completed with only earlier passes, do NOT propagate
> - `PersistenceFailed` during any step → catch, mark analysis run as failed, propagate error
> - Per-concept-group failures within a pass → handled inside `PassExecutor` (log and continue)

**Routing score aggregation** (step 4b, 6b):
- `Pass1RoutingScores.highProfileScore` ← max across all concept groups
- `Pass1RoutingScores.mediaCoverageLevel` ← max across all concept groups
- `Pass1RoutingScores.appropriationsEstimate` ← sum across concept groups (total bill spending)
- `Pass1RoutingScores.stanceConfidence` ← min across concept groups (weakest link)
- `Pass2RoutingScores.overallConfidence` ← min across concept groups
- `Pass2RoutingScores.crossConceptContradictionScore` ← max across concept groups
- `Pass2RoutingScores.expectedVoteContention` ← max across concept groups

### `PassExecutor[F[_]]` — Single Pass Execution (10.6)
**Package:** `repcheck.analysis.pipeline.analysis`

Executes a single analysis pass across all concept groups for a bill.

```scala
class PassExecutor[F[_]: Async](
  llmProvider: LlmProvider[F],
  promptAssembler: BillPromptAssembler,
  config: PassExecutionConfig
)
```

#### `executePass(passNumber: Int, conceptGroups: List[AnalysisConceptGroup], findingTypes: List[AnalysisFindingType], priorResults: Option[List[ConceptGroupPassOutput]]): F[List[ConceptGroupPassOutput]]`

Returns `List[ConceptGroupPassOutput]` — the routing scores are embedded within each `ConceptGroupPassOutput.routingAssessment`. The orchestrator aggregates them.

1. Determine prompt profile based on `passNumber`:
   - Pass 1 → `pass1-extraction` profile (Haiku)
   - Pass 2 → `pass2-deep-analysis` profile (Sonnet)
   - Pass 3 → `pass3-ambiguity-resolution` profile (Opus)
2. Determine model tier: Pass 1 → `Haiku`, Pass 2 → `Sonnet`, Pass 3 → `Opus`
3. For each concept group via `parEvalMap(config.conceptParallelism)`:
   a. Assemble prompt via `promptAssembler.assemble(profile, conceptGroup, findingTypes)`
   b. If `priorResults.isDefined`, inject prior pass results for this concept group into prompt context
   c. Build `LlmRequest(systemPrompt, userPrompt, modelTier, maxTokens, temperature = 0.0)`
   d. Call `llmProvider.completeStructured[ConceptGroupPassOutput](request)`
   e. Set `output.conceptGroupId = conceptGroup.conceptGroupId` (ensures link back to source)
   f. On failure → log error with `conceptGroup.groupId`, return `None` (don't fail entire pass)
4. Collect successful results, filter out `None`s
5. If all concept groups failed → raise `AnalysisPassFailed(billId, passNumber, "all concept groups failed")`
6. Return successful `List[ConceptGroupPassOutput]`

### `Pass1RoutingScores` — Aggregated Routing Scores from Pass 1 (10.6)
**Package:** `repcheck.analysis.pipeline.analysis`

Aggregated from per-concept-group `RoutingAssessment` fields. Feeds the Pass 2 routing decision.

```scala
case class Pass1RoutingScores(
  highProfileScore: Double,
  mediaCoverageLevel: Double,
  appropriationsEstimate: Option[BigDecimal],
  stanceConfidence: Double,
  routingReasoning: String
)
```

- `highProfileScore` — 0.0 to 1.0, max across concept groups. LLM's assessment of bill prominence.
- `mediaCoverageLevel` — 0.0 to 1.0, max across concept groups. LLM's assessment of likely media attention.
- `appropriationsEstimate` — sum across concept groups. Estimated dollar amount if the bill involves spending.
- `stanceConfidence` — 0.0 to 1.0, min across concept groups. LLM's weakest confidence in stance classification.
- `routingReasoning` — concatenated reasoning from concept groups with the most extreme scores.

### `Pass2RoutingScores` — Aggregated Routing Scores from Pass 2 (10.6)
**Package:** `repcheck.analysis.pipeline.analysis`

Aggregated from per-concept-group `RoutingAssessment` fields. Feeds the Pass 3 routing decision.

```scala
case class Pass2RoutingScores(
  overallConfidence: Double,
  crossConceptContradictionScore: Double,
  expectedVoteContention: Double,
  contradictionDetails: Option[String],
  routingReasoning: String
)
```

- `overallConfidence` — 0.0 to 1.0, min across concept groups. LLM's lowest confidence across all output schemas.
- `crossConceptContradictionScore` — 0.0 to 1.0, max across concept groups. Measures conflicting stance/impact assessments.
- `expectedVoteContention` — 0.0 to 1.0, max across concept groups. LLM's assessment of how contentious the expected vote will be (party-line splits, bipartisan disagreement, etc.).
- `contradictionDetails` — description of detected contradictions if score is high.
- `routingReasoning` — concatenated reasoning.

### `PassRouter[F[_]]` — Pass Routing Logic (10.6)
**Package:** `repcheck.analysis.pipeline.analysis`

Determines whether a bill should proceed to the next pass based on LLM-produced routing scores.

```scala
class PassRouter[F[_]: Sync](
  config: PassRoutingConfig,
  logger: PipelineLogger[F]
)
```

#### `shouldRunPass2(scores: Pass1RoutingScores): F[Boolean]`

1. Evaluate criteria (any match → `true`):
   - `scores.highProfileScore > config.highProfileThreshold`
   - `scores.mediaCoverageLevel > config.mediaCoverageThreshold`
   - `scores.appropriationsEstimate.exists(_ > config.appropriationsThreshold)`
   - `scores.stanceConfidence < config.stanceConfidenceMinimum`
2. Log routing decision: scores, thresholds, matched criteria, final decision
3. Return `Boolean`

#### `shouldRunPass3(scores: Pass2RoutingScores): F[Boolean]`

1. Evaluate criteria (any match → `true`):
   - `scores.overallConfidence < config.confidenceMinimum`
   - `scores.expectedVoteContention > config.contentionThreshold`
   - `scores.crossConceptContradictionScore > config.contradictionThreshold`
2. Log routing decision: scores, thresholds, matched criteria, final decision
3. Return `Boolean`

> **No external vote data dependency.** Pass 3 routing uses the LLM's own assessment of expected vote contention (`expectedVoteContention`), not actual vote data from Component 6. This eliminates the cross-pipeline data dependency and works for newly introduced bills that haven't been voted on yet.

## Key Design Decisions

1. **Routing scores are first-class structured output.** `RoutingAssessment` is part of every `ConceptGroupPassOutput` — the model is explicitly asked to assess these dimensions in the prompt. They are not inferred or computed after the fact.
2. **Per-concept routing, aggregated per-bill.** Each concept group produces its own `RoutingAssessment`. The orchestrator aggregates (max for prominence, min for confidence, sum for spending). This ensures the most notable signal from any concept group triggers deeper analysis.
3. **Vote contention assessed by LLM.** Instead of requiring actual vote data from Component 6 (which may not exist for new bills), the LLM assesses expected vote contention as part of Pass 2. This eliminates cross-pipeline data dependencies.
4. **Routing scores are persisted.** Stored as denormalized columns on the analysis run (10.7) so routing decisions are auditable.
5. **Per-concept-group calls.** Each concept group is an independent LLM call. This handles bills of any size but means large bills generate many calls. Parallelism is configurable.
6. **Finding types in every prompt.** The `finding_types` table definitions are injected into every prompt so the LLM classifies results along established categories.
7. **Graceful per-concept failure.** If one concept group fails analysis, the others still proceed. The failed group is logged but doesn't halt the bill's analysis.
8. **Append-only analysis runs.** Every analysis creates a new `bill_analyses` row. Prior runs are never modified.
9. **`PersistenceFailed` handled by orchestrator.** If persistence fails at any step, the orchestrator catches it, marks the analysis run as failed, and propagates the error. This prevents permanently-stuck "in_progress" runs.

## Acceptance Criteria

| Criterion | Test |
|-----------|------|
| Analysis run created in DB before Pass 1 | Mock persister → `createAnalysisRun` called first |
| Concept groups fetched by version ID | Mock persister → `fetchConceptGroups(versionId)` returns `List[AnalysisConceptGroup]` |
| Finding types fetched and injected into prompts | Mock persister → `fetchFindingTypes()` returns types, passed to executor |
| Finding types cached after first call | Two bills processed → `fetchFindingTypes` DB query only once |
| Pass 1 always executes | Any bill → `executePass(1, ...)` called |
| Pass 1 uses Haiku model tier | Verify `LlmRequest.modelTier = Haiku` |
| Pass 1 results persisted before routing evaluation | `persistPassResults` called before `shouldRunPass2` |
| Pass 1 routing scores aggregated from concept groups | 3 groups: scores [0.8, 0.5, 0.7] → `highProfileScore = 0.8` (max) |
| Pass 1 routing scores persisted | `persistPass1RoutingScores` called with aggregated scores |
| Finding embeddings generated after each pass | `embeddingService.embedFindings` called after each pass |
| Pass 2 triggered when high-profile score exceeds threshold | `highProfileScore = 0.8`, threshold `0.7` → Pass 2 runs |
| Pass 2 triggered when media coverage exceeds threshold | `mediaCoverageLevel = 0.7`, threshold `0.6` → Pass 2 runs |
| Pass 2 triggered when appropriations exceed threshold | `appropriationsEstimate = Some(1B)`, threshold `500M` → Pass 2 runs |
| Pass 2 triggered when stance confidence below minimum | `stanceConfidence = 0.4`, minimum `0.6` → Pass 2 runs |
| Pass 2 skipped when no criteria met | All scores below thresholds → Pass 2 does not run |
| Pass 2 uses Sonnet model tier | Verify `LlmRequest.modelTier = Sonnet` |
| Pass 2 receives Pass 1 results as prior context | `priorResults = Some(pass1Outputs)` passed to executor |
| Pass 3 triggered when confidence below threshold | `overallConfidence = 0.5`, minimum `0.7` → Pass 3 runs |
| Pass 3 triggered when cross-concept contradiction high | `crossConceptContradictionScore = 0.6`, threshold `0.5` → Pass 3 runs |
| Pass 3 triggered when expected vote contention high | `expectedVoteContention = 0.7`, threshold `0.6` → Pass 3 runs |
| Pass 3 uses Opus model tier | Verify `LlmRequest.modelTier = Opus` |
| Pass 3 receives Pass 1 + Pass 2 results as prior context | `priorResults` contains both pass outputs |
| Concept groups processed in parallel per config | `conceptParallelism = 4` → verify `parEvalMap(4)` |
| Per-concept failure does not fail entire pass | 1 of 3 groups fails → 2 results returned, error logged |
| All concept groups fail → `AnalysisPassFailed` raised | 0 successful results → exception raised |
| Pass 1 failure is fatal to the bill | `AnalysisPassFailed` from Pass 1 → analysis run marked failed |
| Pass 2/3 failure logged but does not fail the bill | `AnalysisPassFailed` from Pass 2 → logged, analysis completes with Pass 1 |
| `PersistenceFailed` marks analysis run as failed | Persistence throws → `failAnalysisRun` called, error propagated |
| `ConceptGroupPassOutput` decoded from LLM response | Valid JSON with all 6 schemas + routing → decoded |
| `RoutingAssessment` fields are optional per pass | Pass 1 output has `highProfileScore` but not `overallConfidence` → OK |
| Routing decisions logged with scores and thresholds | Log output contains score values and threshold values |
| All routing thresholds configurable | Override all thresholds in config → routing uses overrides |
| No external vote data dependency | Pass 3 routing uses `expectedVoteContention` from LLM, not DB query |
| `AnalysisResult` includes routing scores | Result has `pass1RoutingScores` always, `pass2RoutingScores` if Pass 2 ran |
