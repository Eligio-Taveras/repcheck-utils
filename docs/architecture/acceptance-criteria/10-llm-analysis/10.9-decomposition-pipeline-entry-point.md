# Area 10.9 — Decomposition Pipeline Entry Point

> Project: `bill-decomposition-pipeline` | Package: `repcheck.decomposition.pipeline.app`, `repcheck.decomposition.pipeline.config`

## Purpose

Cloud Run Job entry point for the decomposition pipeline. Subscribes to `bill.text.ingested` events (emitted by Component 4 after text is downloaded and stored), orchestrates decomposition via the `DecompositionOrchestrator`, and emits `bill.decomposition.completed` events.

---

## Classes

### `BillDecompositionPipelineApp` — IOApp Entry Point (10.9)
**Package:** `repcheck.decomposition.pipeline.app`

Extends `IOApp`. Wires all dependencies and runs the pipeline.

```scala
object BillDecompositionPipelineApp extends IOApp
```

#### `run(args: List[String]): IO[ExitCode]`

Pure wiring — no domain logic. All pipeline logic extracted into `runPipeline` companion method.

1. Load `DecompositionPipelineConfig` via PureConfig
2. Build `Resource` bundle:
   - AlloyDB transactor (Doobie/HikariCP)
   - http4s Ember client (shared by Ollama sidecar and Claude API)
   - DJL section embedder model (`SectionEmbedder.resource`)
   - DJL search embedding model (`SemanticEmbeddingService.resource`)
3. Use resource bundle → call `runPipeline(config, transactor, httpClient, ...)`
4. Return `ExitCode`

#### `private[app] def runPipeline(config, transactor, httpClient, sectionEmbedder, embeddingService, ...): IO[ExitCode]`

Testable pipeline logic — receives all dependencies as parameters.

1. **Wire providers:**
   - `StructuredOutputEnforcer[IO](config.llm.structuredOutputMaxRetries)`
   - `OllamaProvider[IO](httpClient, config.llm.ollama.get, enforcer)` — for section parsing (config.llm.ollama is `Some` in decomposition)
   - `ClaudeProvider[IO](anthropicClient, config.llm.claude, enforcer)` — for Haiku simplification
   - `LlmRetryWrapper[IO](config.llm.retry, LlmErrorClassifier())` — wraps both providers
2. **Wire components:**
   - `ConceptClusterer[IO](config.clustering)`
   - `DecompositionPersister[IO](transactor)`
   - `PipelineLogger[IO]` — initialized with correlation context
   - `DecompositionOrchestrator[IO](ollamaProvider, sectionEmbedder, clusterer, embeddingService, claudeProvider, persister, config.decomposition, logger)`
   - `DecompositionEventPublisher[IO](pubsubPublisher, config.pubsub)`
3. **Subscribe to Pub/Sub:** Pull messages from `bill.text.ingested` subscription
4. **Stream processing:** For each event via `parEvalMap(config.parallelism)`:
   a. Deserialize `PipelineEvent[BillTextIngestedPayload]`
   b. Extract `payload.billId` and `payload.versionId` from the event
   c. Call `orchestrator.decompose(payload.billId, payload.versionId, event.correlationId)`
   d. On success → `eventPublisher.publish(decompositionResult, event.correlationId)` → `ProcessingResult.Succeeded`
   e. On failure → log error, `ProcessingResult.Failed`
   f. Ack Pub/Sub message
5. **Aggregate:** Collect all `ProcessingResult`s → `PipelineRunSummary`
6. **Return:** `ExitCode.Success` if no fatal errors, `ExitCode.Error` otherwise

> **`versionId` in input event:** The `bill.text.ingested` event includes `versionId: UUID` (the `bill_text_versions` primary key). Component 4 §4.8 creates this row when it downloads and stores the text, and includes the ID in the `BillTextIngestedEvent` payload. The decomposition pipeline subscribes to `bill.text.ingested` (not `bill.text.available`) because it needs the text content to already be stored in AlloyDB and the `versionId` to exist.

### `DecompositionPipelineConfig` — Full Pipeline Config (10.9)
**Package:** `repcheck.decomposition.pipeline.config`

PureConfig auto-derived. Top-level config aggregating all subsystem configs.

```scala
case class DecompositionPipelineConfig(
  database: DatabaseConfig,
  pubsub: PubSubConfig,
  decomposition: DecompositionConfig,
  clustering: ClusteringConfig,
  embedding: EmbeddingConfig,
  llm: LlmAdapterConfig,
  parallelism: Int,
  retry: RetryConfig
)
```

> **Note:** `LlmAdapterConfig.ollama` will be `Some(OllamaProviderConfig(...))` in this pipeline's config. The entry point asserts `config.llm.ollama.isDefined` at startup.

### `DecompositionEventPublisher[F[_]]` — Event Emission (10.9)
**Package:** `repcheck.decomposition.pipeline.app`

Publishes `bill.decomposition.completed` events to Pub/Sub.

```scala
class DecompositionEventPublisher[F[_]: Sync](
  publisher: PubSubPublisher[F],
  config: PubSubConfig
)
```

#### `publish(result: DecompositionResult, correlationId: UUID): F[Unit]`

1. Build `DecompositionCompletedPayload` from `result`:
   - `billId` ← `result.billId`
   - `versionId` ← `result.versionId`
   - `conceptGroupCount` ← `result.conceptGroupCount`
   - `sectionCount` ← `result.sectionCount`
2. Wrap in `PipelineEvent[DecompositionCompletedPayload]` with `correlationId`
3. Serialize to JSON via Circe
4. Publish to `config.outputTopic`

## Key Design Decisions

1. **Ollama sidecar is co-located.** The entry point assumes Ollama is on localhost:11434. No discovery or external networking needed.
2. **Two LlmProvider instances.** The decomposition pipeline wires both an `OllamaProvider` (for section parsing via sidecar) and a `ClaudeProvider` (for Haiku simplification). Both implement `LlmProvider[F]` — the orchestrator receives them as separate constructor parameters.
3. **DJL models loaded as Resources.** Two separate DJL models: 384-dim for clustering (SectionEmbedder) and search-quality for persistence (SemanticEmbeddingService). Both loaded at startup, released on shutdown.
4. **Testability via companion extraction.** All orchestration logic lives in a testable method. The IOApp `run` is pure wiring.
5. **New event type.** `bill.decomposition.completed` is a new event not in the original Component 2 event catalog. It must be registered in pipeline-models.
6. **Enforcer maxRetries from config.** `StructuredOutputEnforcer` receives `config.llm.structuredOutputMaxRetries` — this is the explicit wiring path.

## Acceptance Criteria

| Criterion | Test |
|-----------|------|
| Config loads all subsystem configs via PureConfig | Valid `application.conf` → `DecompositionPipelineConfig` populated |
| Startup asserts `config.llm.ollama.isDefined` | Missing ollama config → fail fast at startup |
| AlloyDB transactor created as `Resource` | Transactor acquired and released on shutdown |
| http4s Ember client created as `Resource` | Client acquired and released on shutdown |
| DJL section embedder loaded as `Resource` | Model loaded at startup, released on shutdown |
| DJL search embedding service loaded as `Resource` | Model loaded at startup, released on shutdown |
| `OllamaProvider` wired with http4s client and Ollama config | Provider receives `httpClient` and `config.llm.ollama.get` |
| `ClaudeProvider` wired with Anthropic SDK and Claude config | Provider receives SDK client and `config.llm.claude` |
| `StructuredOutputEnforcer` receives `structuredOutputMaxRetries` from config | `config.llm.structuredOutputMaxRetries` → enforcer constructor |
| `PipelineLogger` wired into orchestrator | Logger created and passed to `DecompositionOrchestrator` |
| Pipeline subscribes to `bill.text.ingested` | Pub/Sub subscription created for configured topic |
| Input event contains `billId` and `versionId` | `BillTextIngestedPayload` deserialized → both fields available |
| Each event triggers decomposition orchestrator | Event received → `orchestrator.decompose()` called with `billId` + `versionId` |
| Successful decomposition emits `bill.decomposition.completed` | `orchestrator.decompose` succeeds → event published |
| Failed decomposition produces `ProcessingResult.Failed` | `orchestrator.decompose` throws → `Failed` result, no event |
| Pub/Sub message acked after processing | Both success and failure → message acknowledged |
| Bills processed in parallel per config | `parallelism = 4` → `parEvalMap(4)` |
| `PipelineRunSummary` aggregated from results | 3 succeeded, 1 failed → summary reflects counts |
| Correlation ID propagated from input event to output event | Input event correlation ID → output event has same ID |
| `DecompositionCompletedPayload` serializes correctly | Payload → JSON → deserialize → same payload |
| Pipeline logic in testable companion method | `runPipeline` callable with mock dependencies |
| `run` method is pure wiring with no domain logic | `run` only creates Resources and calls `runPipeline` |
| Exit code 0 when all succeed, 1 on any failure | All success → `ExitCode.Success`; any failure → `ExitCode.Error` |
| Skipped decompositions (already existed) still emit event | `alreadyExisted = true` → event still published with existing counts |
