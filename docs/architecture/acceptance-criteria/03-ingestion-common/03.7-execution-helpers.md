### New/Extended Classes

#### `PipelineBootstrap[F[_]]` — Object

Standard bootstrap sequence that every pipeline application follows. NOT a base class — it's a helper that applications compose.

```
object PipelineBootstrap {
  def loadConfig[F[_]: Sync, C: ConfigReader](args: List[String]): F[C]
  def extractRunId[F[_]: Sync](args: List[String]): F[String]
  def initTransactor[F[_]: Async](dbConfig: DatabaseConfig): Resource[F, Transactor[F]]
}
```

> Each method is parameterized by `F[_]` with the appropriate type class bound. Callers specify `F` at the call site (e.g., `PipelineBootstrap.loadConfig[IO, MyConfig](args)`). This avoids the compilation error of referencing an unbound `F` on an `object`.

**`loadConfig`** behavior:
1. Takes first arg as JSON string (passed by the Launcher via Cloud Run Job arguments)
2. Parses with PureConfig using the provided `ConfigReader[C]`
3. Returns typed config or raises `ConfigLoadFailed`

> **Config source is args-only.** The Launcher serializes the full config as a JSON string and passes it as the first CLI argument. There is no environment variable merging or config file fallback. This ensures reproducibility — the exact config for any pipeline run is captured in `workflow_run_steps.original_message` and can be replayed.

**`extractRunId`** behavior:
- The run ID is generated by the Launcher and passed to the pipeline app via Cloud Run Job arguments
- `extractRunId` reads it from the args (not generated locally)
- This ties the pipeline execution back to `workflow_runs.run_id` from Component 2

| Acceptance Criterion | Test |
|---------------------|------|
| `loadConfig` parses valid JSON arg | `["{\"apiKey\":\"abc\"}"]` → typed config |
| `loadConfig` raises `ConfigLoadFailed` for empty args | Empty list → `ConfigLoadFailed` |
| `loadConfig` raises `ConfigLoadFailed` for invalid JSON | `["not json"]` → `ConfigLoadFailed` |
| `extractRunId` reads run ID from Launcher-provided args | Args contain run ID → returns it |
| `extractRunId` raises error if run ID missing from args | Args without run ID → error |

#### `WorkflowStateUpdater[F[_]]` — Trait

Each pipeline application updates its own execution state in the `workflow_run_steps` table (per Component 2 §2.6). This trait provides the interface.

```
trait WorkflowStateUpdater[F[_]] {
  def recordStepStarted(runId: String, stepName: String): F[Unit]
  def recordStepCompleted(runId: String, stepName: String): F[Unit]
  def recordStepFailed(runId: String, stepName: String, error: String): F[Unit]
  def incrementRetryCount(runId: String, stepName: String): F[Int]
  def getRetryCount(runId: String, stepName: String): F[Int]
}
```

**State transitions:**
- `recordStepStarted` → status = `running`, `started_at` = now
- `recordStepCompleted` → status = `completed`, `completed_at` = now
- `recordStepFailed` → status = `failed`, `completed_at` = now, `error_message` = error
- `incrementRetryCount` → retry_count += 1, returns new count

| Acceptance Criterion | Test |
|---------------------|------|
| `recordStepStarted` creates row with status `running` | Start step → query → status = "running" |
| `recordStepCompleted` sets status `completed` and `completed_at` | Complete step → query → status = "completed", completed_at non-null |
| `recordStepFailed` sets status `failed` with error message | Fail step → query → status = "failed", error_message = expected |
| `incrementRetryCount` returns incremented count | Start at 0 → increment → returns 1 → increment → returns 2 |
| `getRetryCount` returns 0 for new step | No prior increments → returns 0 |
| Multiple steps for same run tracked independently | Step A completed, Step B failed → both correct |

#### `PipelineFailureHandler[F[_]]` — Trait

Encapsulates the retry-or-fail logic per Component 2 §2.3: on failure, requeue original message to the **same Pub/Sub topic** with incremented retry count. After 3 retries, mark step as failed and stop.

```
trait PipelineFailureHandler[F[_]] {
  def handleFailure(
    runId: String,
    stepName: String,
    originalMessage: String,
    error: Throwable
  ): F[FailureAction]
}
```

**`FailureAction`** — enum:
- `Requeued(retryCount: Int)` — message requeued, pipeline stops this execution
- `PermanentlyFailed(retryCount: Int)` — max retries exceeded, step marked failed

| Acceptance Criterion | Test |
|---------------------|------|
| First failure → `Requeued(1)` | retryCount=0 → requeue → `Requeued(1)` |
| Second failure → `Requeued(2)` | retryCount=1 → requeue → `Requeued(2)` |
| Third failure → `Requeued(3)` | retryCount=2 → requeue → `Requeued(3)` |
| Fourth failure → `PermanentlyFailed(3)` | retryCount=3 → no requeue → `PermanentlyFailed(3)` |
| Requeue publishes original message back to Pub/Sub | Capture published message → matches original payload |
| `PermanentlyFailed` calls `recordStepFailed` | Verify state updater called with error message |
| Max retries configurable (default 3) | Config with maxRetries=5 → fails on 6th attempt |
